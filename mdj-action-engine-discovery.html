The Problem We're Solving

The Machine has 82 tools today, growing to 200+. Each tool does one thing well. But connecting them into workflows requires conversations. Right now:

  • Workflows are hardcoded chains (REVIEW_PREP → PULL_DOCUMENTS → BUILD_CASEWORK)
  • Outputs are static (templates, pre-rendered PDFs)
  • User input is clicking buttons, not conversing
  • There's no "registry" of what MDJ can do

MDJ Action Engine fixes this. It makes conversational workflow triggering the default instead of the exception.

Architecture Pattern: Tools × Super Tools × Wires

Same architecture pattern as ATLAS — three tiers of increasing complexity:

Tier 1: MDJ Tools (Atomic)

Single, well-defined actions. No decision-making, no sub-steps. Examples:

  • search_client — Find a client by name, ID, or phone
  • get_accounts — List all accounts for a client
  • send_email — Compose and send email to a client
  • check_pipeline — Get status of an opportunity in RAPID_CORE
  • get_client_docs — List Google Drive files tagged for a client
  • pull_rmd_calcs — Fetch RMD calculations from QUE for given year/age

Tier 2: MDJ Super Tools (Compound)

Orchestrates MDJ Tools with decision gates and output transformation. Examples:

  • REVIEW_PREP — search_client → get_accounts → get_client_docs → compile summary
  • PULL_DOCUMENTS — get_client_docs → organize by type → validate currency
  • BUILD_CASEWORK — get_accounts → get_rmd_calcs → analyze gaps → draft recommendations
  • DRAFT_COMMUNICATION — pull review/casework → consult templates → generate first draft
  • RMD_ANALYSIS — get_accounts + pull_rmd_calcs → calculate shortfalls → generate report
  • COVERAGE_GAP — get_accounts → compare to portfolio model → identify missing products

Tier 3: MDJ Wires (Full Workflows)

Complete business processes. Each wire has defined inputs, gates, outputs, and post-execution delivery. Examples:

  • ONBOARD_AGENCY — New agency kicks off. search_client → send_welcome_package → schedule kickoff → create case ticket
  • ANNUAL_REVIEW — Review cycle. REVIEW_PREP → BUILD_CASEWORK → DRAFT_COMMUNICATION → email draft + schedule meeting
  • AEP_ENROLLMENT — Medicare enrollment window. pull_docs → run_comparison → draft_communication → create opportunity
  • NEW_BUSINESS — New client inquiry. qualify → build_casework → send_proposal → assign to sales

How It Works Conversationally

User says: "Hey MDJ, run a review prep for the Aswegans"

MDJ Action Engine:

  1. Parses intent: REVIEW_PREP wire, client = Aswegans
  2. Validates input exists (searches Firestore, ATLAS registry)
  3. Triggers the REVIEW_PREP Super Tool
  4. Each MDJ Tool (search_client, get_accounts, etc.) runs in sequence with gates
  5. Output is compiled (client summary, account list, document list, recommendations)
  6. Returns structured output: JSON or human-readable markdown

Outputs = Contextual, Not Static

Current system: "Here's a PDF template you fill in manually."

MDJ Action Engine: Outputs are aware of context.

  • CaseWork Output: Generated from BUILD_CASEWORK Super Tool. Not a static template. Contains account data, recommendations, calculations — all real data.
  • Review Letter: Generated from DRAFT_COMMUNICATION. Pulls actual client name, account values, RMD calcs, product gaps. Not a template.
  • Mineart Presentation: Generated from a WIRE output. Pulls 5-year performance data, recommendation summary, RMD analysis. Uses actual data.

The Ecosystem

MDJ Action Engine is the glue. It uses, but does not replace, existing systems:

QUE (Medicare Intelligence)

Already has wires (wireIncomeNow, wireGrowthMax, wireAssembleB4, etc.). MDJ Action Engine CALLS these wires as Super Tools. MDJ doesn't rebuild quoting logic — it orchestrates QUE quoting.

ATLAS (Data Registry)

Already has the registry pattern (sources, tools, wires all registered + introspectable). MDJ Action Engine adopts this pattern for actions. MDJ Tools register in a MDJ Action Registry (mirrors ATLAS structure). Introspection API tells you "what can MDJ do?" Same as "what data does ATLAS know about?"

Firestore (Primary Database)

MDJ Tools read from Firestore (clients, accounts, opportunities). MDJ Super Tools write to Firestore (case tasks, next steps). Full audit trail in Firestore activity logs.

Google Drive (Document Store)

MDJ Tools pull documents from Drive (via tag-based lookup, not folder search). Super Tools organize output as Drive folders. Wires create shareable links.

RAPID_CORE + GAS Sheets (Legacy, Read-Only)

MDJ Tools read pipeline status, client master, producer data from Sheets for now. Bridge writes to Sheets in dual-write mode. After migration: MDJ reads Firestore only.

NBX (Next Business Experience)

MDJ Action Engine is NOT NBX. NBX is client-facing portal. MDJ is internal tool orchestration. NBX may eventually CALL MDJ Wires (like an NBX client triggers "run a review prep for me"), but today they're separate.

Data Flow

MDJ Action Engine ├─ Parses user intent ├─ Looks up in MDJ Action Registry ├─ Routes to appropriate Super Tool │ ├─ Calls MDJ Tools in sequence │ │ ├─ Tool queries Firestore/Sheets/Drive │ │ ├─ Returns { success, data } │ │ └─ Next tool uses output as input │ ├─ Applies decision gates │ └─ Compiles output └─ Returns structured result ├─ JSON for API consumers ├─ Markdown for chat UI └─ Creates Drive folder + Firestore task record

Phase 1: Platform Vision STUB PHASE

Define scope: What CAN MDJ do? Start with 15 wires covering 80% of annual workflow volume.

1

Inventory Annual Workflows

Vince (sales) + Nikki (service) + Aprille (legacy): "What do you do every year? Every quarter? Every month?"

2

Identify Core Wires (15 MVP)

ANNUAL_REVIEW, AEP_ENROLLMENT, NEW_BUSINESS, RENEWAL_LETTER, RMD_ALERT, COVERAGE_GAP_ANALYSIS, etc.

3

Design Wire Inputs & Outputs

Each wire: what's required to start? What's the final deliverable? Who sees it?

4

Define Decision Gates

When does a wire pause for human approval? When does it self-correct? When does it escalate?

Phase 2: Tool Inventory

Catalog 82 existing tools + identify 20 new ones needed.

1

Audit Existing Tools

What already exists? (search_client, send_email, get_accounts, pull_rmd_calcs, etc.) What's partially working?

2

Define New Tools

Gaps: send_sms, schedule_meeting, create_proposal, check_compliance_status, pull_performance_data, etc.

3

Standardize Tool Signatures

Every tool: inputs (JSON schema), outputs ({ success, data, error }), error modes, permissions.

Phase 3: Super Tools & Wires

Build orchestration layer.

1

Define Super Tool Specs

REVIEW_PREP, BUILD_CASEWORK, DRAFT_COMMUNICATION, etc. Each: tools used, order, gates, output format.

2

Implement Orchestration Engine

Sequence execution, handle failures, apply gates, compile output. TypeScript module in @tomachina/core.

3

Wire-Level Integration

ANNUAL_REVIEW wire = REVIEW_PREP → BUILD_CASEWORK → DRAFT_COMMUNICATION → create Firestore task → send email.

Phase 4: MDJ Action Registry

Make MDJ self-aware.

  • Firestore collection: mdj_registry with tools, super_tools, wires
  • Schema mirrors ATLAS: name, description, inputs, outputs, gates, permissions
  • API endpoint: GET /api/mdj/registry — "what can MDJ do?"
  • Introspection: tools can be queried, wires can be listed, capabilities can be filtered by role

Phase 5: FORGE Runner Integration

MDJ Action Engine becomes the execution engine for FORGE. FORGE tickets become wires.

  • FORGE Sprint Ticket: "Build ANNUAL_REVIEW wire"
  • SPC runs ticket using FORGE protocol
  • Output: MDJ Super Tool registered + tested + wired into ANNUAL_REVIEW workflow
  • Result: New capability available to users immediately post-deployment

Out of Scope (Today)

  • NBX integration (future phase: NBX clients trigger MDJ wires)
  • Multi-step approval workflows (future: gates can fork to human review)
  • Real-time notifications (future: webhooks on wire completion)
  • AI-assisted recommendations (future: Claude integration for "what should we do next?")
