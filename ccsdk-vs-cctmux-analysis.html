CCSDK vs CC-tmux

Architecture Analysis for RONIN, RAIDEN, and VOLTRON
Prepared by MUSASHI, ARTxBLADE — 2026-03-30
Based on 72 hours of production data from MDJ_SERVER

What Are They?

CCSDK

@anthropic-ai/claude-agent-sdk

A Node.js wrapper that spawns Claude Code CLI as a subprocess. Programmatic control via JavaScript. Each claudeInvoke() = new process.

Think: Hiring a contractor for one task. They show up, do the job, leave. Next task = new contractor.

CC-tmux

claude CLI running in a persistent tmux session

A single Claude Code process that stays alive. Context accumulates. Multiple tasks executed sequentially in the same session.

Think: A full-time employee at a desk. They learn the codebase once, then build everything in sequence. No re-onboarding.

The Saturday Proof

What Happened with CCSDK (FORGE v1)

18 tickets × 1 claudeInvoke() each = 18 cold-start processes

  • Each process: load CLAUDE.md (50K tokens), read repo, read discovery doc, read plan, build ONE ticket, exit
  • RAM: 29GB peak (crashed the 32GB server)
  • Time: 90+ minutes for 18 tickets
  • Cost: ~$20-40 per sprint
  • Ticket #18 had ZERO context from tickets #1-17

What Happened with Single-Session Fix (FORGE v2)

18 tickets × 1 session = 1 warm process, all tickets sequential

  • One process: load context ONCE, build all tickets in sequence
  • RAM: 5.9GB steady (83% reduction)
  • Time: ~25 minutes total (seed → plan → build → deploy)
  • Cost: ~$5-8 per sprint
  • Ticket #18 had full context from tickets #1-17

Per-Agent Recommendation

RONIN (The Builder)

CCSDK: BAD for builds

18 cold starts = 18x overhead. Proven on Saturday. Each ticket rebuilds context from scratch.

CC-tmux: IDEAL for builds

One session, all tickets, warm context. This is what JDM did on his MacBooks to build 500K lines in 2 months.

Verdict: Switch RONIN build phase to CC-tmux. Keep CCSDK for FORGE orchestration (phase transitions, Firestore writes). The builder becomes a persistent session; the orchestrator stays programmatic.

RAIDEN (The Guardian)

CCSDK: CORRECT fit

RAIDEN handles single events: triage ONE ticket, fix ONE bug, route ONE issue. Stateless is correct. Each triage is independent.

CC-tmux: Overkill

Persistent context is unnecessary — RAIDEN doesn't need to remember prior triages. Each event is self-contained.

Verdict: KEEP on CCSDK. Single-task, stateless, event-driven = perfect fit. No change needed.

VOLTRON (The Team's Weapon)

CCSDK: GOOD for Mode 1 (Wires)

Wire execution is a single task: user triggers wire → VOLTRON executes tools → returns result. Stateless per execution.

CC-tmux: BETTER for Mode 2 (Chat)

Multi-turn conversations benefit from persistent context. "What did we just discuss?" works naturally in a persistent session.

Verdict: HYBRID. Keep CCSDK for Mode 1 (wire execution — single task). Consider CC-tmux for Mode 2 (chat — multi-turn where context matters).

Dojo UI Exposure: YES — VOLTRON's work (tool calls, wire executions, conversation summaries) can be surfaced in the Dojo tab alongside RONIN builds and RAIDEN triages. The ForgeTracker pattern already exists.
