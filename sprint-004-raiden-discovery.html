Why RAIDEN Exists

The Problem Before RAIDEN

Every bug that hit production on March 25 consumed JDM's personal attention. Phone routing down. Auth sessions dropping. ACF scans timing out. MDJ tools returning empty. Eight issues — all identified, none auto-resolved. Hours of JDM time burned on triage, context-switching, Slack threads.

The World With RAIDEN

A bug appears in #prodashx-urgent-fixes-and-bugs at 2:14am. By 2:29am, RAIDEN has read it, triaged it, found the root cause (one missing setInterval call), committed the fix to a new branch, opened a PR, and posted back to the channel: "Fixed. PR #412. — @RAIDEN."

JDM wakes up at 7am to a Slack DM: "2 quick fixes shipped overnight. 1 issue routed to RONIN sprint. Nothing pending." The team never stopped. No one waited. The Machine protected itself.

RAIDEN's Identity

Decision Tree
New item arrives (Slack message or FORGE ticket) ↓ Does this already exist in the platform? → Query knowledge_entries (Sprint 003) or CLAUDE.md fallback → Signals: "how do I", "where is", "does ProDash have", "show me how" → YES: TRAIN — find docs, @mention person, post answer, close → NO: continue ↓ Is this a quick fix? (effort ≤ 30 min) → Signals: single file change, known pattern, no data migration → Examples: missing setInterval, type export gap, wrong URL in config → YES: FIX — branch → commit → PR → notify channel → close → NO: continue ↓ Is this complex? (multi-file, architectural, risky) → Signals: touches auth/Firebase, requires schema changes, cross-service → Examples: ACF batch processing, MDJ search route audit, COMMS E2E → YES: ROUTE — create TRK ticket → RONIN queue → DM JDM → NO: continue ↓ Is this a feature request or enhancement? → Signals: "would be nice", "can we add", "enhancement", "new feature" → YES: QUEUE — tag as enhancement → next RONIN sprint → notify requester
The Four Outcomes

TRAIN

Feature already exists. User needs education, not code. RAIDEN queries knowledge_entries for matching docs, formats a clear response, @mentions the requester, and posts to channel. Closes immediately. No dev work needed.

FIX

Real bug, scope ≤ 30 minutes. RAIDEN invokes RONIN's claudeInvoke() pattern to execute the fix autonomously. Creates branch, commits, opens PR. Posts to channel with PR link and root cause summary. JDM is NOT notified unless explicitly P0.

ROUTE

Real bug, scope > 30 min or structurally complex. RAIDEN creates a TRK item in the FORGE board, routes it to RONIN's sprint queue, and DMs JDM with the issue summary, complexity estimate, and TRK number.

QUEUE

Feature request, not a bug. No urgency. RAIDEN tags the FORGE ticket as type=enhancement, adds to the backlog, and notifies the requester that it's been captured for the next sprint cycle.

The 5 Recurring Bug Categories

From historical analysis of #prodash-hl-fixes. RAIDEN recognizes these patterns instantly and applies the correct fix approach without re-investigation.

📞
Phone / COMMS
Most Volatile P0 Urgency
  • Twilio webhook URL mismatch after redeploy
  • OIDC identity token expiry on Cloud Run
  • Call routing tables out of sync
  • Softphone UI disconnects from WebRTC bridge
  • Inbound call falls to voicemail unexpectedly

RAIDEN triage: if +18886208587 is down → P0 DM to JDM immediately. Webhook URL mismatch → check Cloud Run URL vs Twilio config → update → verify.

🔐
Auth Failures
High Blast Radius All Users
  • Firebase session persistence drops
  • ID token refresh loop on re-auth
  • Entitlement check 403 on valid users
  • Service account key expiry (MDJ1)
  • CORS rejection on portal API proxy

RAIDEN triage: Auth bugs always ROUTE — blast radius too high to auto-fix. Create TRK, DM JDM. Exception: SA key rotation on MDJ1 is a known quick fix.

🌱
Data Seeding Gaps
Code Exists, Data Missing
  • New Firestore collection has no seed data
  • Security rules missing for new collection
  • Feature works locally, 403 in prod
  • Schema added to code but not seeded
  • FORGE tracker missing initial TRK records

Quick-fixable. Check firestore.rules for missing collection. Add rules → deploy → verify.

🧭
Navigation / UI Regressions
User-Visible Usually Quick
  • Row click handler broken after component refactor
  • Popup blocked by browser security policies
  • Route mismatch after Next.js app dir change
  • Modal renders off-screen on mobile
  • Tab navigation jumps to wrong section

Strong auto-fix candidate. Single component, no data risk. Check onClick removal, Next.js route group changes, CSS z-index conflicts.

⏰
Scheduler / Cron Failures
Silent Until It Isn't
  • Cloud Scheduler hitting wrong URL after redeploy
  • OIDC token not attached to scheduled requests
  • setInterval never registered in server.ts
  • Cron job succeeds but handler throws silently
  • Time zone offset causing off-schedule triggers

URL mismatches and missing setInterval calls are quick-fixes (checkStaleGates is a live example). OIDC auth mismatches route to RONIN.

RAIDEN's 6-Stage Status Flow

Compare to RONIN's 10-stage FORGE lifecycle. RAIDEN runs fast. Anything too complex exits through ESCALATED and becomes a RONIN sprint ticket automatically.

1
NEW new
Just came in, unprocessed. RAIDEN has seen it but hasn't classified it yet. Tickets from Slack, Quick Submit, or auto-detect land here first.
2
TRIAGING triaging
RAIDEN is classifying — reading context, checking recent deploys, checking for duplicates, determining whether it can self-fix or must route.
3
FIXING fixing
RAIDEN is actively working — writing a fix, creating a PR, or generating training content. This is the execution phase.
4
VERIFYING verifying
Fix applied. RAIDEN is checking it worked — CI status, Slack acknowledgment, follow-up from reporter. Stays here until confirmed resolved.
5
DONE done
Resolved. Resolution field filled. RAIDEN DMs JDM overnight summary. Archived after 30 days.
6
ESCALATED → RONIN escalated
Too complex for a fast fix. RAIDEN creates a RONIN sprint ticket automatically, links back to the RAIDEN item, and posts context in #prodashx-urgent-fixes-and-bugs. The issue doesn't disappear — it upgrades.

Design Rules

No discovery doc. No description field. No component selector. No epic link. Team members are not engineers — they describe symptoms, not causes. RAIDEN reads the symptom and figures out the rest. Keep the form to 3 fields + submit.

Slack Channel — Primary Submission Path

Team members can also submit directly from Slack — no need to open The Dojo. This is the zero-friction path for people already living in Slack.

#dojo-fixes — The Source of Truth

A dedicated Slack channel (#dojo-fixes or #prodash-report) gives the team a natural-workflow submission path. RAIDEN monitors the channel as a first-class submission source — identical processing to the Quick Submit button.

Channel purpose: submission, acknowledgement, status updates, resolution confirmation. The channel IS the source of truth for what's broken and what's been fixed.

Why This Matters — The 10-Duplicate Problem

When 10 people hit the same bug at the same time, they all submit a ticket. Without visibility, you get 10 duplicate tickets, 10 people asking "is it fixed yet?", and noise that buries the real signal.

With the channel: Person 1 submits → everyone sees the ID immediately → persons 2–10 see it's already reported and stand down. RAIDEN posts the fix. Everyone sees it. Done. No follow-up DMs. No support queue. The channel IS the source of truth.
