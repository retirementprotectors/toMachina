Current Architecture
What Needs to Change

Current State (Today)

The pipeline works, but it's narrow:

  • Claude Code sessions → MEMORY.md files → knowledge-promote.js (daily 11:15pm) → CLAUDE.md
  • Violation logs → compliance-history.json → trend tracking
  • Hookify rules (37 rules) → auto-enforced on every commit
  • Works for: Claude Code session learnings, code violations, feature planning
  • Broken for: MDJ Agent conversations, team knowledge, client insights, interaction patterns

The Problem

MDJ Agent runs on the Dell 24/7, but it doesn't teach the Machine anything.

Every conversation with MDJ is ephemeral. When Nikki asks MDJ "what did we learn about client ABC's tax situation?", MDJ doesn't know. It only knows what's in that one conversation and Firestore snapshots.

Meanwhile, CLAUDE.md is at 53k characters and growing. Knowledge is trapped in files, not queryable. Violations are logged but isolated per session. Team patterns are invisible.

Result: The Machine isn't learning. It's just remembering.

Target State (Learning Loop 2.0)

Every interaction writes to a unified knowledge store. Every system reads from everywhere. The entire organization gets smarter.

Claude Code Session ↓ Firestore Knowledge Store ← MDJ Conversations ↓ ↑ Team Interactions ──┘ ↓ knowledge-promote.js ← Client Insights ↓ CLAUDE.md (rules only) ↓ Next session smarter

CLAUDE.md shrinks to rules + references only. Deep context lives in Firestore, queryable, cross-referenced, versioned. Every interaction (Claude Code, MDJ, team, clients) feeds the same knowledge engine. The system makes connections humans miss.

The One-Liner

"Knowledge enters from everywhere. Wisdom gets distributed to everyone."

The Flywheel (What Works)

Violation ← New code pushed ↓ Caught by hookify ↓ Logged to violation-log.jsonl ↓ Daily promotion (11:15pm) ↓ Rule created or CLAUDE.md updated ↓ Rule enforced next session ↓ Violation impossible (next cycle)

This works beautifully. We want the same closed loop for MDJ, team, and client knowledge.

Changes Required

1. Firestore Collections for Knowledge

Replace file-based MEMORY.md with queryable Firestore. Collections: knowledge_entries, learning_sessions, patterns, violations, client_insights_enriched.

2. MDJ Agent Write Integration

Every MDJ conversation writes learnings to knowledge_entries. Conversation context, client insights, patterns detected, questions asked/answered.

3. Team Interaction Capture

Extend MDJ widget + Slack integration to log team questions, decisions, approvals. Build queryable record of "how did we decide this?" and "what did we learn from that case?"

4. CLAUDE.md Trimmed Down

CLAUDE.md stays lean: rules, references, session URLs only. Deep context moves to Firestore. Reduces token load on LLM context window, keeps instructions current.

5. knowledge-promote.js Enhanced

Updated to read/write Firestore collections. Aggregates patterns from all sources (Claude Code, MDJ, team, clients). Promotes actionable insights back to CLAUDE.md + creates new Firestore references.

6. Cross-Application Learning

Claude Code ↔ MDJ ↔ Team ↔ Clients all feed the same store. A lesson learned in one channel becomes visible to all others within hours.

    - severity: "info" | "caution" | "critical"
    - created_at: timestamp
    - promoted_to_claude_md: boolean
    - expiry: timestamp (for transient entries)

patterns/
  {id}
    - pattern_name: string
    - occurrences: integer
    - related_knowledge: string[]
    - recommendation: string
    - confidence: 0-1
    - last_observed: timestamp
