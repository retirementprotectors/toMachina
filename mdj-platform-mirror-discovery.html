The Problem

MDJ currently runs 82 hand-coded tools. Every tool was written manually, reflecting a snapshot of the platform at the moment it was built. When a developer adds a field, updates a route, or ships a new module — MDJ is blind to it until someone writes another tool by hand.

The portal has hundreds of capabilities. MDJ knows maybe 30% of them. The gap grows every sprint.

At 181 agencies, MDJ will be the primary interface for hundreds of people. A 70% capability gap is not a backlog item. It is a structural failure.

The Solution

MDJ becomes a MIRROR of ProDashX — same navigation tree, same data paths, same capabilities, auto-synced. Not a replacement. Not a parallel system. A mirror.

MDJ doesn't REPLACE the portal. MDJ IS the portal, operated by conversation.

"The portal is the dashboard. MDJ is the steering wheel."

The UI becomes visual verification. MDJ becomes the primary interface. At 181 agencies, most interactions will happen through MDJ — not by clicking through menus, but by talking to the machine.

Auto-Registry: How MDJ Stays in Sync

The current 82-tool registry was hand-written. Every tool is a maintenance burden. Every deploy potentially orphans a tool. The Platform Mirror replaces this with a generated registry — the tool list is an output of the codebase, not a separate document.

The Introspection Pipeline

Portal Routes (apps/prodash/app/**/page.tsx) └─ Parse route tree └─ Extract page metadata (title, params, data dependencies) └─ → Tool definitions: navigate_to_*, get_*_detail API Routes (services/api/src/routes/*.ts) └─ Parse route handlers + middleware └─ Extract: method, path, params, response shape └─ → CRUD tools: create_*, update_*, delete_*, list_* Firestore Schema (packages/db/src/collections/*.ts) └─ Parse collection types + subcollections └─ Extract: fields, types, query patterns └─ → Data access tools: query_*, search_* MCP Servers (claude mcp list → enumerate tools) └─ Already structured as tool definitions └─ → Pass through as-is, filter by user level ↓ ┌─────────────────────────────┐ │ MERGED TOOL REGISTRY │ │ (generated at deploy time) │ └─────────────┬───────────────┘ │ ↓ ┌─────────────────────────────┐ │ Filter by user entitlement │ │ (Coordinator / Director / │ │ Specialist / VP+) │ └─────────────┬───────────────┘ │ ↓ ┌─────────────────────────────┐ │ Passed to Claude as tools │ │ in the MDJ system prompt │ └─────────────────────────────┘

Same Pattern as ATLAS Introspection

ATLAS already does this for data sources — it reads the codebase to understand what data flows through the platform. The Auto-Registry is the same concept applied to UI capabilities.

The Web UI vs. API Gap

Not everything has an API. A significant portion of the carrier ecosystem — illustrations, enrollment systems, portal status checks — exists only as a web UI. No API. No integration available. No workaround through standard MCP tools.

This is where MDJ hits a wall without Playwright. The conversation starts strong, then hits "I'd need you to go to the carrier portal and..." — and the human is back in the loop.

Playwright browser automation on MDJ1 closes this gap permanently. MDJ launches a headless browser, navigates to the carrier portal, fills the form, runs the illustration, captures the result, and returns it to the conversation. The human never leaves the chat.

Team Credential Management

The vault solves MDJ's auth problem. It also solves the team's password problem. Currently, carrier portal credentials are scattered — sticky notes, personal password managers, shared docs. No central management. No rotation policy. No access logging. No compliance trail.

This is a FINRA compliance gap. Shared credentials with no access logging are an exam finding waiting to happen.

Slack DM + Private Channel Access

Current gap: MDJ can search public Slack channels but cannot read DMs, group DMs, or private channels. This means MDJ is blind to the majority of where decisions actually get made at RPI — the direct conversations.

The Fix: Two Paths, Both Available Now

1

claude.ai Slack MCP — DM Reading via User ID

The slack_read_channel tool in the claude.ai Slack MCP accepts a user ID as the channel_id parameter. This reads the DM thread between the bot and that user. JDM's user ID is U09BBHTN8F2.

2

Local Slack MCP (shouting-mcp-slack) — Broader Permissions

The local slack MCP has broader token permissions than the cloud version. It can access channels the bot has been invited to, including private channels. The gap is that the bot must be in the channel — MDJ cannot read a private channel it hasn't been added to.

MDJ SUPER TOOLS (Compound)

Multi-step sequences that combine multiple atomic tools into a single named capability. These are what make MDJ feel like a skilled advisor, not a query engine.

REVIEW_PREP

Full client analysis before a meeting. Pulls profile, all accounts, pipeline status, recent activity, open items, and drafts a meeting agenda. What MDJ did for the Aswegans review.

PULL_DOCUMENTS

List and retrieve files from a client's ACF Drive folder. Filter by type (policy, statement, form). Return download links or summarize content.

DRAFT_COMMUNICATION

Draft a client-facing email or text with full context — name, policy detail, tone, RPI branding. Holds for approval before sending.

RMD_ANALYSIS

Calculate required minimum distributions. Pull IRA/annuity balances, apply IRS tables, project deadlines. Flag clients approaching or past RMD age.

COVERAGE_GAP

Analyze a client's life, health, and annuity coverage. Identify gaps relative to household income, debt, and dependents. Surface cross-sell opportunities.

MEETING_PREP

Full pre-meeting package: REVIEW_PREP + agenda draft + key talking points + follow-up task template. Ready to go 30 minutes before any client meeting.

RUN_ILLUSTRATION

Playwright automation on carrier portals. Takes carrier + product + client params. Runs the illustration headlessly, captures results, saves PDF to ACF folder.

SUBMIT_CASE

DEX document assembly + carrier submission workflow. Pulls required forms, prefills from client record, packages for submission, logs to pipeline.

MDJ WIRES (Full Workflows)

End-to-end business processes. A Wire coordinates multiple Super Tools and atomic tools across time, with decision points, approvals, and notifications built in.

ONBOARD_AGENCY

Complete M&A intake pipeline. 7-day sequence: data import → agent setup → carrier appointments → system access → training protocol → go-live sign-off.

ANNUAL_REVIEW

Full household review cycle. MEETING_PREP → conduct review → log outcomes → update accounts → schedule follow-up → trigger any product changes.

AEP_ENROLLMENT

Medicare Annual Enrollment Period workflow. Eligibility check → plan comparison (QUE) → client recommendation → enrollment submission → confirmation logging. AEP blackout Oct-Dec enforced.

NEW_BUSINESS

Full new business pipeline. Client intake → needs analysis → illustration → application → underwriting status → issue → policy delivery → pipeline close.

Sprint Roadmap: Building the Platform Mirror

Ten sprints to go from 82 hand-coded tools to a fully mirrored, auto-synced, multi-audience conversational interface for the entire platform. Each sprint delivers a discrete, shippable capability.

Sprint 1

Auto-Registry

Build the registry generator. Parse portal routes, API routes, and Firestore schema. Emit tool definitions on every deploy. Replace the 82 hand-coded tools with generated output. This is the unlock for everything else.

Sprint 2

ACF / Drive Integration

Full document access for every client. PULL_DOCUMENTS and GET_DOCUMENT Super Tools. List folder contents, retrieve files, summarize documents, save to conversation context. MDJ can see everything in the client's digital file cabinet.

Sprint 3

Playwright Carrier Automation

RUN_ILLUSTRATION Super Tool wired to top-5 carrier portals. North American, Athene, Allianz, Nationwide, Lincoln. Headless browser on MDJ1. Credentials via vault. Results returned to conversation and saved to ACF folder.

Sprint 4

Credential Vault + Team Password Management

Centralized encrypted vault on MDJ1. Admin UI in ProDashX. Rotation alerts via Slack. Team credential access via MDJ conversation with permission check + audit log. FINRA-ready access trail.

Sprint 5

Slack DM + Private Channel Access

Wire DM reading via user_id as channel_id. Invite MDJ bot to key private channels. Default search to public_and_private. MDJ can now see the full conversation context of the business.

Sprint 6

Super Tools

Build REVIEW_PREP, PULL_DOCUMENTS, DRAFT_COMMUNICATION, RMD_ANALYSIS, COVERAGE_GAP, MEETING_PREP as named compound capabilities. Each one a tested, documented, reliable workflow that MDJ can invoke by name.

Sprint 7

Wires (Full Workflows)

ONBOARD_AGENCY, ANNUAL_REVIEW, AEP_ENROLLMENT, NEW_BUSINESS. End-to-end pipelines with decision points, approvals, notifications. The machine runs entire business processes on request.

Sprint 8

Client Portal MDJ Persona

Client-facing MDJ scoped to household data only. "What's my balance?" "When's my next appointment?" "Can you show me my Medicare coverage?" Auth via client JWT + Firestore household-scope rules.

Sprint 9

Partner / M&A MDJ Persona

MDJ-DAVID for SENTINEL audience. Deal pipeline, BoB valuation, acquisition status. Plus MDJ-Onboard for acquired agencies — process guides, carrier forms, go-live checklists. Both scoped and auth-gated.

Sprint 10

MDJ-Coach (Career Development)

The sixth specialist. Team member career development, licensing guidance, CE tracking, performance coaching. "What do I need to get my Series 65?" "What's my commission trajectory?" "What's next in my career path at RPI?"
