What We're Building

FORGE Runner is an autonomous orchestrator running on MDJ1. Give it a Discovery Document and it drives the entire 11-step FORGE lifecycle without JDM in the loop. Creates the sprint, fires audits, distributes work to parallel Sonnet builders, fixes CI failures, and deploys. JDM taps Approve at two gates. Everything else is automated.

Creates the Sprint

POSTs to /api/sprints, generates Discovery HTML, creates Tracker items for all tickets. No manual setup.

Drives the Lifecycle

Advances through all 13 phases automatically. Audits pass → next phase. Audits fail → auto-fix and re-audit.

Routes to Builders

Spawns parallel Sonnet subagents per ticket (max 3 concurrent). Results flow back to the runner.

Deploys Code

Runs the full #SendIt sequence. If CI is red, diagnoses and fixes (max 5 retries). Auto-merges when green.

Survives Restarts

All state stored in Firestore mdj_forge_runs. If MDJ1 reboots mid-sprint, the runner resumes from exactly where it left off.

Pauses for JDM

Two human gates with ProDashX notifications. JDM taps Approve in RIIMO or via POST /forge/approve/:id.

TRK-001 → TRK-003 — SDK, Types, Invoke Wrapper
TRK-001
Install SDK
Sonnet   Depends: nothing

Install and verify the Anthropic Claude Agent SDK in ~/mdj-agent/. This ticket is the gate for all others — nothing else starts until the SDK is confirmed working.

  • Run npm install @anthropic-ai/claude-agent-sdk in ~/mdj-agent/
  • Test: node -e "const { query } = require('@anthropic-ai/claude-agent-sdk'); console.log('SDK loaded')"
  • Test headless: claude -p "Hello" --model haiku --output-format json
  • Verify --model flag routes correctly (haiku vs sonnet vs opus)

Acceptance: SDK loads without error. Headless invocation returns JSON output with content field populated.

TRK-002
Types + Firestore
Sonnet   Parallel with TRK-003 after TRK-001

Create ~/mdj-agent/src/forge/types.ts — the single source of truth for all FORGE Runner interfaces.

  • ForgePhase enum: idle | creating_sprint | running_discovery | auditing_discovery | running_plan | auditing_plan | building | auditing_build | deploying | ci_loop | confirming | complete | blocked
  • ForgeRunState: sprint_id, sprint_name, phase, started_at, phase_started_at, artifacts (Record<string, string>), log (LogEntry[]), error?
  • LogEntry: timestamp, phase, message, level ('info' | 'warn' | 'error')
  • ForgePhaseResult: success, result?, error?
  • InvokeOptions: model, maxBudgetUsd, timeoutMs, cwd, allowedTools, permissionMode, resume, hooks
  • Firestore collection: mdj_forge_runs

Acceptance: TypeScript strict mode. No any types. All interfaces exported. Collection name constant exported.

TRK-003
claude-invoke.ts
Sonnet   Parallel with TRK-002 after TRK-001

SDK wrapper. Every phase executor calls this. It handles model routing, budget enforcement, timeouts, and session resumption.

  • Import { query } from @anthropic-ai/claude-agent-sdk
  • Signature: claudeInvoke(prompt: string, opts?: InvokeOptions): Promise<string>
  • Default model: sonnet
  • Default maxBudgetUsd: 3.00
  • Default timeoutMs: 300_000 (5 min)
  • Default cwd: ~/Projects/toMachina
  • Default permissionMode: 'acceptEdits'
  • Concatenate all streamed output chunks into a single string return value
  • Wrap with AbortController for timeout enforcement

Acceptance: Returns string. Timeout fires after timeoutMs. Model param passed to SDK. Resume works when session ID provided.

Orchestration

TRK-006 + TRK-007 — Runner State Machine + Phase Executors

Opus Required

TRK-006 and TRK-007 are the two most complex tickets in this sprint. The state machine must handle parallel execution, error recovery, Firestore persistence, CI retry loops, and human gate pausing. Use Opus for both. Do not substitute Sonnet.

TRK-006
runner.ts — The Orchestrator
Opus   Depends: TRK-003, 004, 005

The main ForgeRunner class. This is the state machine that drives the entire FORGE lifecycle. Every other module is called from here.

  • run(discoveryDoc, opts) — starts the full lifecycle from idle
  • approve(runId) — resumes from a blocked state
  • State machine: each phase: update Firestore → execute → check result → advance or block
  • Human gates: after auditing_plan and auditing_build, set phase to 'blocked', fire notification, await approve()
  • Parallel builders: during 'building', spawn concurrent claudeInvoke sessions per ticket, max 3 concurrent (Promise.allSettled with concurrency limiter)
  • CI loop: during 'deploying', poll CI every 60s; if red → diagnose → fix → retry (max 5 attempts); if 5 failures → gate JDM
  • Restart-safe: on startup, check Firestore for existing run — if phase is not idle/complete, resume from current phase

Acceptance: All 13 states reachable. Blocked states persist across process restarts. Parallel builder concurrency enforced. CI retry max 5. All state writes to Firestore before execution (write-ahead).

TRK-007
phases.ts — Phase Executors
Opus   Depends: TRK-006 interfaces

Executor functions for each phase. Each is an async function returning ForgePhaseResult. The runner calls these; phases do not call each other.

  • runCreateSprint() — forge-api: create sprint + tracker items in bulk
  • runDiscovery() — claudeInvoke(discoveryPrompt, { model: 'sonnet' }), save HTML to artifacts
  • runAuditDiscovery() — claudeInvoke(auditPrompt), parse GAPS/PASS from output, return pass/fail
  • runPlan() — claudeInvoke(planPrompt, { model: 'opus' }), save HTML to artifacts
  • runAuditPlan() — claudeInvoke(auditPrompt), parse result → PASS (advance) or FAIL (gate)
  • runBuild() — per ticket, claudeInvoke with builder prompt (model: sonnet), parallel where no deps, sequential where deps exist
  • runAuditBuild() — claudeInvoke with audit prompt per phase, aggregate results
  • runSendIt() — git checkout -b → git add → git commit → git push → gh pr create --auto --squash --delete-branch
  • runCILoop() — polls gh run list every 60s, handles failure with diagnose+fix cycle
  • runLandedIt() — bulkUpdateStatus('confirmed'), create ProDashX #LandedIt notification

Acceptance: Each function returns { success, result?, error? }. Phase outputs stored as named artifacts in Firestore. No phase calls another phase directly.

TRK-008 + TRK-009 — Express Routes + Human Gate Mechanism
TRK-008
Express Routes
Sonnet   Parallel with TRK-009

Mount FORGE routes on the existing mdj-agent Express server. All routes are authenticated.

MethodPathBody / ParamsReturns
POST /forge/sprint { discovery_doc, name, opts } 202 { run_id } — kicks off async
POST /forge/run/:id run ID in path 200 { phase } — advance to next phase
POST /forge/approve/:id run ID in path 200 { phase } — approve current gate
GET /forge/status/:id run ID in path 200 ForgeRunState
GET /forge/runs — 200 ForgeRunState[]

Acceptance: 202 returned immediately on sprint start (async kick-off). Status always reflects Firestore state, not in-memory state. All routes return { success, data?, error? }.

TRK-009
Human Gate Mechanism
Sonnet   Parallel with TRK-008

The pause/resume cycle that keeps JDM in control at key decision points without requiring him to babysit the runner.

Gate Trigger (Runner calls this)

1. Write to mdj_forge_runs: { phase: 'blocked', gate_reason, blocked_at }
2. Create Firestore notification: { type: 'forge_gate', target: 'josh@retireprotected.com', title, body, metadata: { run_id, sprint_id, phase } }
3. Notification surfaces in ProDashX Notifications module immediately.

Approve (JDM triggers via UI or API)

POST /forge/approve/:id → reads blocked state → clears blocked_at → runner resumes from current phase. Idempotent — double-approve is a no-op.

  • 24hr re-notify: scheduled check; if run is still blocked after 24hr, re-create notification (idempotent)
  • Gate 1: after auditing_plan — JDM reviews Discovery Doc + Plan HTML
  • Gate 2: after auditing_build — JDM reviews build audit results per phase

Acceptance: Gate write persists across process restarts. Notification appears in ProDashX within 5 seconds of gate trigger. Approve resumes runner within 5 seconds. 24hr re-notify fires on schedule.

TRK-010
E2E Test
Sonnet   Depends: all previous

Run the complete FORGE Runner lifecycle programmatically against a minimal 2-ticket test sprint. Every phase transition is asserted. Cleans up after itself.

  • Create test sprint with 2 trivial tickets (e.g., add a comment to an existing utility file)
  • Invoke runner.run(testDiscoveryDoc, { testMode: true })
  • Test mode: auto-approves gates (skips notification delivery, uses programmatic approve)
  • Test mode: --dry-run on all git ops (branch/commit/push do not execute)
  • Assert every phase transition recorded in mdj_forge_runs with correct timestamps
  • Assert notifications created at Gate 1 and Gate 2
  • Assert all artifacts stored (discoveryUrl, planUrl, buildResults)
  • Assert final phase is 'complete'
  • Cleanup: delete test run doc, test sprint, test notifications from Firestore
Run: npx ts-node src/forge/tests/e2e.test.ts File: ~/mdj-agent/src/forge/tests/e2e.test.ts

Acceptance: All assertions pass. Zero orphaned Firestore docs after run. CI not triggered. Run completes in under 10 minutes (test mode).
