TRK-002 MDJ Agent Service scaffold New No dependencies — start immediately

Build the Express server at ~/mdj-agent/ on MDJ_SERVER. This is the AI brain — the process that talks to Claude and streams back SSE.

~/mdj-agent/ package.json tsconfig.json src/ server.ts — Express, port 4200, CORS, health endpoint agent/ claude-client.ts — Anthropic SDK wrapper, messages.create with stream system-prompt.ts — Base system prompt with user context injection middleware/ auth.ts — Verify X-MDJ-Auth shared secret header .env — ANTHROPIC_API_KEY, MDJ_AUTH_SECRET

Implementation steps:

  • npm init + install: express @anthropic-ai/sdk cors dotenv firebase-admin

Track C

Permissions + Gates + UI

AIR / GA
TRK-008 Permission-scoped tool filtering New Depends on TRK-006, TRK-007

Create ~/mdj-agent/src/agent/tool-filter.ts. Called before every Claude API request to build the exact right tool list for this user.

  • filterToolsForUser(allTools, userContext):
    • Filter by required_level <= user.level
    • Filter by module_key in user's module_permissions
    • Specialist mode: if active specialist, filter to specialist's available_tools
  • Returns filtered tool array ready to pass to Claude API
TRK-009 Review gates New Depends on TRK-008

When Claude returns a tool_use block for a gated tool, pause execution and wait for human approval before proceeding.

  • Agent service: when Claude returns tool_use and requires_approval: true:
    • Emit SSE approval_required event with tool name, input preview, call_id
    • Pause execution (store state in Firestore)
    • Wait for POST /api/mdj/conversations/:id/approve or reject
  • Cloud Run: add approve/reject endpoints that forward decision to MDJ_SERVER
  • On approve: resume tool execution, return result to Claude
  • On reject: return rejection message to Claude, Claude acknowledges gracefully
TRK-010 Tool execution display (ToolCallCard) New Depends on TRK-009

Create packages/ui/src/modules/MDJPanel/ToolCallCard.tsx. Renders inline in ChatThread when a message includes tool calls.

  • Shows: tool icon, tool name, status badge (pending / running / completed / failed)
  • Expandable: click to see input parameters + result output
  • Approve / Reject buttons when status is pending and requires_approval: true
  • Approve action: POST /api/mdj/conversations/:id/approve with call_id

Track A

Specialists — Router + Prompts

MDJ_SERVER / Jr
TRK-011 Specialist routing engine New Phase 2 complete

Create ~/mdj-agent/src/agent/specialist-router.ts. Dynamically routes conversations to the right specialist based on message content.

  • Load specialist configs from Firestore mdj_specialist_configs on startup (cache in memory)
  • routeToSpecialist(message, currentSpecialist):
    • Step 1: keyword match — scan message for routing_keywords, score each specialist
    • Step 2: if top score < threshold, call Claude (Sonnet) with classify_specialist tool to confirm
    • Return specialist key or null (stay with current)
  • When specialist changes: swap system prompt, re-filter tools, emit SSE specialist_switch event
TRK-012 Seed specialist prompts New Phase 2 complete

Create services/api/src/scripts/seed-mdj-specialists.ts. Write full, production-quality system prompts for all 6 specialists.

Each prompt must include:

  • Role definition + domain expertise (e.g., MDJ-Medicare knows FIA, MYGA, MAPD, AEP, T65 terminology)
  • RPI-specific terminology from CLAUDE.md industry terms
  • PHI handling rules (always included in every specialist)
  • Available tool descriptions and when to use them
  • User context injection template: {user.display_name}, {user.level}, {page_context}

  • On next conversation start referencing this client: load insights into system prompt automatically

Track A

Infrastructure — MDJ_SERVER Hardening

MDJ_SERVER / Jr
TRK-016 MDJ_SERVER production hardening Modify Phase 3 complete

Lock MDJ_SERVER down and make it bulletproof for 24/7 operation.

  • Cloudflare Tunnel:
    • cloudflared tunnel create mdj-agent
    • Route mdj-agent.tomachina.ai → localhost:4200
    • Auth: X-MDJ-Auth header with shared secret from env var (no exposed ports)
  • /health endpoint returns: uptime, memory usage, active conversation count, MCP server status
  • systemd service configured from TRK-002 — verify auto-restart on crash (Restart=always)
  • Rate limiting: per-user, 60 requests/minute (429 response with retry-after)
  • Request logging: structured JSON to /var/log/mdj-agent/ with log rotation
  • Error notifications: POST to /api/notifications on the tm-api for critical failures
Track B

Features — Long Context + Preferences

AIR / GA
TRK-017 Long context management New Phase 3 complete

Conversations can go on forever without losing context or hitting token limits.

  • Before each Claude API call: count tokens in conversation history
  • If > 80K tokens: call Claude (Sonnet) to summarize older messages into a compact block
  • Replace old messages with the summary in the context window
  • Store summary string on mdj_conversations Firestore doc
  • New conversation with same user: load last 3 conversation summaries as initial context
TRK-018 User preferences New Phase 3 complete

Let users configure how MDJ behaves for them personally.

  • Add mdj_preferences section to existing Admin/Config tab in ProDashX
  • UI: per-user toggle cards for:
    • auto_approve_tools — skip review gate for low-risk tools
    • show_tool_details — show/hide tool call cards by default
    • default_specialist — which specialist to start with
    • retention — how long to keep conversation history
  • Store on users/{email}.mdj_preferences in Firestore
  • Agent service reads preferences on each request, applies to tool filtering + approval gates
