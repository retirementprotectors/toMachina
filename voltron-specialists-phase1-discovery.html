Sprint Plan

Two Systems. One Problem.

VOLTRON is two separate machines that were built in parallel and have never talked to each other. The voice is here. The tools are over there. Until now, they've been strangers.

System 1 — The Voice

VOLTRON Specialist Router. 6 chat personas (General, Medicare, Securities, Service, DAVID, Ops) in Firestore with full system prompts and keyword routing. VOLTRON knows who to sound like.

System 2 — The Tools

QUE Product Specialists. 5 domain specialists (Investment, Medicare, Annuity, Life/Estate, Legacy/LTC) with calculation engines, wire workflows, and real analysis capability. VOLTRON knows what it can DO.

The Disconnect in Plain English

When the Medicare Specialist persona is active, VOLTRON speaks with deep Medicare fluency — but it cannot automatically chain calcIrmaa, COVERAGE_GAP, or AEP_ENROLLMENT wires. It sounds like a Medicare expert. It cannot act like one.

When the Securities Specialist is active, VOLTRON can talk tax strategy all day. But ask it to actually run a Tax Harvesting analysis on a real portfolio and it hits a wall. The tools exist. The wire exists. The specialist has no idea.

The Real Gap

Specialist personas and QUE tool suites are two separate systems with no bridge between them. Activating a specialist persona does not unlock the corresponding tools. This sprint builds that bridge.

The Fix

Wire QUE specialist modes INTO the VOLTRON specialist personas. When a specialist persona activates, it injects the corresponding QUE tool definitions into the conversation context. The voice and the tools become one.

Before

VOLTRON specialist active → persona + system prompt only. Tools: general 82-tool set. QUE tools: unavailable.

After

VOLTRON specialist active → persona + system prompt + QUE tool injection. Specialist-specific tools auto-available. Wire chaining enabled.

Implementation Details

specialist-router.ts Modification

On specialist match, the router now makes a second Firestore read to load mode_config alongside the existing persona config. If mode_config exists, it passes it downstream. If not (General, Ops), it passes null and nothing changes — existing behavior preserved.

voltron-os-rules.ts Update

QUE specialist tools receive SAFE classification when activated by a matching specialist. The rule: if the tool is in the specialist's que_tools list and the specialist is active, it executes without approval. This is scoped — the same tool requested outside specialist context keeps its original tier.

Wire Execution Pattern

Reuses the VoltronDualMode wire selector pattern already established in apps/prodash/app/(portal)/modules/voltron/. No new UI surface required. Wire execution is handled by the agent's existing tool-chaining logic (15-round multi-step).

No Breaking Changes

All modifications are additive. Existing specialist routing, tool execution, MDJ Panel behavior, and wire execution are untouched. The bridge layers on top of what's there. If mode_config is absent, the specialist behaves exactly as before.

The New Config Format

Each specialist gets a specialist_mode_config document in Firestore. This is the bridge record that connects the routing system to the QUE tool system. One config per specialist. Loaded alongside the persona on every match.

Schema Definition

// Firestore collection: specialist_mode_configs
// Document ID: matches specialist_id (e.g., "mdj-securities")
{
  specialist_id: "mdj-securities",         // matches mdj_specialist_configs doc ID
  que_tools: [                               // QUE tool names to inject on activation
    "analyzeTaxHarvest",
    "analyzeRoth",
    "calcRmd",
    "calcPortfolioGainLoss",
    "calcCostBasisOptimize",
    "calcWashSaleRisk",
    "calcEstimatedTaxLiability",
    "calcBracketProjection",
    "calcDistributionSchedule"
  ],
  wires: [                                   // Wire workflows unlocked in this mode
    "wireTaxHarvest",
    "wireRothConversion"
  ],
  auto_chain: true,                        // when user asks, select the right wire automatically
  output_format: "pdf_branded",            // MUSASHI-designed template key
  approval_tier: "SAFE",                  // override for injected tools in this specialist context
  context_keywords: [                       // optional: additional routing hints for mode
    "tax", "harvest", "roth", "RMD",
    "portfolio", "investment", "bracket"
  ]
}

TypeScript Interface — specialist-mode-config.ts

// mdj-agent/src/agent/specialist-mode-config.ts (NEW)
export interface SpecialistModeConfig {
  specialist_id: string;
  que_tools: string[];
  wires: string[];
  auto_chain: boolean;
  output_format: 'pdf_branded' | 'structured_text' | 'none';
  approval_tier: 'SAFE' | 'DANGEROUS';
  context_keywords?: string[];
}

// Loads from Firestore with 5-minute cache (matches specialist router)
export async function loadModeConfig(
  specialistId: string
): Promise<SpecialistModeConfig | null>

Why This Config Format

The config is the bridge. It does three things simultaneously:

Declares Tools

que_tools tells the injector exactly which QUE tools to append to the agent context. No hardcoding in agent code. Adding a new tool to a specialist is a Firestore write, not a deploy.

Declares Wires

wires tells the agent which wire workflows are available in this specialist mode. auto_chain: true lets the agent pick the right wire without prompting the user to choose.

Controls Trust

approval_tier overrides the default tool classification for injected tools in this specialist context. Analysis tools that are normally SAFE stay SAFE. Writes stay DANGEROUS regardless.

Architecture — Build Once, Works Everywhere

The same template architecture as the CMO brochures. One design system. Content slots populated by VOLTRON's analysis output. Output rendered to PDF via PDF_SERVICE.

Template Structure

PDF_SERVICE Integration

Existing PDF_SERVICE handles rendering. No new infrastructure. VOLTRON calls POST /render with template ID + data payload. Gets back a signed URL to the finished PDF.

Delivery

PDF URL returned in the VOLTRON chat response. Advisor clicks, downloads, hands to client. Or: direct email delivery via RAPID_COMMS. Phase I: URL delivery. Phase II: email automation.
