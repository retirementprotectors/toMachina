What Is OPERATE?

OPERATE is Track 2 of the VOLTRON CXO lifecycle: CONSUME → OPERATE → DEVOUR. CONSUME built the foundation — the Command Center, the registry consolidation, the identity documents. OPERATE activates the machine. This is when Lions go live, cases start flowing, and VOLTRON stops being a concept and starts running client outcomes.

CONSUME asked: "Does VOLTRON know who he is?" OPERATE asks: "Can VOLTRON run a client case end-to-end without human engineering support?"

By the end of OPERATE, 5 CCSDK Lions are deployed on MDJ_SERVER, the case pipeline tracks every wire execution in Firestore, 4 intake channels feed client work into the Lions, and VOLTRON can deploy new Lion configurations without a FORGE sprint.

Why This Comes After CONSUME

OPERATE requires everything CONSUME built: the Command Center (where cases and Lion status are visualized), the domain-tagged QUE registry (so Lions know which wires belong to them), the identity documents (so Lions route consistently), and the consolidated backlog (so we know what to build first). Without CONSUME, Lions would route blind.

Why This Comes Before DEVOUR

DEVOUR requires outcome data — which wires produce correct results, which products have coverage gaps, and where accuracy needs audit. That data only exists after Lions are executing real wires on real client scenarios. OPERATE generates the signal. DEVOUR refines it.

01
Build 5 CCSDK Lion Agents
Each Lion is a CCSDK agent on MDJ_SERVER with product-specific system prompts, QUE wire configs, and domain knowledge. Build order: Medicare → Annuity → Investment → Life/Estate → Legacy/LTC.

What This Workstream Builds

5 CCSDK agents, each a product-domain specialist that reads QUE wire definitions and executes end-to-end client workflows. Every Lion gets: a system prompt built from soul.md + spirit.md + domain knowledge, a list of domain-tagged QUE wires it owns, and registration in the specialist_configs Firestore collection.

Lions ARE the specialists. The existing 5 specialist personas in the MDJ agent become formal CCSDK agents. Same domain expertise. New execution model — each Lion runs its own QUE wires autonomously and reports results back to VOLTRON.

02
Case Pipeline
Track every client case from intake to resolution in Firestore. Cases auto-created from wire execution — not manual entry. Command Center surfaces open cases per Lion domain.

Case Lifecycle

CASE LIFECYCLE Wire executes → intake | Case doc created in voltron_cases | Fields: client_id, wire_name, lion_domain, agent_id, timestamp v Lion processes → wire_running | Lion executes QUE wire end-to-end | Output accumulates in case.wire_output v Wire completes → output_ready | Wire output written to case record | Notification sent to assigned agent v Agent reviews → agent_review | Agent marks outcome: accepted | revised | escalated | Revision notes captured for QUE improvement backlog v Case closed → resolved Wire output accepted. Case archived. Feeds DEVOUR analytics.

03
Intake Channels
4 entry points for client work to reach VOLTRON's Lions. Each channel creates a voltron_cases record and dispatches to the appropriate Lion.

4 Intake Channels

MDJ Panel (Primary)

Existing chat interface in all 3 portals. Agent describes client situation → VOLTRON identifies product domain → dispatches to Lion → output returned to chat.

  • Source: packages/ui/src/modules/MystAI/
  • Routing: services/api/src/routes/mdj.ts
  • Priority: Enhance existing

Slack Channel

Team drops client name + scenario in dedicated #voltron-cases channel → VOLTRON picks up → runs wire → posts output as threaded reply.

  • Channel: Create #voltron-cases
  • Monitoring: Cron on MDJ_SERVER, 5 min interval
  • Priority: New build

Email Forward

Forwarded client email → VOLTRON parses → identifies product type → routes to Lion → drafts response.

  • Inbox: voltron@ alias (JDM creates)
  • Monitoring: Gmail MCP, 10 min interval
  • Priority: New build (needs JDM)

Command Center “Run Wire”

Agent selects client + wire from dropdowns → Lion executes → case created. Direct-execution path for agents who know what they need.

  • UI: New panel in Command Center
  • API: POST /api/voltron-wire/execute
  • Priority: Extend existing

All 4 channels feed the same pipeline. Every intake creates a voltron_cases document with the same schema. The intake_channel field records origin. Command Center shows all work from all sources in a single view.

04
Client Outcome Tracking
Close the loop on every Lion execution. Agent marks outcome. Revision notes feed QUE improvement. CXO dashboard shows outcome rates.

Outcome Actions

OutcomeMeaningWhat Happens
AcceptedWire output was correct. Agent delivered to client.Case resolved. No further action.
RevisedOutput partially correct, agent modified before delivery.Revision notes captured. Feeds QUE improvement backlog.
EscalatedOutput wrong or case too complex for the Lion.Escalation notes captured. May trigger new tool request.

CXO Dashboard Metrics

  • Acceptance Rate by Lion: % of cases per Lion where outcome = accepted. Target: >85%.
  • Acceptance Rate by Wire: Which wires produce correct results? Which need refinement?
  • Escalation Trends: Are escalations decreasing over time?
  • Resolution Time: Average time from intake to resolved per Lion domain.
  • Revision Note Patterns: Common reasons → prioritized QUE improvement tickets.

05
CXO Self-Sufficiency
VOLTRON as CSO should not require a FORGE sprint to manage his domain. Lion config UI, wire admin, and gap request workflow.

This is what makes VOLTRON a CXO, not just an agent. An agent uses tools. A CXO manages their own tools, monitors outcomes, and requests new capabilities — all without blocking on engineering.

Lion Config UI

Admin panel in Command Center for managing Lion specialist configs. Update system prompts, wire assignments, domain knowledge, enabled/disabled status.

  • CRUD on specialist_configs Firestore
  • Prompt editor with live preview
  • Wire assignment multi-select
  • Enable/disable toggle per Lion

Wire Admin Panel

View all QUE wires by domain with execution stats: total runs, success rate, last executed. Flag wires for improvement.

  • Wire list with execution stats
  • Flag for improvement (creates backlog)
  • Domain filter from CONSUME tagging

Gap Request Workflow

When VOLTRON identifies a missing wire during case processing, create a structured gap request from Command Center. Domain, description, scenario, expected output.

  • Firestore: voltron_gap_requests
  • Surfaces in War Room for CTO review
  • Input for MUSASHI Discovery Docs
