MUS-O01 Wire Executor for CMO Wires

Create the generic wire execution engine that powers all three CMO wires. The executor takes a CmoWireDefinition + typed input params, walks each step, calls the appropriate tool, logs step output, and halts on failure with a structured error report.

File Paths

packages/core/src/cmo/wire-executor.ts — NEW packages/core/src/cmo/types.ts — UPDATE (add WireExecutionResult, WireStepResult)

Acceptance Criteria

  • WireStepResult interface: stepId, toolId, success: boolean, output?: unknown, error?: string, durationMs: number
  • WireExecutionResult interface: wireId, success: boolean, steps: WireStepResult[], completedAt?: Date, haltedAt?: string (stepId of failure), error?: string
  • executeWire(wire: CmoWireDefinition, input: Record<string, unknown>): Promise<WireExecutionResult> exported as named export
  • Each step logs: [MUSASHI][WIRE_ID][stepId] executing toolId... to console (NOT PHI-bearing data)
  • On step failure: remaining steps are skipped, haltedAt is set, success: false, error message from step propagated
  • On full success: success: true, all steps have success: true, completedAt set
  • Executor is tool-agnostic — it receives a step executor callback via dependency injection (pattern: executeWire(wire, input, toolRunner)). This allows unit testing without live MCP calls.
  • All types exported from packages/core/src/cmo/types.ts
  • npm run type-check passes after this ticket

Reference Pattern

Follow ATLAS wire executor pattern at packages/core/src/atlas/wire-executor.ts if it exists. If not, follow the flow engine step execution pattern at packages/core/flow/. The key principle: steps are sequential, each step receives the previous step's output as part of its input context.

  • Unit: call executeWire with a mock wire (2 steps) and a mock toolRunner that always succeeds → returns success: true, 2 step results
  • Unit: call executeWire with a mock wire where step 2 fails → returns success: false, haltedAt: 'step-2', step 3 never called
  • Type check: npm run type-check 13/13 pass

MUS-O02 WIRE_BROCHURE Live Execution

Upgrade WIRE_BROCHURE.ts from a stub step definition to a fully executable wire. The Print Artisan wire integrates Canva MCP (design generation + export) + PDF_SERVICE (print rendering) + Drive (asset archival).

File Paths

packages/core/src/cmo/wires/WIRE_BROCHURE.ts — UPDATE (stub → live executor)

Wire Input Schema

interface BrochureInput { market: string; // e.g. "Medicare AEP 2026", "Annuity — Retirement Income" product: string; // e.g. "FIA", "MYGA", "MAPD" target: string; // e.g. "T65 prospects", "existing clients age 70+" tone: string; // e.g. "professional", "warm", "urgent" brandKitId?: string; // override default brand kit (optional) }

Step Implementation

  • Step 1 — canva_generate_design: Call mcp__rpi-workspace__canva_generate_design. Pass market + product as design context. Use brand kit from brandKitId or default from canva_list_brand_kits. Store returned designId in step output.
  • Step 2 — canva_export_design: Call mcp__rpi-workspace__canva_export_design with designId from step 1. Format: PDF. Store export URL in step output.
  • Step 3 — PDF_SERVICE render: Call the PDF_SERVICE render endpoint (via services/api/src/routes/pdf.ts if available, or direct service call). Apply print spec: bleed 0.125in, color space CMYK, resolution 300dpi. Store rendered PDF URL in step output.
  • Step 4 — Drive archive: Upload final PDF to CMO asset folder on Shared Drive (folder ID from env var CMO_ASSETS_DRIVE_FOLDER_ID). File name: [market]-[product]-brochure-[timestamp].pdf. Store Drive file URL in step output as the wire's final output.

Acceptance Criteria

  • Given valid BrochureInput, wire executes all 4 steps in sequence
  • Step 1 output (designId) is available to step 2 via execution context
  • Step 2 output (export URL) is available to step 3
  • Step 3 output (print-spec PDF URL) is available to step 4
  • Final step output is the Drive file URL of the registered brochure asset
  • If Canva returns error (e.g. brand kit not found): wire halts at step 1, returns WireExecutionResult with success: false, haltedAt: 'canva-generate'
  • If PDF_SERVICE unavailable: wire halts at step 3, reports error clearly
  • CMO_ASSETS_DRIVE_FOLDER_ID read from environment, not hardcoded
  • No PHI in any log statements

Reference Pattern

Wire executor from MUS-O01 handles the step loop. This ticket provides the step implementations (the toolRunner callbacks for each step). Look at how rpi-workspace Canva tools are called in any existing Canva integration in the codebase for parameter patterns.

Test Plan

  • Integration: call with valid BrochureInput → verify Drive file URL in final step output
  • Error path: pass invalid brandKitId → verify wire halts at step 1 with descriptive error
  • Log verification: no PHI strings appear in console output during execution

Dependencies

  • MUS-O01 (wire executor engine)
  • MUS-C08 (WIRE_BROCHURE stub from Track 1 — this ticket upgrades it)
  • rpi-workspace MCP Canva tools (live)
  • PDF_SERVICE running at services/api or standalone
  • Drive MCP or gdrive MCP (for asset archival)
  • Env var: CMO_ASSETS_DRIVE_FOLDER_ID set in Cloud Run secrets
MUS-O03 WIRE_CAMPAIGN Live Execution

Upgrade WIRE_CAMPAIGN.ts from stub to live executor. The Digital Artisan wire creates a C3 campaign, targets an audience, schedules it, and executes the send — all via the toMachina API C3 routes.

File Paths

packages/core/src/cmo/wires/WIRE_CAMPAIGN.ts — UPDATE (stub → live executor)

Wire Input Schema

interface CampaignInput { market: string; // e.g. "Medicare", "Life", "Annuity" templateId: string; // C3 template ID (must exist in CMO_REGISTRY or Firestore) audience: { segment: string; // e.g. "T65", "existing-clients-age-70-plus" filters?: Record<string, unknown>; // optional Firestore query filters }; schedule: { type: 'drip' | 'blast'; startAt: string; // ISO 8601 datetime endAt?: string; // drip only cadenceDays?: number; // drip only }; sendChannels: Array<'email' | 'sms'>; }

Step Implementation

  • Step 1 — campaigns create: POST to /api/campaigns. Payload: { templateId, market, name: "[market]-[timestamp]" }. Store returned campaignId in step output. Read services/api/src/routes/campaigns.ts for exact field names before writing this step.
  • Step 2 — audience targeting: POST to /api/campaigns/:campaignId/audience. Payload: audience object from input. Store targeting confirmation in step output.
  • Step 3 — schedule: POST to /api/campaigns/:campaignId/schedule. Payload: schedule object from input. AEP blackout enforcement: if startAt is Oct 1 – Dec 7 and market is "Medicare", return { success: false, error: "AEP Blackout: Medicare campaigns blocked Oct 1 – Dec 7" } without proceeding.
  • Step 4 — execute send: POST to /api/campaigns/:campaignId/send. Payload: { channels: sendChannels }. Store send execution result in step output as the wire's final output.

Acceptance Criteria

  • Given valid CampaignInput, wire creates campaign, targets, schedules, and sends
  • campaignId from step 1 is passed to steps 2, 3, 4 via execution context
  • AEP Blackout enforced at step 3: Medicare campaigns during Oct 1 – Dec 7 halt gracefully with clear error
  • Non-Medicare campaigns during AEP window proceed normally
  • Step 4 final output includes: campaignId, status, recipientCount, channels
  • If templateId not found at step 1: wire halts, error clearly states template lookup failed
  • API contract verified: field names in step payloads match services/api/src/routes/campaigns.ts exactly

Reference Pattern

Read services/api/src/routes/campaigns.ts before writing ANY step payload. Field names must match exactly — this is a hard contract verification requirement (see CLAUDE.md API Contract Verification rule). Also verify C3 campaign routes for the audience targeting and schedule endpoints.

Test Plan

  • Integration: execute with a valid template ID → campaign created, audience targeted, scheduled, send triggered
  • AEP gate: execute with Medicare + startAt in October → halts at step 3 with AEP Blackout error
  • AEP gate: execute with Life market + startAt in October → proceeds to step 4 normally
  • Missing template: execute with non-existent templateId → halts at step 1, descriptive error

MUS-O04 WIRE_LANDING_PAGE Live Execution

Upgrade WIRE_LANDING_PAGE.ts from stub to live executor. The Web Artisan wire creates a WordPress page draft, applies Elementor layout, uploads media assets, and publishes the page live via the rpi-workspace WordPress MCP tools.

File Paths

packages/core/src/cmo/wires/WIRE_LANDING_PAGE.ts — UPDATE (stub → live executor)

Wire Input Schema

interface LandingPageInput { market: string; // e.g. "Medicare", "FIA Rollover" purpose: string; // e.g. "lead capture", "product education", "event registration" content: { title: string; headline: string; subheadline: string; bodyText: string; ctaText: string; ctaUrl: string; }; design: { elementorTemplateId?: string; // existing Elementor template ID (optional) heroImageUrl?: string; // pre-existing image URL to upload (optional) colorScheme?: 'rpi-blue' | 'rpi-gold' | 'neutral'; // maps to WP template variants }; slug: string; // URL slug for the page (e.g. "medicare-2026-fia") }

Step Implementation

  • Step 1 — wordpress_create_page: Call mcp__rpi-workspace__wordpress_create_page. Payload: { title: content.title, status: 'draft', slug, content: content.bodyText }. Store returned pageId in step output.
  • Step 2 — wordpress_update_elementor_element: Call mcp__rpi-workspace__wordpress_update_elementor_element with pageId from step 1. If design.elementorTemplateId provided, apply that template. Otherwise apply the default colorScheme layout. Store confirmed layout in step output.
  • Step 3 — wordpress_upload_media: If design.heroImageUrl is provided, call mcp__rpi-workspace__wordpress_upload_media. Store returned mediaId in step output. If no image provided, skip step (mark step as skipped, not failed).
  • Step 4 — wordpress_update_page (publish): Call mcp__rpi-workspace__wordpress_update_page with { id: pageId, status: 'publish' }. If step 3 produced a mediaId, set it as the featured image. Store final page URL in step output as the wire's final output.

Acceptance Criteria

  • Given valid LandingPageInput, wire creates draft → applies layout → uploads media → publishes
  • pageId from step 1 flows into steps 2, 3, 4 via execution context
  • Step 3 is optional: if heroImageUrl not provided, step is skipped (not failed) and wire continues to step 4
  • Final output includes: pageId, pageUrl, status: 'publish'
  • Slug conflict (page with that slug already exists): wire halts at step 1 with clear error — do NOT overwrite existing pages
  • Elementor template not found: wire halts at step 2 with clear error
  • Published page URL matches: [wordpress-site-url]/[slug]

Reference Pattern

Read mcp__rpi-workspace__wordpress_create_page and mcp__rpi-workspace__wordpress_update_elementor_element tool definitions from the rpi-workspace MCP server before writing step implementations. Elementor data structure is non-trivial — use wordpress_get_elementor_data on an existing page as a reference for what the update payload should look like.

  • Minimal: execute without heroImageUrl or elementorTemplateId → steps 1+4 succeed, step 3 skipped
  • Slug conflict: execute with a slug that already exists → halts at step 1 with clear error
  • Visual: confirm published page is reachable at expected URL

MUS-O07 Content Pipeline Orchestrator

Build the pipeline orchestrator that takes a creative brief, evaluates which Artisans to activate, dispatches to their wire executors in the correct order (parallel where possible), collects results, and returns a unified pipeline result with tracking ID.

File Paths

packages/core/src/cmo/pipeline.ts — NEW packages/core/src/cmo/types.ts — UPDATE (add CmoBrief, CmoPipelineResult, CmoPipelineJob)

Key Types

interface CmoBrief { id: string; // auto-generated UUID market: string; channels: CmoChannel[]; // which artisans to activate inputs: { print?: BrochureInput; digital?: CampaignInput; web?: LandingPageInput; }; requestedBy: string; // user email createdAt: Date; } interface CmoPipelineJob { jobId: string; // auto-generated UUID briefId: string; artisanId: string; wireId: string; status: 'queued' | 'running' | 'complete' | 'failed'; result?: WireExecutionResult; startedAt?: Date; completedAt?: Date; } interface CmoPipelineResult { briefId: string; jobs: CmoPipelineJob[]; overallStatus: 'complete' | 'partial' | 'failed'; completedAt?: Date; }

Acceptance Criteria

  • processBrief(brief: CmoBrief): Promise<CmoPipelineResult> exported as named export
  • For each channel in brief.channels, looks up the Artisan via getArtisanByChannel, dispatches to the wire executor
  • Jobs for independent channels run in parallel (Promise.allSettled)
  • Each job tracks status transitions: queued → running → complete/failed
  • overallStatus: 'complete' if all jobs succeed, 'partial' if some succeed and some fail, 'failed' if all fail
  • Returns CmoPipelineResult with all job results regardless of individual job success/failure (never throws)
  • If brief.channels includes a channel with no corresponding Artisan, that channel is skipped with a warning in the job result
  • Brand compliance check (MUS-O08) is called on each completed job output BEFORE marking job as complete
  • Types exported from packages/core/src/cmo/types.ts

Reference Pattern

Use Promise.allSettled (not Promise.all) for parallel dispatch — we want all results even if some fail. Pattern for job tracking: each job has its own status object that is mutated as steps complete. This is internal state within the function call; no Firestore writes in this file (the API route handles persistence).

MUS-O08 Brand Compliance Check Hook

Every wire output passes through a brand compliance check before final publish or registration. The check validates logo usage, color palette, font family, and tone guide adherence. Non-compliant outputs halt publish and return a structured compliance report.

File Paths

packages/core/src/cmo/compliance.ts — NEW packages/core/src/cmo/types.ts — UPDATE (add BrandComplianceResult, BrandComplianceViolation)

Brand Standards (RPI)

  • Logo: Must be present in print/web outputs. Accepted: prodashx-mark.svg, tomachina-logo variants from packages/ui/src/logos/
  • Color palette: Primary blue #4a7ab5, gold #d4a44c, SENTINEL green #40bc58, RIIMO purple #a78bfa. No off-brand hex values in output metadata.
  • Font: No hardcoded font families in generated content (use CSS variables or Canva brand kit fonts)
  • Tone: Flag outputs where tone param was overridden or not respected (heuristic check on output metadata)

Acceptance Criteria

  • BrandComplianceViolation interface: rule, severity: 'error' | 'warning', description, fix?: string
  • BrandComplianceResult interface: passed: boolean, violations: BrandComplianceViolation[], checkedAt: Date
  • checkBrandCompliance(wireId: string, output: WireExecutionResult): BrandComplianceResult exported
  • Returns passed: true if zero 'error' severity violations (warnings are allowed)
  • Returns passed: false if any 'error' severity violation exists
  • WIRE_BROCHURE check: verifies Canva brand kit was used (from step 1 output metadata)
  • WIRE_CAMPAIGN check: verifies template ID was an approved template (not an ad-hoc creation)
  • WIRE_LANDING_PAGE check: verifies page was assigned to an approved color scheme
  • Unknown wireId: returns { passed: true, violations: [{ rule: 'unknown-wire', severity: 'warning', description: 'No compliance rules defined for this wire' }] }
  • Function is pure — no API calls, no Firestore reads. Compliance rules are hardcoded config in this file.

Test Plan

  • WIRE_BROCHURE output with brandKitId present → passed: true
