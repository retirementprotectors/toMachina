4. TRK-13602: Client Dedup — 149 Clusters

TRK-13602 Client Duplicate Detection + Merge acf-client-dedup.ts

What

New script acf-client-dedup.ts detects and merges 149 client duplicate clusters using the existing matching library.

How It Works

  1. Read all clients from Firestore clients collection
  2. Run findDuplicates(threshold=85) to identify duplicate clusters
  3. For each cluster, run matchClient for detailed tier scoring
  4. Output dry run report: all proposed merges with scores, field comparisons
  5. JDM reviews dry run report (review gate)
  6. On approval: npx tsx acf-client-dedup.ts --commit executes merges

ACF folder referencesUpdate ACF folder mappings to primary client_id

6. TRK-13604: Orphan Resolution — 1,294

TRK-13604 Orphan Account → Client Matching acf-orphan-resolve.ts

What

New script acf-orphan-resolve.ts finds accounts where the parent client doesn’t exist and matches them to valid clients using 5-tier scoring.

How It Works

  1. Scan all accounts — identify those where parent client_id doesn’t exist
  2. For each orphan, run matchClient 5-tier matching against all clients
  3. Score ≥ 85: auto-link to matched client
  4. Score 50–84: flag for manual review by JDM
  5. Score < 50: mark as unresolvable (logged, no action)
  6. Output dry run report: all proposed links with confidence scores
  7. npx tsx acf-orphan-resolve.ts --commit for live execution

What

New script acf-extract-batch.ts runs SUPER_EXTRACT in vision mode on all ~1,807 ACF documents, extracting structured data into Firestore.

How It Works

8. TRK-13606: Client Enrichment

TRK-13606 3-Phase Client Enrichment Pipeline acf-enrich-clients.ts

What

New script acf-enrich-clients.ts runs a 3-phase enrichment pipeline to fill data gaps on client records.

Phase 1: Backfill from Extractions

Source: Firestore extractions collection (from TRK-13605)

Strategy: Fill blanks only — never overwrite existing data

Fields: DOB, SSN4, address, phone, email, Medicare #, spouse

Pre-flight: Run check_whitepages_usage() to verify quota before bulk requests

Targets: Only clients still missing phone, email, or address after Phase 1

Strategy: Fill blanks only — never overwrite

Phase 3: SUPER_NORMALIZE

Engine: 90+ normalizers, 16 types (names, phones, addresses, dates, SSN, etc.)
