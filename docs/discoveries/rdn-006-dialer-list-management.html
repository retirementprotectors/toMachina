The Problem

The dialer exists only inside ProZone. Vince's team runs calls from every surface — Contacts, client profiles, campaign views — and every one of those surfaces forces a manual phone copy-paste. That is an agent productivity tax we keep charging ourselves.

The Core Gap
The autodialer is trapped in ProZoneApp.tsx. It cannot survive a route change. If an agent navigates away mid-session — intentionally or not — the session state is gone. Vince's team has learned to stay on the ProZone page like it is a paper form they cannot put down.

What Vince's Team Asked For

Requests from #raiden-reactive
  • Request 1 — Global Dialer: "Can we launch call sessions from the main Contacts list, not just ProZone? We do carrier service calls from there and it would save a lot of time."
  • Request 2 — Carrier Filter: "We need to pull a list of all clients on a specific carrier so we can run mass service calls when a carrier makes an update. Right now we do it manually in the Contacts page."
  • Request 3 — ProZone Custom Lists: "Can we save a filtered list in ProZone as a named list and launch it as a session? We keep re-building the same filtered views."
Also Found — Call Initiation Bug (P1)
During discovery, a confirmed bug was found in CallPanel.tsx ~line 155. When useTwilioDevice().isReady === false, the start-call function fires POST /api/comms/send-voice with TwiML that dials the prospect directly — without ever connecting the agent's browser. The prospect's phone rings. The agent hears nothing. The call is logged as if connected. This ships in Phase 1 as RDN6-04.

User Stories

  • As Vince's team, I want to select clients on the Contacts page and launch a call session without going to ProZone.
  • As a service rep, I want to filter all clients on UnitedHealthcare and launch a mass service call session in one click.
  • As a ProZone user, I want to save a filtered TARGET view as "Medicare Follow-Up June" and launch it next week without rebuilding filters.
  • As an agent in an active session, I want to go back to a prospect I already passed if they call back.

Non-Goals

That is acceptable for Phase 2 — the carrier filter uses whatever Account.carrier string values exist in Firestore today.

Current State

Where the Dialer Lives Today

The dialer is fully contained inside packages/ui/src/modules/ProZone/. Three files own the entire stack:

  • useCallSession.ts — State machine hook: idle/active/paused/ended, stats accumulation, session lifecycle
  • CallSession.tsx — UI wrapper: stats bar, pause/resume/end controls, progress indicator
  • CallPanel.tsx — Individual call UI: pre_call → ringing → calling → disposition state machine

ProZoneApp.tsx calls useCallSession() directly and renders CallSession at the bottom of its JSX. Session state is local to ProZoneApp — navigate away, state is gone.

apps/prodash/app/(portal)/layout.tsx already wraps the entire portal in TwilioDeviceProvider. The useTwilioDevice() hook is available everywhere in ProDash. This is the right architecture — we build on it, not around it.

Carrier Filter — Backend Architecture

Carrier values live on Account documents in the clients/{id}/accounts subcollection, not on the client document itself. There is no top-level field to filter by. Three approaches:

Approach A — Denormalized field: Add carrier_names: string[] to the client document. Backfill via migration script. Filter client-side via array-contains. Requires bridge service to maintain in sync. Clean and fast once deployed.

Approach B — Server-side aggregation: New API endpoint GET /api/clients/carrier-options aggregates distinct carrier values from accounts subcollection via Firestore collection group query. Returns carrier list + optional client_ids pre-filtered. No schema migration.

Approach C — Client-side subcollection reads: For every client in the list, fetch their accounts subcollection and filter locally. N+1 problem. Unacceptable at scale.

Approach B is implemented in this sprint. Approach A is the long-term answer but requires a data migration sprint. SHINOB1 must choose — see Decisions tab.

prozone_lists Collection

New top-level Firestore collection for custom saved lists:

prozone_lists/{list_id} {
  list_id: string
  name: string
  description?: string
  owner_type: 'specialist' | 'agent'  // see Decision 1
  owner_id: string
  prospect_ids: string[]              // client_id array
  created_at: Timestamp
  updated_at: Timestamp
  source_filters?: Record<string, unknown>  // snapshot of filter state
}
goToPrevious() — State Machine Addition

New action in useCallSession.ts. Decrements currentIndex if currentIndex > 0. Does NOT modify stats (completed call counts are immutable once logged). Does NOT undo the disposition log entry written to Firestore — audit logs are append-only. Simply moves the cursor back so the agent can place another call to the same prospect.

Decisions Needed

Three decisions must be made before RONIN starts Phase 2. Phase 1 has no open decisions — RONIN can start immediately.

⚠ Decision 1 — Custom list ownership: per-specialist or per-agent?
The prozone_lists collection needs an owner model. ProZone currently has specialist configs (SpecialistConfig.specialist_email) but also relates to individual agent users. Should a list belong to a specialist config (shared across whoever holds that territory) or to the individual authenticated user (personal to the agent)?
A Per-authenticated-user (owner_id = user.uid). Lists are personal. If an agent leaves, their lists go with them. Simplest auth model — Firestore rule is just request.auth.uid == resource.data.owner_id.
B Per-specialist-config (owner_id = config_id). Lists travel with the territory. A new agent who inherits a territory gets the old agent's lists. More complex auth — need to check if user has a SpecialistConfig referencing that config_id.
Rec Option A. Simpler to build and audit. Agents curate their own lists. Territory-level list sharing can be added in a future sprint if Vince asks for it.
⚠ Decision 2 — Carrier filter scope: active accounts only or all accounts?
When filtering by carrier, should the system include clients who have an account with that carrier in any status (including terminated, lapsed, surrendered) — or only active/in-force accounts?
A Active accounts only (status not in ['terminated', 'lapsed', 'surrendered']). Vince's use case is service calls to current clients — terminated accounts are irrelevant noise.
B All accounts regardless of status. Allows filtering former-carrier clients (e.g., for win-back campaigns). Adds noise for service call use case.
Rec Option A for the initial filter. A "Include inactive" checkbox can be added later if the team asks for it.
⚠ Decision 3 — SHINOB1: carrier filter backend architecture (Approach A vs B)
See Architecture tab for full details. Short version: Approach A (denormalized carrier_names[] on the client doc) is fast and clean but requires a data migration sprint. Approach B (collection group aggregation endpoint returning pre-filtered client IDs) ships in this sprint with no migration. This sprint implements Approach B. SHINOB1 must confirm this is acceptable as the permanent approach OR flag that a migration sprint should be scheduled to implement Approach A before this code ships.
B (default) Ship Approach B in this sprint. Accept that carrier filtering has a 1-2 second latency for large client lists. Schedule Approach A migration as a separate ZRD- ticket.
Block Do not ship carrier filter until Approach A migration is complete. Phase 2 ships without carrier filter. Carrier filter added in a later sprint after migration.
