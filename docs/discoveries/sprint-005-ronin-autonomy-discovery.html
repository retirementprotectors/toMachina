Why Sprint 001 Failed — and What Sprint 005 Fixes

The Core Problem

RONIN (Sprint-000) shipped. Sprint 001 ran. Eight bugs hit in sequence. Every bug required a GA-level debug session to diagnose, patch, and re-run. By the time Sprint 001 completed, we had patched RONIN's engine during the flight — a plane repaired while airborne.

Sprint 005 is the maintenance bay. We land the plane, fix the root causes architecturally, and get RONIN to a state where it can fly unassisted.

The GA Sprint Concept

This sprint is NOT built by RONIN. It is built by the GA (Shinobi) directly — because it modifies RONIN's own engine. A surgeon cannot operate on themselves.

Every ticket below has exact file paths, exact function signatures, and exact code changes. There is zero ambiguity. The GA reads this document and executes. No planning phase. No discovery phase. This document IS the plan.

The Architectural Insight (JDM, 2026-03-26)

FORGE has two distinct ticket types that have always been conflated:

  • Sprint stage statuses — Discovery, Plan, Audit, Build, Ship — proactive, structured, RONIN's domain
  • Reactive statuses — Queue, Backlog, Bug, Not Touched, Deferred, Won't Fix — reactive, unstructured, RAIDEN's domain

Zero overlap. The board status already encodes the routing. This insight resolves a longstanding confusion about what RONIN should pick up. Sprint 005 documents this split in FORGE_STANDARDS.md and wires it into how RONIN selects tickets.

The 8 Bugs We Hit Running Sprint 001

All 8 are patched in the current codebase. Sprint 005 addresses the architectural root causes so this class of failure cannot recur.

Bug 1 — Discovery Doc URL Ignored

What happened: We passed a Discovery Doc URL in the POST body. runDiscovery() regenerated a blank discovery from scratch instead of using it.

Root cause: The URL was stored in state.artifacts['discovery_doc'] but runDiscovery read from opts.discovery_doc, which was only available in-memory at run start — not re-read from state on resume.

Fix in code: runDiscovery now checks opts.discovery_doc first and returns early if present, skipping regeneration.

Architectural fix (S005): runSeedDiscovery phase reads the URL from state.artifacts['discovery_doc'] directly — decoupled from opts entirely. No opts needed post-seeding.

Bug 2 — Artifacts Never Saved to Firestore

What happened: state.artifacts was always empty when any phase after creating_sprint ran.

Root cause: recordPhaseResult() only persisted artifactPath and planPath from result.result — it never wrote back to state.artifacts in-memory after the Firestore update. The next phase read stale local state.

Fix in code: After recordPhaseResult(), the run loop now syncs artifacts back into local state: if (phaseArtifacts?.artifactPath) state.artifacts['discovery'] = ...

Architectural fix (S005): getTickets() helper always falls back to getSprintTickets(state.sprint_id) — phases are never dependent on in-memory opts passing correctly.

Bug 3 — sprint_id Never Persisted

What happened: Every phase after creating_sprint had state.sprint_id = ''.

Root cause: runCreateSprint returned sprint_id in result.result. recordPhaseResult() wrote it to Firestore. But local state.sprint_id was never updated — next phases called getSprintTickets('').

Fix in code: if (phaseArtifacts?.sprint_id) state.sprint_id = phaseArtifacts.sprint_id added to the run loop after recordPhaseResult.

Architectural fix (S005): TRK-S05-005 — getTickets() consolidates all ticket fetching. One place to fix, all phases benefit.

Bug 4 — In-Memory State Not Synced After recordPhaseResult

What happened: Firestore had correct data. Local state object was stale. Phases read local state.

Root cause: recordPhaseResult() wrote to Firestore via update() but returned void — local state was never refreshed. Same issue exists in _runFromExisting().

Fix in code: Explicit sync after recordPhaseResult for sprint_id, artifactPath, planPath in both run() and _runFromExisting().

Architectural fix (S005): getTickets() always re-fetches from Firestore by sprint_id — doesn't trust local opts.tickets.

Bug 5 — No /forge/resume Endpoint

What happened: We called POST /forge/resume/:id to restart a stalled sprint. Got 404. The endpoint didn't exist — only /forge/run/:id existed.

Root cause: routes.ts was written with /run/:id doing resume logic, but the semantically correct URL was never added.

Fix in code: Currently requires using /forge/run/:id (undocumented workaround).

Architectural fix (S005): TRK-S05-006 — add POST /forge/resume/:id as the canonical resume endpoint.

Bug 6 — claude Binary Not in systemd PATH

What happened: claudeInvoke() shell-execs claude. The mdj-agent.service systemd unit had a restricted PATH that didn't include /home/jdm/.local/bin where claude lives.

Root cause: systemd units start with a clean environment. User-level PATH from .bashrc/.profile is not inherited.

Fix in code: Added Environment=PATH=/home/jdm/.local/bin:/usr/local/bin:/usr/bin:/bin to the systemd unit.

Architectural fix (S005): TRK-S05-011 — document this in .env and verify in smoke test. The fix is a deployment concern, not a code concern.

Bug 7 — opts.tickets Empty in All Phases

What happened: Build phase got zero tickets. Every phase that consumed opts.tickets ?? [] got an empty array because tickets were never passed in opts — they were created in Firestore during creating_sprint.

Root cause: The Sprint 001 POST body didn't include a tickets array — it included a Discovery Doc URL. Tickets were meant to be seeded from the doc. No seeding phase existed.

Fix in code: runBuild and runAuditDiscovery already had a fallback: if (ticketList.length === 0 && state.sprint_id) { const res = await getSprintTickets(state.sprint_id)... }. This pattern was not consistent across all phases.

Architectural fix (S005): TRK-S05-005 — getTickets() helper enforces this fallback everywhere. All phases call getTickets(state, opts).

Bug 8 — Audit Prompts Didn't Force Verdict Format

What happened: parseAuditResult() looked for AUDIT PASSED on its own line. Haiku returned it inline: "The audit has AUDIT PASSED all checks." — failed the regex.

Root cause: Prompt said "end your response with AUDIT PASSED" but Haiku embedded it in prose. parseAuditResult() regex was line-anchored.

Fix in code: parseAuditResult() updated to strip leading non-alpha chars before pattern matching, making it more tolerant. Prompts now say "The LAST LINE of your response MUST be exactly: AUDIT PASSED".

Architectural fix (S005): TRK-S05-003 — buildSeedDiscoveryPrompt enforces JSON-only output for the seeding phase, no prose. Same discipline applied to all audit prompts in TRK-S05-008.

The New FORGE Lifecycle — #LetsSeedTheDiscovery

The fundamental gap in Sprint 001: RONIN had no way to bootstrap itself from a Discovery Doc. The doc existed. The tickets didn't. RONIN needed a human to pre-create tickets or pass them in the POST body. That's not autonomous.

The fix: A new seeding_discovery phase that runs immediately after sprint creation. RONIN reads the Discovery Doc, extracts the Sprint Plan ticket table using Haiku (cheap), and creates the tickets in Firestore. From that point forward, every phase fetches tickets from Firestore — no human ticket input required.

The Architectural Split: RONIN vs RAIDEN

JDM identified this on 2026-03-26: FORGE tickets are not all the same. The board status tells you who owns the ticket. This was implicit before. Sprint 005 makes it explicit.

RONIN's Domain — Sprint Stages

Proactive, structured work. RONIN picks these up and runs the full FORGE lifecycle.

StatusMeaning
in_discoveryBeing researched
in_planBeing planned
in_sprintActive sprint work
builtCode written, awaiting audit
in_reviewIn audit phase
shippedPR merged, CI passed
confirmedDeployed, verified live

Rule: A ticket with a sprint stage status belongs to an active FORGE run. RONIN created it; RONIN finishes it.

RAIDEN's Domain — Reactive Statuses

Reactive, unstructured work. RAIDEN handles these on-demand, without the full FORGE lifecycle.

StatusMeaning
queueWaiting to be prioritized
backlogAcknowledged, not scheduled
bugDefect report — needs fix
not_touchedCreated but untriaged
deferredExplicitly postponed
wont_fixClosed without action

Rule: A ticket with a reactive status is waiting for a human (or RAIDEN) to triage and act. RONIN never picks these up automatically.

Why This Matters for Sprint 005

The seeding_discovery phase creates tickets with status not_touched initially, then immediately transitions them to in_discovery. This is the handoff point from the reactive domain to the sprint domain. Once a ticket is in_discovery, it belongs to RONIN's lifecycle.

Future RAIDEN work: RAIDEN watches for bug status tickets, triages them, and either promotes to queue for a future sprint or self-resolves. No FORGE lifecycle required — just a targeted Sonnet fix + PR.

Board Routing Logic (Post-S005)

function routeTicket(ticket) {
  const sprintStages = ['in_discovery','in_plan','in_sprint','built','in_review','shipped','confirmed']
  const reactiveStages = ['queue','backlog','bug','not_touched','deferred','wont_fix']

  if (sprintStages.includes(ticket.status)) return 'RONIN'   // active FORGE run owns it
  if (reactiveStages.includes(ticket.status)) return 'RAIDEN' // reactive queue
  return 'UNKNOWN' // should never happen
}

async function getTickets(
  state: ForgeRunState,
  opts: ForgeRunOpts
): Promise<Array<Record<string, unknown>>> {

Change 2: Add runSeedDiscovery()

New exported phase function. Reads Discovery Doc URL from state.artifacts['discovery_doc'], invokes Haiku to extract tickets as JSON, calls createTrackerItems(). Also scans for data-cwd attribute and stores in state.artifacts['cwd'].

export async function runSeedDiscovery(
  state: ForgeRunState,
  opts: ForgeRunOpts
): Promise<ForgePhaseResult> {
  if (opts.testMode) {
    return {
      success: true,
      result: { seeded: 3, cwd: '/home/jdm/mdj-agent' },
      output: '[TEST MODE] Seeded 3 test tickets.',
    }
  }

  const discoveryUrl = (state.artifacts as Record<string, string>)['discovery_doc']
  if (!discoveryUrl) {
    return { success: false, error: 'No discovery_doc URL in state.artifacts. Cannot seed tickets.' }
  }

  // Invoke Haiku to extract tickets from the Discovery Doc
  const raw = await claudeInvoke(
    buildSeedDiscoveryPrompt(state.sprint_name, discoveryUrl),
    { model: 'haiku', maxBudgetUsd: 0.5, timeoutMs: 120_000 }
  )

  let tickets: Array<Record<string, unknown>>
  try {
    // Strip any accidental code fences before parsing
    const cleaned = raw.replace(/^```[\w]*\n?/gm, '').replace(/```$/gm, '').trim()
    tickets = JSON.parse(cleaned)
    if (!Array.isArray(tickets)) throw new Error('Not an array')
  } catch (err) {
    return {
      success: false,
      error: `Failed to parse ticket JSON from Haiku output: ${String(err)}\nOutput: ${raw.slice(0, 500)}`,
    }
  }

  if (tickets.length === 0) {
    return { success: false, error: 'Haiku returned empty ticket array. Check Discovery Doc has Sprint Plan section.' }
  }

  const trackerRes = await createTrackerItems(tickets, state.sprint_id)
  if (!trackerRes.success) {
    return { success: false, error: trackerRes.error ?? 'createTrackerItems failed during seeding' }
  }

  // Extract cwd from Discovery Doc data attribute if present
  // Expected format: <html data-cwd="/home/jdm/mdj-agent">
  const cwdMatch = raw.match(/data-cwd="([^"]+)"/) ?? discoveryUrl.match(/data-cwd=([^\s>]+)/)
  const cwd = cwdMatch?.[1] ?? null

  return {
    success: true,
    result: { seeded: tickets.length, cwd, trackerIds: trackerRes.data },
    output: `Seeded ${tickets.length} tickets from Discovery Doc.${cwd ? ` cwd: ${cwd}` : ''}`,
  }
}

async function autoResumeInProgress() {
  const resumable = await forgeRunner.findResumableRuns()
  const active = resumable.filter(r =>
    !GATE_PHASES_SET.has(r.phase as ForgePhase) &&

  }
}

/**
 * POST /forge/resume/:id — Resume an in-progress run.
 * Semantically correct URL for what /run/:id was doing.
 */
forgeRoutes.post('/resume/:id', async (req, res) => {
  try {
    const runId = req.params.id
    const r = getRunner()
    const state = await r.resume(runId)
    res.json({
      success: true,
      data: { run_id: runId, phase: state.phase, status: 'resuming' },
    })
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Unknown error'
    const status = msg.includes('not found') ? 404 :
                   msg.includes('terminal') ? 409 :
                   msg.includes('gate') ? 409 : 500
    res.status(status).json({ success: false, error: msg })
  }
})

/**
 * POST /forge/seed/:id — Manually trigger seeding phase on an existing run.
 * Useful if seeding failed and needs retry.
 */
forgeRoutes.post('/seed/:id', async (req, res) => {
  try {
    const runId = req.params.id
    const r = getRunner()
    // Resume will re-enter phaseSeq from seeding_discovery if that's current phase
    const state = await r.resume(runId)
    res.json({
      success: true,
      data: { run_id: runId, phase: state.phase, status: 'seeding' },
    })
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Unknown error'
    res.status(500).json({ success: false, error: msg })
  }
})

The Improved Builder Prompt

What Builders Need (Minimum Viable Spec)

1. The File(s)

Exact absolute path. No guessing. /home/jdm/mdj-agent/src/forge/types.ts — not "the types file."

"tsc passes" is not enough. "tsc passes AND ForgePhase.seeding_discovery is importable in runner.ts" is a testable criterion.

How Discovery Docs Power This

{
  "title": "Add seeding_discovery to ForgePhase enum",
  "type": "feature",
  "description": "Add `seeding_discovery = 'seeding_discovery'` after `creating_sprint` in ForgePhase enum. tsc must pass. Importable from types.ts.",

The description field becomes the acceptance criteria. The file_paths array tells the builder exactly what to touch. The builder prompt template injects both directly.

The Contract

Discovery Doc authors (JDM + Shinobi) must write the Sprint Plan table with enough detail that a Sonnet agent can execute without asking questions. The measure: could you hand this ticket to a junior developer who just joined today and have them deliver it correctly? If yes, the spec is sufficient. If no, it needs more detail.

Sprint 005 enforces this by extracting description from the HTML table and passing it verbatim to builders. The quality of the build is directly proportional to the quality of the spec.

Working Directory Per Sprint

Sprint 001 bug: The builder agents were spawned with cwd = /home/jdm/Projects/toMachina hardcoded in phases.ts. Sprint 001 built changes to /home/jdm/mdj-agent. Every builder ran npm run build in the wrong directory.

Discovery Doc HTML files specify their working directory via a data-cwd attribute on the <html> tag:
