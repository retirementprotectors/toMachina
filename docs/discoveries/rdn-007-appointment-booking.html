Overview

What This Sprint Delivers

A staff-initiated appointment booking flow embedded in ProDashX. Any RPI team member opens a client contact page or a past call row, clicks Book Appointment, picks a staff member, selects an available slot, and confirms. The event lands on both the staff member's Google Calendar and the client's communications timeline.

Architecture

System Diagram (prose)

Staff action: Staff member clicks "Book Appointment" in ProDashX. Modal opens. React component calls GET /api/booking/internal/staff (returns department-grouped staff list), then GET /api/booking/internal/availability?email=&date= (returns free/busy slots), then POST /api/booking/internal (creates booking).

API layer: New route file services/api/src/routes/booking-internal.ts. Mounts at /api/booking/internal. Reads users collection to build staff list. Calls Google Calendar API using per-user OAuth tokens fetched from users/{email}/calendar_oauth/primary subcollection. Writes booking record to booking_log collection.

Calendar sync: On first connect, the API registers a push notification channel via calendarClient.events.watch(). Google sends webhook POST to /api/calendar/webhook when events change. Handler verifies HMAC token, reads updated event, updates booking_log.status if event was deleted or moved. Channel expiry renews on each token refresh cycle.

Token store: Each staff member's OAuth tokens live in the users/{email}/calendar_oauth/primary subcollection document. Values are AES-256-GCM encrypted before write. The route layer decrypts on read using CALENDAR_ENCRYPTION_KEY env var. google-auth-library auto-refresh rotates the refresh token via oauth2.on('tokens', handler).

Calendar Auth Decision: Per-User OAuth vs Domain-Wide Delegation (DWD)

Decision: Per-User OAuth

Per-user OAuth is the correct choice for this sprint. DWD was evaluated and rejected for the reasons below.

Key Invariants

Booking-Before-Calendar Invariant

Always write to booking_log before calling Google Calendar API. If the Calendar API fails, the booking record exists and the staff member can manually create the event. Calendar failure must never roll back the booking write. Calendar success is idempotent — calendar_event_id is updated after successful Calendar call.

Encrypt-Before-Write Invariant

OAuth tokens must be encrypted with AES-256-GCM using CALENDAR_ENCRYPTION_KEY before any Firestore write. The raw token string must never appear in Firestore, logs, or API responses. Decryption happens only inside booking-internal.ts route handlers, never in shared packages.

Token Storage Schema

Firestore path: users/{email}/calendar_oauth/primary

// calendar_oauth/primary document fields
{
  access_token_enc: string     // AES-256-GCM encrypted access token
  refresh_token_enc: string    // AES-256-GCM encrypted refresh token
  expiry_date: number          // Unix ms — not encrypted, used for expiry check
  scope: string                // 'https://www.googleapis.com/auth/calendar.events'
  push_channel_id: string      // UUID for push notification channel

}

Encryption Spec

Algorithm: AES-256-GCM

Key source: CALENDAR_ENCRYPTION_KEY env var on Cloud Run API (32-byte hex string, stored in Secret Manager)

IV: Random 16 bytes per encryption, prepended to ciphertext as hex

Auth tag: Appended after ciphertext (GCM guarantees integrity)

Implementation: services/api/src/lib/token-crypto.ts (new file) — encryptToken(plaintext: string): string and decryptToken(ciphertext: string): string

Never: Log the key, log decrypted token values, return tokens in API responses, store unencrypted tokens.

Push Notification (Two-Way Sync) Spec

Register channel: After storing tokens, call calendar.events.watch() with a UUID channel ID, HMAC token, and webhook address https://prodash.tomachina.com/api/calendar/webhook.

Webhook handler: On POST to /api/calendar/webhook: (1) verify X-Goog-Channel-Token HMAC, (2) read X-Goog-Resource-State, (3) if not_exists — find matching booking_log by calendar_event_id and set status: 'cancelled_by_calendar', (4) if exists — optionally sync updated time back to booking_log.

Channel expiry: Google push channels expire in 7 days maximum. The renewExpiredChannels() function runs daily via cron. It reads all users with calendar_connected: true where push_channel_expiry is within 24 hours, stops the old channel, registers a new one, updates the token doc.
