## Architecture Decisions Made in This Session

Every decision below shapes how The Machine is built. They are not suggestions — they are
the original intent.

1. **MDJ Agent is STANDALONE** — ~/mdj-agent/ on MDJ_SERVER. Never part of the toMachina
   monorepo. Never deployed to Cloud Run. Lives on the server, runs as a systemd service.

2. **Portal → Cloud Run → Tailscale Funnel → MDJ_SERVER** — the proxy architecture.
   Portals call the toMachina API. The API proxies to MDJ_SERVER via Tailscale Funnel.
   MDJ_SERVER never exposes a public IP.

3. **6 Specialists, not a monolith** — General, Medicare, Securities, Service, DAVID, Ops.
   Each specialist has its own system prompt + tool subset. Routing is by intent, not keywords.

4. **Anti-hallucination is the Golden Rule** — "If you didn't READ IT, don't REPORT IT."
   MDJ (VOLTRON) reads Firestore before answering questions about clients. No inference.
   No assumptions. Read the data, report the data.

5. **Standalone SA key auth** — /home/jdm/mdj-agent/sa-key.json. Permanent, never expires,
   scoped to GCP project claude-mcp-484718. No OAuth dance per request.

6. **Model routing is mandatory** — Opus thinks, Sonnet executes, Haiku lookups. This was
   established in this session and saves ~$4,000/month. Applies to all sub-agent spawning.

7. **VOLTRON is the steering wheel** — "The portal is the dashboard. VOLTRON is the steering
   wheel." Every module in every portal is accessible via conversation. The UX is the chat.

8. **MDJ = Josh times infinity** — MDJ doesn't replace the team, it arms them. Every new
   hire on Day 1 has Josh-level capability. Growth is no longer limited by JDM's bandwidth.

9. **The OS makes every agent ACCOUNTABLE** — "The OS doesn't make every agent great. It
   makes every agent ACCOUNTABLE. Big difference." Hooks, gates, audit loops — the guardrails
   force quality even when the session variance is mediocre.

10. **The business doesn't need investors** — "It costs your API bill and a Dell PowerEdge
    on the floor." The infrastructure was already built. MDJ was the last mile.

11. **Parallel execution is default** — never sequential when parallel is possible. This was
    the session where 3 builders ran overnight. That's the operating model forever.

12. **Don't tell the team until fully deployed** — not when CI passes. When it's live.
    Users break things differently than tests do.

## Routing Rules

VOLTRON receives every query first. Before any Lion is dispatched, VOLTRON reads the incoming message and assigns a domain tag. The domain tag determines which specialist handles the query.

**How routing works:**

The MDJ Panel sends a `message` string, a `portal` context, and an optional `context` payload to `POST /api/mdj/chat`. The API proxies to VOLTRON's agent service at `VOLTRON_URL/agent/chat`. The agent service does intent classification on the message body, then routes to the matching specialist.

Routing is keyword-weighted, not rule-based. A message can contain multiple signals — the highest-confidence domain wins. When no domain exceeds threshold, routing falls to **Medicare (default)**.

**The Five Lion Domains:**

| Domain | Specialist | Keyword Examples |
|--------|-----------|-----------------|
| `medicare` | Medicare Lion | "MAPD", "Medicare Advantage", "Part D", "AEP", "enrollment", "formulary", "plan change", "Medicare Supplement", "Medigap", "CMS", "SNP", "DSNP", "carrier", "drug list", "HMO", "PPO" (Medicare context) |
| `investment` | Investment Lion | "RIA", "BD", "Schwab", "Gradient", "investment", "advisory", "portfolio", "annuity" (investment context), "FIA", "MYGA", "1035", "transfer", "broker dealer", "investment" |
| `service` | Service Lion | "client account", "access", "portal", "RMD", "required minimum distribution", "Beni Center", "service request", "case status", "account update", "policy", "beneficiary" |
| `life_estate` | Life/Estate Lion | "B2B", "Life/Estate", "M&A", "partnership", "acquisition", "SENTINEL", "firm", "agency", "deal", "pipeline", "prospect firm" |
| `legacy_ltc` | Legacy/LTC Lion | "import", "data", "cleanup", "admin", "system", "ATLAS", "RAPID", "migration", "sync", "registry", "user management", "bulk" |

**Fallback:**

Any query with no clear domain match — or any query from a user whose portal level does not satisfy a specialist's `required_level` — routes to **Medicare (default)**.

Medicare (default) handles: greetings, general questions about RPI, team info, navigation help, anything cross-domain, and any query VOLTRON cannot confidently classify.

**Specialist configuration source:**

Specialist routing keywords are stored in Firestore collection `mdj_specialist_configs`, field `routing_keywords`. VOLTRON reads this at session start via `GET /api/mdj/specialists`. The config is not hardcoded. Changes to specialist keywords take effect on next session initialization without a code deploy.

**User level enforcement:**

The user's level is loaded from Firestore `users` collection by email (see `mdj.ts` lines 79–88). If a specialist requires `required_level: 4` and the user's level is `3`, routing falls to Medicare (default). The user is not told the routing was downgraded — they receive a general-quality response.

## Wire Execution Protocol

Wires are multi-step operations defined in the QUE registry. They are not ad-hoc tool calls. Every wire has a registered `wire_id`, a minimum entitlement level (`entitlement_min`), and an ordered set of Super Tool stages.

**The Five Steps — in order, no skipping:**

1. **Read registry entry.**
   Call `getVoltronWireById(wire_id)` from `@tomachina/core`. If the wire is not in the registry, execution stops immediately. Error: `{ success: false, error: "Wire not found: [wire_id]", code: "WIRE_NOT_FOUND" }`. VOLTRON does not improvise. If the wire is not registered in QUE, it does not exist.

2. **Validate client context.**
   Confirm `client_id` is present in the request body. Confirm the authenticated user has entitlement rank >= the wire's `entitlement_min`. Rank is resolved from `VOLTRON_ROLE_RANK[userRole]` (see `voltron-wire.ts` lines 64–76). If entitlement is insufficient: `{ success: false, error: "Insufficient entitlement: [role]+ required", code: "ENTITLEMENT_INSUFFICIENT" }`.

3. **Confirm with MDJ Panel.**
   If the wire's approval tier is RED or YELLOW (see Approval Tiers in soul.md), execution pauses. VOLTRON surfaces a confirmation prompt in the MDJ Panel before proceeding. The wire status is set to `approval_pending` in Firestore `wire_executions`. Execution resumes only when `POST /api/voltron/wire/:id/approve` is called by an authorized user. No approval = no execution.

4. **Execute wire.**
   Call `executeVoltronWire(wireInput, context, { simulate, writeAudit })` from `@tomachina/core/voltron/wire-executor`. The executor runs each Super Tool stage in sequence. Real-time stage progress is streamed via `GET /api/voltron/wire/:id/stream` (SSE). If `simulate: true` is passed, the wire runs dry — no external writes, no mutations, status set to `simulated`.

5. **Log result.**
   The `writeAudit` callback writes the full execution record to Firestore `wire_executions` collection, keyed by `execution_id`. The record contains: `wire_id`, `client_id`, `user_email`, `user_role`, `entitlement`, `params`, `stage_results`, `status`, and timestamps. This record is permanent. It cannot be deleted by VOLTRON. Audit logs are the accountability layer — they surface in Command Center open cases when status is `failed`.

## AEP Blackout Enforcement

**Blackout window: October 1 – December 7, annually.**

During this window, all wires tagged with domain `medicare` are suspended. This applies without exception to every wire whose `entitlement_min` or domain classification involves Medicare plan changes, enrollment, or formulary modifications.

**Behavior during blackout:**

- VOLTRON detects AEP blackout by comparing the current date against the window `[Oct 1, Dec 7]`.
- Any wire execution request for a `medicare` domain wire during this window is rejected before Step 1 completes.
- The Medicare Lion still handles read-only queries (lookups, plan comparisons, formulary checks) — blackout applies only to wires that execute plan changes or enrollment actions.
- There is no manual override at the VOLTRON layer. Override requires explicit JDM authorization at the infrastructure level.

**Exact error message returned to the MDJ Panel:**

## Error Protocol

**All VOLTRON errors return this exact shape:**

```json
{
  "success": false,
  "error": "Human-readable explanation of what went wrong and what the user can do.",
  "code": "ERROR_CODE"
}
```

**Rules:**

- `error` is written for a human. It is shown in the MDJ Panel. Use plain English. Never include stack traces, internal variable names, Firestore paths, or raw exception messages.
- `code` is machine-readable. It is used for logging, Command Center categorization, and escalation routing.
- HTTP status codes still apply at the route layer (400, 403, 404, 500) — the JSON body always follows this shape regardless.
- If an error occurs after SSE headers have been sent, VOLTRON writes the error as a final SSE event before closing the stream. It does not silently drop the connection.

**Documented error codes:**

## Escalation Path

When a wire fails, VOLTRON does not silently discard the failure. The following chain executes automatically:

**Step 1 — Log.**
The `writeAudit` callback writes the failure to Firestore `wire_executions` with `status: "failed"`. The record includes: `wire_id`, `client_id`, `user_email`, `stage_results` (including the failed stage and its error), and a timestamp. This is non-negotiable — the audit write happens before any user-facing message is returned.

**Step 2 — Notify via MDJ Panel.**
VOLTRON surfaces the failure to the requesting user in the MDJ Panel using the standard error shape: `{ success: false, error: "[human-readable]", code: "[ERROR_CODE]" }`. The message tells the user what failed and whether they should retry or contact support.

**Step 3 — Surface in Command Center open cases panel.**
Any `wire_executions` document with `status: "failed"` is surfaced as an open case in Command Center. This is a read from the `wire_executions` collection — no additional write is required. Legacy/LTC and admins monitoring Command Center see failed wires in real time. The case includes: wire name, client, user, time, and the specific stage that failed.

**What triggers escalation:**
- `executeVoltronWire` throws or returns a failed stage result
- SSE stream times out at 5 minutes with an incomplete execution
- Approval gate abandoned (wire stays in `approval_pending` indefinitely — surfaced as a stale open case after 24 hours)

**What does NOT trigger escalation:**
- Entitlement refusals (`ENTITLEMENT_INSUFFICIENT`) — these are expected rejections, not failures
- AEP blackout rejections (`AEP_BLACKOUT_ACTIVE`) — these are enforced blocks, not failures
- Simulation runs (`simulate: true`) — these write `status: "simulated"`, not `status: "failed"`

## Session Memory

**Within a session:**

VOLTRON maintains full conversation context across every message in the same conversation thread. The `conversation_history` array is passed on every `POST /api/mdj/chat` request (see `mdj.ts` lines 109–113). VOLTRON uses this history to answer follow-up questions, maintain context across specialist switches, and avoid asking the user to repeat themselves.

Conversation records are stored in Firestore `mdj_conversations` collection, with messages as a subcollection. The user can retrieve prior conversations via `GET /api/mdj/conversations` and `GET /api/mdj/conversations/:id`.

**Between sessions:**

VOLTRON does not carry assumptions from a prior session. Every new session begins with a clean state. On initialization VOLTRON:

1. Re-reads the QUE registry (wire definitions from `@tomachina/core`)
2. Re-loads specialist configs from Firestore `mdj_specialist_configs`
3. Re-loads user level and display name from Firestore `users`
4. Does NOT assume any client data, account state, or wire status from a previous session

**The rule:** If it was not READ in this session, it is not known. This is not a limitation — it is the anti-hallucination guarantee. Stale state causes wrong answers. A cold read at session start costs milliseconds. A wrong answer costs trust.

Conversation history for prior sessions is available on request via the conversations API — VOLTRON can pull prior threads and summarize them, but it does so through a read, not from memory.

## CTO Review Lessons — 2026-04-05 Night Shift

SHINOB1 (CTO) reviewed 9 Track Discovery Docs (3 per CXO) written by 3 parallel MUSASHI sessions. These are the lessons from that mentorship. **Load these BEFORE writing any Discovery Doc.**

### Lesson 1: Renames require a grep manifest
Any ticket that renames, rekeys, or re-routes anything needs a "STEP 0: audit all references" acceptance criterion. The blast radius of a rename is never obvious from the files you already know about. It's the files you DON'T know about that break. Example: ZRD-C04 (atlas → megazord) had 12 reference locations, not the 9 originally listed. The builder must grep, document every reference with file + line, and include the full manifest in the PR.

### Lesson 2: Architecture decisions cannot live in implication
If a ticket could be built three different ways, specify which way and WHY. The "why not" is as important as the "why" — it prevents a smart builder from re-deriving the wrong answer. RONIN doesn't know your mental model. RONIN knows what you write down. Example: ZRD-O08 "dispatch Rangers" could mean CCSDK agent, Cloud Function, microservice, or Express route. Only one is correct for v1. Spec it.

### Lesson 3: Prefixes are routing metadata, not branding
Ticket IDs aren't brand marks. They're machine-readable routing keys. The orchestrator greps them. RONIN greps them. The Dojo greps them. `*-C*` must work across ALL CXOs. Format: `{PREFIX}-{TRACK}{NUMBER}` — ZRD-C01, VOL-O03, MUS-D14. Never invent a new format per CXO. Example: VLT-T1-001 was wrong because it broke the cross-CXO grep pattern.

### Lesson 4: Acceptance Criteria are gates, not notes
If a RONIN builder can't run `npm run build` and get green after completing a ticket, the ticket isn't done — even if the AC says it is. The gap between "I know this needs to work" and "the AC tests that it works" is where builds die. Example: Empty barrel index.ts files must satisfy the type system and pass build, not just exist.

### Lesson 5: When a CTO challenges, bring receipts
"Show me WHERE" means line numbers, not "trust me." If the CTO asks whether wire definitions are specced, point to the exact tickets and sections. The correct response is evidence, not assurance. Example: MUS-O02/O03/O04 had the wire step chains — showed the line numbers.

### Lesson 6: Domain extension > pattern copying
Following an existing pattern is baseline. EXTENDING a pattern for your domain is CXO thinking. QUE's registry doesn't need channel awareness. CMO's does — because creative assets are inherently multi-channel. Adding `channel?: CmoChannel` and `toolDomain?: CmoToolDomain` was the right move because it was informed by domain expertise, not just pattern matching.

### Lesson 7: The CTO review IS the immune system
Praise where earned + correction where needed = the next doc is better than the last one. The CTO doesn't review to gatekeep — he reviews to ensure what ships is what scales. Both the praise and the challenge make the pattern stick. Accept both with zero ego.
