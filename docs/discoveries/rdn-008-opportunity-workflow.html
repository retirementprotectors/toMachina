Why This Exists

Vince's team books appointments. After every booking, a coordinator manually opens Pipeline Studio, finds the right board, creates an Opportunity card, fills in the meeting reason, and sets the stage. For follow-ups, they also manually advance the existing card and pull notes from memory or Slack history.

This is dominoes waiting to fall. The first booking tips the domino — every card after it should cascade automatically: create, link, advance, pull forward, generate the next-steps list. The rep closes the booking and walks away. The pipeline is already updated.

Definition of Done. A service rep books a follow-up for a client who already has an active Opportunity card. The booking: links to that opportunity via dropdown, advances its stage per the pipeline's workflow rules, surfaces the prior meeting's summary at the top of the card, and auto-generates the new stage's required to-do list — all without the rep manually touching the pipeline board.

Decisions Needed from JDM

DECISION 1 — Auto-advance gating behavior when prior stage has incomplete required tasks

When a follow-up booking arrives but the current stage has incomplete required tasks (gate_enforced = true), should the system:

  • (a) Block — refuse to advance; return an error to the booking UI; staff must close tasks first
  • (b) Warn and advance — advance the stage, but flag a warning on the new opportunity card ("Prior stage tasks incomplete")
  • (c) Silent advance — advance without warning; tasks remain open on the prior stage record

Recommendation: (b) — matches the existing flow engine's non-blocking advance pattern (see FlowInstanceAdvanceResult.warning in packages/core/src/api-types/flow.ts line 86).

DECISION 2 — Default pipeline-to-meeting-type mapping

What pipeline board does a "Discovery Call" auto-resolve to? Proposed defaults below — confirm or override:

  • Discovery Call → pipeline: SALES_RETIREMENT or SALES_MEDICARE (needs sub-type or team context)
  • Follow-Up → link to existing Opportunity (no new card)
  • Consultation → same board resolution as Discovery Call
  • Quick Question → no auto-create (informational, no pipeline intent)

The team slug from the booking (e.g. retirement-sales) is available on booking_log.agent_slug — this can drive the mapping resolution. Confirm whether team slug is the correct resolver or if we need an explicit dropdown on the booking form.

DECISION 3 — Prior meeting summary edit permissions

The pull-forward summary shows the previous meeting's notes at the top of the Opportunity card. Should it be:

  • Read-only — historical record; staff writes new notes in the current stage's task fields
  • Editable — staff can amend prior summary before saving the new stage

Recommendation: Read-only — the prior summary is a receipt, not a draft. Editable opens integrity questions.

DECISION 4 — Coordinator role pipeline sharing

Does the Coordinator role (Nikki's team) share the same pipeline boards as Sales (Vince's team), or do reactive service calls create Opportunities on a separate Coordinator board? Currently REACTIVE_MEDICARE and REACTIVE_RETIREMENT pipeline keys exist in custom-fields.ts. Confirm whether reactive/service bookings should auto-create on those boards or on the NBX sales boards.

Key Architectural Decision — Extend Flow Engine, Don't Fork

The flow engine already knows how to advance stages and generate tasks (PATCH /api/flow/instances/:id action=advance). This sprint EXTENDS that existing path by wiring booking events as its trigger. We do not build a parallel advancement system. Booking becomes a first-class trigger event for flow_instance advancement — the same gates, same task template generation, same activity log write.

Core State Machine

Every POST /api/booking runs this decision tree. Steps 1–2 are new. Step 3 is the existing booking write.

TRIGGER
POST /api/booking receives: agentEmail, slot, meetingType, client (with client_id from RDN-007)
↓
STEP 1 — Resolve pipeline
Look up meeting_type_pipeline_map where meeting_type == booking.meetingType.
If no mapping found: skip auto-create, log warning, write booking_log as normal. Staff creates card manually.
↓
STEP 2 — Check for existing active Opportunity
Query opportunities where client_id = booking.client_id AND status != 'closed_lost'.
If multiple found: surface all in dropdown — staff picks one (or selects "New Opportunity").
↓ Branch on result
NO active opportunity found → FIRST BOOKING PATH
  • POST /api/opportunities with resolved pipeline_key, first stage, client_id, meeting_reason, assigned_to, created_from='auto-booking', first_booking_id
  • POST /api/flow/instances — creates flow instance for new opportunity (opportunity_id linked)
  • PATCH booking_log: set opportunity_id = new opp id
  • Write meeting_summaries subcollection entry (empty — first meeting, no prior summary)
YES active opportunity found → FOLLOW-UP PATH
  • Staff selects opportunity from dropdown in booking UI (or auto-selects if only one)
  • Write prior meeting notes to opportunities/{id}/meeting_summaries
  • PATCH /api/flow/instances/:instanceId action=advance — gate check per DECISION 1
  • New stage tasks auto-generated by existing flow engine task template logic
  • PATCH opportunity: booking_history[] += booking_id, last_booking_summary = summary text, updated_at = now
  • PATCH booking_log: opportunity_id = selected opp id
↓
STEP 3 — Existing booking write (unchanged)
booking_log written to Firestore + bridge sync. Response returned to client. Steps 1–2 are pre-write async ops; if they fail they log a warning but do NOT block the booking confirmation.

Meeting Type → Pipeline Resolution

The meeting_type_pipeline_map collection (new) stores documents keyed by a composite of team slug + meeting_type. Resolution logic:

// Resolution order:
// 1. Exact match: agent_slug + meeting_type  (e.g. "retirement-sales:discovery-call")
// 2. Team-level match: team_category + meeting_type  (e.g. "retirement:discovery-call")
// 3. Meeting-type-only match  (e.g. "discovery-call")
// 4. No match → skip auto-create, log WARN

// Example documents in meeting_type_pipeline_map:
{
  id: "retirement-sales:discovery-call",
  team_slug: "retirement-sales",
  meeting_type: "Discovery Call",
  pipeline_key: "SALES_RETIREMENT",
  first_stage: "prospect"   // stage_id of first stage on that pipeline
}
{
  id: "discovery-call:default",
  meeting_type: "Discovery Call",
  pipeline_key: "SALES_RETIREMENT",
  first_stage: "prospect"
}

What This Sprint Adds to Flow Engine

The extension rule: Booking is a new trigger for existing flow instance operations. This sprint wires that trigger. It does not modify the flow engine itself (services/api/src/routes/flow.ts) — it calls the existing advance endpoint from the new opportunity-automation.ts library.

  2. Where task templates are queried for generation — confirm the collection and index

flow_instance ↔ opportunity linkage

The 1:1 relationship between flow_instances and opportunities is via flow_instances.opportunity_id (existing field) and the corresponding query. For auto-created opportunities, this sprint creates both records in sequence:

  1. POST /api/opportunities → returns opportunity_id
  2. POST /api/flow/instances with opportunity_id in body → links the instance

Task template generation on follow-up advance

When a follow-up booking triggers a stage advance, the flow engine's existing task template generation runs automatically. No new logic needed — the templates in flow_task_templates for the new stage are already seeded by the Pipeline Studio admin flow. This sprint assumes those templates exist for each pipeline stage. If they do not, the advance returns a warning (existing behavior, RDN-DOJO-GAP-05 pattern).

Activity log entry on booking-triggered advance

The existing advance handler writes a ADVANCE_STAGE activity log entry. For booking-triggered advances, append metadata to the notes field: "Triggered by booking: {booking_id}". This requires passing notes in the advance PATCH body — verify the existing handler accepts this field.
