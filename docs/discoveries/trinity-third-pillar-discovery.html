The Holy Trinity

The Trinity Data Method™ reconciles three data sources to achieve 100% Data Confidence. On the platform, that trinity maps to three equal pillars:

Contacts

CRM Data

Actionable Intelligence

Accounts

Carrier Data

Source of Truth

ACF

Digital Files

Supporting Documentation

The gap today: Contacts and Accounts are first-class entities with their own grids, detail views, search, filters, dedup, and cross-links. ACF is still a tab on the Contact detail page — a subordinate. The Trinity isn't complete until ACF stands on equal footing. Three grids. Three detail views. Fully integrated. Fully deduped. Fully cross-linked.

What the Trinity Data Method™ calls Supporting Documentation — the PDFs, statements, scans, and files that prove the data is real — lives in Google Drive as Active Client Files. Each ACF is a structured folder: 5 subfolders that form a lifecycle from onboarding through reactive service. This document defines how ACF graduates from "a tab" to "a pillar."

Data Source

The grid queries the existing GET /api/acf/audit endpoint (bulk) combined with Firestore clients collection to join client names. Each row represents one ACF folder in Google Drive, enriched with:

  • Completeness score — (subfolder_count / 5) * weight + (document_count / expected_docs) * weight
  • Document count — total files across all subfolders (from ACFDetail.subfolders[].file_count)
  • Dedup badge count — potential duplicate ACFs (same client name, different folder IDs)
  • Last modified — most recent modifiedTime across all files in the ACF

The 5 Subfolder Tabs

These are NOT categories — they are a lifecycle. Documents flow through them as a client relationship evolves.

TabContainsLifecycle Stage
Client Medicare card, driver's license, SSN card, AI3, HIPAA auth, tax returns, POA, birth cert, passport Foundation — static identity documents that rarely change
NewBiz Applications (signed), 1035 exchange forms, transfer/TOA/ACAT, delivery receipts, replacement forms, enrollment docs Onboarding — documents from when a policy or account is first established
Cases Comparisons, illustrations, analysis docs, quotes, proposals, discovery meeting notes Active work — the analytical work behind a recommendation
Account Statements, confirmations, annual reviews, account summaries, performance reports, distribution records Ongoing — recurring documents tied to active accounts
Reactive Everything that doesn't match the above patterns — correspondence, misc uploads, unclassified documents Catch-all — auto-routed here by default, manually reclassified as needed

File Classification

The existing classifyFileName() function in ACFSection.tsx handles auto-routing. Same rules apply here. Files dropped into the detail view get classified and routed to the right tab automatically. The ACFRoutingRule[] in ACFConfig allows admin-configurable overrides.

Cross-Links

The header always shows:

  • Contact — /contacts/{clientId} (jump to the person)
  • Accounts (N) — filtered view of /accounts?client={clientId} (jump to their accounts)
  • Open in Drive — folder_url from the ACF record (direct Drive link)

The Filter: File Name Is the Index

Zero tokens. Zero AI. Pure path lookup. 
acfFiles.filter(f => f.name.includes(policyNumber))

The naming convention IS the database. If a file contains the policy number in its name, it belongs to that account. No search index. No metadata query. String.includes().

This works because ACF naming conventions embed the policy/account number: {LastName}_{FirstName}_{Carrier}_{PolicyNumber}_{DocType}.pdf. The file name IS the index.

4. One-Click Document Buttons

Summary cards on Contact and Account detail pages get document quick-access buttons. These follow the existing AccountDocuments checklist pattern — buttons are blue when found, gray when missing. One click opens the file. Zero searching.

Contact Card Buttons

📄 Medicare Card 📄 Driver's License 📄 AI3 📄 HIPAA Auth 📄 POA

Source: Client/ subfolder. Pattern match on file name keywords.

Account Card Buttons

📄 Latest Statement 📄 Policy Doc 📄 Application 📄 Annual Review

Source: ACF subfolders filtered by policy number. Most recent match per doc type wins.

Resolution Logic

// Contact card: Medicare Card
const medicare = clientFiles.find(f =>
  /medicare.*(card|id)|cms.card/i.test(f.name)
)
// button = medicare ? BLUE (onClick opens preview) : GRAY

// Account card: Latest Statement
const statements = acfFiles
  .filter(f => f.name.includes(policyNumber))
  .filter(f => /statement|confirm/i.test(f.name))
  .sort((a, b) => new Date(b.modifiedTime) - new Date(a.modifiedTime))
// button = statements[0] ? BLUE : GRAY
No AI involved. This is string matching on file names that already follow a naming convention. The classifyFileName() rules that exist in ACFSection.tsx handle the pattern matching. The buttons just reflect what's already there. If a file exists and matches — blue. If not — gray. Zero tokens consumed.

5. Dedup at Every Level

ACF dedup follows the exact same pattern already live on Contacts (via PossibleDuplicates.tsx). Three surfaces, three actions.

Level 1: Grid Badge

On the ACF grid, rows with potential duplicates show an amber badge: 3 possible dupes. Detection: multiple ACF folders with similar client names (via the existing calculateNameScore() from dedup.ts).

Level 2: Detail Banner

On the ACF detail view, a collapsible banner (same UX as Contact PossibleDuplicates) with three action buttons:

  • Merge — opens the dedup merge workflow (/ddup?ids={id1},{id2}&type=acf)
  • Dismiss — marks this pair as not-duplicate (persisted, won't resurface)
  • Check with JDM — creates a /q item for CEO review (classification: ACF_DEDUP)

Level 3: Dedicated Dedup Page

At /acf/dedup — a batch processing view showing all ACF duplicate pairs sorted by confidence score. Same layout as the existing contact dedup page. Bulk select, bulk merge, bulk dismiss.

7. AI3 Assembly — Meeting Prep

"Generate Meeting Prep" is the payoff of the entire ACF structure. Because files are named consistently and organized into lifecycle subfolders, assembly is pure path lookup — no AI classification needed at generation time.

Assembly Recipe

1. Client/ subfolder:
   → Medicare card (most recent)
   → AI3 form (most recent)
   → HIPAA authorization

2. Account/ subfolder (per active account):
   → Most recent statement (sorted by modifiedTime)
   → Annual review (if exists)

3. Cases/ subfolder:
   → Active proposals (modified in last 90 days)
   → Most recent illustration per carrier

4. NewBiz/ subfolder:
   → Pending applications (modified in last 30 days)

Output: a single assembled PDF or a Drive folder link with the meeting-ready documents. One button on the Contact detail or ACF detail page. The button queries ACF subfolders by naming convention, pulls the most recent match per category, and either generates a merged PDF (via PDF_SERVICE) or creates a temporary Drive folder with shortcuts.
