How it works:
1. GitHub fires workflow_run webhooks to our API on every status change
2. API writes to Firestore ci_runs collection (one doc per run)
3. STATUS tab reads from Firestore via GET /api/ci — auto-refreshes every 15 seconds
4. The dispatcher on MDJ_SERVER reads the same Firestore data — no GitHub dependency
5. Click any card to open the GitHub Actions run page

Summary strip at the top gives instant health: "All Green", "CI Running", "Deploy Failed" etc.
Active run (amber border) auto-expands to show per-job breakdown with live timing.
Deploy pipeline shows "Waiting for CI" when CI is still running — updates to "Deploying" once CI passes.

Data stored in Firestore means:
- Portals show CI status (this tab)
- Dispatcher reads deploy state without polling GitHub
- RAIDEN can auto-create fix tickets when CI fails
- Historical data for velocity tracking
