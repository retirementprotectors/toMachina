The Problem

7 concurrent RONIN agents crashed MDJ_SERVER at 32GB RAM.

Each RONIN runs a Claude Code session via the Agent SDK + spawns sub-agents for build/test/audit phases. With tsc and webpack builds running simultaneously across 7 agents, memory exceeded physical capacity. The Machine outgrew its body.

The Architecture — Cloud Run Burst

When MDJ_SERVER hits capacity, overflow RONIN agents launch as Cloud Run Jobs instead of local tmux sessions. Same FORGE lifecycle. Same code. Same results. Different metal.

┌──────────────────────────────────────────────────────────────┐ │ 2HINOBI (COO) — The Dispatcher │ │ Decides: local or cloud based on MDJ_SERVER headroom │ ├──────────────────────┬───────────────────────────────────────┤ │ │ │ │ MDJ_SERVER (local) │ Cloud Run Jobs (overflow) │ │ ┌────────────────┐ │ ┌────────────────────────────────┐ │ │ │ RONIN 1 (tmux)│ │ │ RONIN 5 (Job: 4Gi/2CPU) │ │ │ │ RONIN 2 (tmux)│ │ │ RONIN 6 (Job: 4Gi/2CPU) │ │ │ │ RONIN 3 (tmux)│ │ │ RONIN 7 (Job: 4Gi/2CPU) │ │ │ │ RONIN 4 (tmux)│ │ │ RONIN N (auto-scale) │ │ │ └────────────────┘ │ └────────────────────────────────┘ │ │ │ │ │ Firestore ◄─────────┼──────────► Firestore │ │ Git push ◄──────────┼──────────► Git push │ │ Slack war room ◄────┼──────────► Slack war room │ └──────────────────────┴───────────────────────────────────────┘

Key Principle: Same Interface, Different Metal

A Cloud Run RONIN and a local RONIN are indistinguishable to the rest of The Machine. Both write to Firestore. Both push to git. Both post to the Slack war room. Both follow the same 13-phase FORGE lifecycle. 2HINOBI doesn't care where a RONIN runs — only that it ships.

Phased Rollout

Phase 1 — Hardware (Now)

Install 64GB RAM + P40 GPU. This alone handles 5-6 concurrent RONINs comfortably. Buys time.

No code changes. Just hardware.

Phase 2 — Cloud Run Burst (When Ready)

Build the Dockerfile + forge-job.ts + dispatch logic. Test with a single overflow RONIN. Validate that results are identical to local runs.

SHINOB1 architects. RONIN builds. MUSASHI writes the Discovery Doc if needed.

Phase 3 — Auto-Scale (Future)

2HINOBI dynamically decides concurrency based on sprint priority + server load + cost budget. No human in the loop. The Machine scales itself.

The Machine managing The Machine.

Recommendation

Phase 1 is the move right now. 64GB + P40 handles 90% of use cases. Cloud Run Burst is the insurance policy for when The Machine outgrows 64GB — and it will.

The architecture is clean because everything already talks to Firestore and git. A Cloud Run RONIN doesn't need MDJ_SERVER for anything except dispatch. The plumbing is there. The gap is a Dockerfile and an entry point.

When SHINOB1 is ready to spec Phase 2, MUSASHI will write the Discovery Doc. Until then — install the RAM, install the P40, and let the RONINs run.
