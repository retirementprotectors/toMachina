The Sequence

AActivity
→
PProcess / Tools
→
PPeople / Systems
→
AAutomation
A
Activity Do the work. A lot of it.

Nothing starts without activity. Not planning. Not architecture. Not documentation. Doing.

Activity is volume. It's making mistakes. It's breaking things. It's learning the domain by living in it, not reading about it. You run the import manually 50 times. You handle the edge cases yourself. You see what breaks and what doesn't.

The purpose of activity isn't to produce a deliverable — it's to expose the patterns. You can't see the pattern from a whiteboard. You can only see it after enough reps that your hands know the shape of the work.

The Rule

You cannot document what you haven't done. You cannot process-ify what you haven't experienced. Skip this stage and everything downstream is built on theory, not reality.

What this looks like

AI Team

MEGAZORD runs carrier field migrations manually with JDM. Breaks production. Fixes it. Discovers 9 fields, 5 die, 4 stay. Learns that code must deploy before data migrates. Learns that Firestore indexes must exist before queries reference new fields. None of this was in a doc — it came from doing.

IRL Team

The service team handles client calls for months. Discovers that clients can't be found in one system because the data lives in another. The pattern — dual-system friction — only surfaced through volume of real calls.

P
Process / Tools Document it. Encode it.

After enough activity, the patterns become obvious. Now you document them. Not before — after.

The process is what you learned from the activity, written down so someone else (or a system) can follow it. The tools are the process encoded into something executable — a checklist, a hookify rule, a protocol, a UI.

Process and tools are inseparable. A process without tools is a suggestion. Tools without a process are random automation. Together, they're a repeatable, enforceable standard.

The Rule

You cannot build tools for processes you haven't lived. A tool built on theory will enforce the wrong thing. The 7-stage Data Migration Protocol wasn't designed in a planning session — it was extracted from a production incident.

What this looks like

AI Team

Data Migration Protocol — 7 stages, 2 JDM gates, hookify enforcement. Born from the carrier_name incident. Every stage exists because we hit it in production. Carrier field review HTML — interactive tool for JDM to make field decisions. Built because we learned that field parity requires human judgment, not automation.

IRL Team

Client onboarding checklist — the steps the service team follows for every new client. Didn't come from a consultant — came from handling 500 clients and seeing what got missed. The checklist is the process. The CRM template is the tool.

P
People / Systems Train on proven foundations.

Now you have a process and tools that were forged through real activity. Now you can train people or configure systems to follow them.

Training on unproven tools is a waste of everyone's time. The tool changes, the process changes, and the training is obsolete before it sticks. But training on battle-tested process + tools? That compounds. Every person trained is a multiplier.

This is where you prove it works at scale with real humans or real systems executing it — not just the person who built it.

The Rule

You cannot train people on tools that don't exist or aren't battle-tested. You cannot deploy a system on an unproven process. Prove it works manually first. Then hand it off.

What this looks like

AI Team

CIO Ownership Grid — JDM and MEGAZORD walk it weekly. The process is proven through multiple sessions. When a new MEGAZORD session starts, the grid IS the training — every domain, every status, every next step. The warrior doesn't need a tutorial — the tool teaches.

IRL Team

New hire onboarding — after the checklist (Process/Tools) is proven with 3-4 hires, you hand it to a coordinator. They follow the checklist. They don't need to know WHY each step exists — the process handles that. The person is trained on the tool, not the theory.

A
Automation Only after everything is hardened.

Automation is the last thing, not the first. It can only be done through the hardening of People/Systems. Until those are proven, automation will multiply your problems instead of solving them.

Imagine creating an automated pipeline for an unproven process. Or a scheduled job that runs a migration nobody's ever done manually. That's automation without APPA.

The test for automation readiness: could a brand-new hire with zero context follow this process and get it right on day one? If yes — automate it. The simpler the execution can be, the stronger the process is. That's the goal: a foundation so solid that the executor doesn't need to think — just follow the steps.

And here's the part people miss: automation doesn't replace people — it elevates them. When the repetitive work gets automated, the people who used to do it move into higher-value roles. Better work. Better pay. More impact. The automation handles the floor so your team can reach the ceiling.

The Goal

"Follow these steps. Report the result." That's what good automation sounds like. No judgment calls. No domain expertise needed. Just execution on a proven path — freeing your people to do the work that actually requires their brain.

What this looks like

AI Team

Rangers — CCSDK Haiku-class agents that execute ATLAS wires. They don't think. They read the wire definition, execute each step, report results. The wire IS the process. The Ranger IS the automation. But only after the wire has been run manually enough times to be bulletproof.

IRL Team

Automated client welcome sequence — after the onboarding checklist has been proven by 3 coordinators across 50 clients, the repetitive parts (welcome email, portal setup, document request) get automated. Not before. The automation inherits the proven process, not a guess.

What Happens When You Skip Stages

×

Skip Activity → Process: You document processes you've never done. The docs are wrong. The tools enforce the wrong thing. Everyone follows a broken map.

×

Skip Process → People: You try to train people with no playbook. Everyone does it differently. Tribal knowledge. When someone leaves, the knowledge walks out the door.

×

Skip People → Automation: You automate an unproven process. Now your mistakes run at machine speed, 24/7, with nobody watching. A broken cron at 2am is worse than no cron at all.

×

Skip straight to Automation: "Let's just build a bot for that." The bot doesn't know the edge cases because nobody does. The bot breaks production. The bot creates more work than it saves. Now you're debugging automation AND doing the work manually.

The Automation Readiness Test

Before building ANY automation — pipeline, timer, bot, cron, sequence — ask one question:

"Could someone with zero context follow this process and get it right on day one?"

If yes — automate it. If no — more activity first. The process isn't ready yet.

Real Example — A Database Migration

The Activity (Layer 1)

The data team spent hours reviewing 12,000+ account records across 43 carriers. 9 redundant database fields needed to be consolidated into 4. Leadership reviewed every carrier, made every naming decision, and built an interactive review tool for the cleanup.

The Mistake

The team migrated the database (renamed a field on 11,000+ records) BEFORE updating the 249 code references across 52 application files that still read the old field name. The accounts page broke immediately — showed 35 records instead of 12,000+. Then the database index for the new field didn't exist, so even after the code fix, queries failed. Three production issues from one migration.

The Process/Tools (Layer 2)

From that one incident, three permanent artifacts were created:

1. Data Migration Protocol — 7-stage pipeline with 2 leadership approval gates. Inventory → Impact Report → Code First → Dry Run → Execute → Verify → Cleanup.

2. Automated enforcement gate — blocks any field deletion without protocol acknowledgment.

3. Operations dashboard tab — Data Migration pipeline with active migrations tracker and incident record.

Why This Matters

None of those three things could have been designed from a whiteboard. The protocol is correct BECAUSE the migration failed. The gates exist BECAUSE the mistake happened. The next migration will be better — not because someone read a doc, but because the tools won't let you skip the steps. Every mistake, treated correctly, becomes a permanent improvement in speed and accuracy.
