WS2 — QUE Tools Accuracy Audit

The Lions are only as trustworthy as the numbers they produce. This workstream verifies every authoritative data source used by QUE calc tools. Wrong brackets, stale surrender schedules, or outdated RMD factors erode client trust fast.

Golden Rule enforced here: QUE calc tools must NEVER use hardcoded tax tables, bracket thresholds, or carrier data that could go stale. Every calc that depends on year-specific authoritative data must load from a verified, updateable source.
Data SourceUsed ByVerification TargetUpdate Cadence
Federal tax brackets CALC_TAX_ESTIMATE IRS Revenue Procedure (current tax year) Annual (Jan)
State income tax rates CALC_TAX_ESTIMATE State DOR publications or Tax Foundation summary Annual (Jan)
IRMAA brackets CALC_IRMAA CMS Annual Notice of Medicare Cost Sharing Annual (Oct)
IRS Uniform Lifetime Table CALC_RMD IRS Publication 590-B (current revision) Rare — last updated 2022
Carrier surrender schedules CALC_1035_OPPORTUNITY, CALC_FIA_PROJECTION Gradient carrier agreements + product prospectuses Per product launch / rate change
FYC / commission rates CALC_COMMISSION Current Gradient carrier agreements Annual renewal / carrier change
Medicare plan data (MAPD) LOOKUP_MAPD, Humana/Aetna MCPs CMS Plan Finder API + carrier-direct APIs Annual (Oct 1 plan year)
FIA crediting rates CALC_FIA_PROJECTION Carrier rate announcements (monthly/quarterly) Monthly or quarterly

Accuracy Audit Protocol

For each data source above:

  • Locate where the value is used in packages/core/src/que/calc/
  • Determine: is it hardcoded, loaded from config, or fetched live?
  • Compare current value against authoritative source
  • If stale or hardcoded: refactor to config-driven or live-lookup pattern
  • Document source URL + verification date in code comment
Blocking rule: No QUE calc tool may use a year-specific constant (tax bracket, IRMAA threshold, RMD factor) that is not loaded from a config file or live source. Hardcoded 2023 brackets in 2026 code is a compliance failure.

WS3 — AEP Preparation

Annual Enrollment Period (Oct 1 – Dec 7) is the highest-stakes window in Medicare. Every RPI Medicare client is eligible. VOLTRON treats AEP as a hard calendar event — preparation begins 90 days out, not the week before.

AEP Blackout Rule: WIRE_AEP_ENROLLMENT is blocked outside the Oct 1–Dec 7 window. This is not optional. Agents cannot enroll Medicare clients outside the blackout window via VOLTRON. The wire must enforce this with a clear error message and next-window date.

Pre-AEP Checklist (before Oct 1 each year)

#TaskOwnerDeadline
1Medicare Lion end-to-end test — full MAPD flow with Humana + Aetna MCPsRONIN / QASept 1
2Verify Humana humana_get_medicare_plans returns current plan yearRONINSept 1
3Verify Aetna aetna_search_insurance_plans returns current plan yearRONINSept 1

5Confirm WIRE_AEP_ENROLLMENT blackout logic rejects pre-Oct-1 requestsRONINSept 15
6MUSASHI campaign coordination — marketing materials ready for Oct 1 triggerMUSASHI / CMOSept 15
7E2E test suite run on Medicare wires — all boundary conditions passRONINSept 20
8Proactive Oct 1 trigger configured — Medicare Lion batch for all eligible clientsRONINSept 25

AEP Wire Architecture

  • WIRE_AEP_ENROLLMENT — gated by date window (Oct 1–Dec 7). Blocked outside window with { success: false, error: 'AEP window not open', next_window: '...' }
  • WIRE_AEP_REVIEW — NOT gated. Can run year-round. Triggers MAPD comparison + cost analysis for eligible client.
  • WIRE_AEP_BATCH_TRIGGER — Oct 1 automated batch. Fires Medicare Lion for every client with Medicare coverage or T65 approaching.
  • CALC_IRMAA — must use plan-year brackets. Requires annual update before Oct 1.
Humana + Aetna MCPs are already configured via rpi-healthcare. No new API setup needed. AEP prep is verification and orchestration work, not integration work.

WS4 — E2E Tests for QUE Wire Accuracy

Type-check passes. Build passes. But does the RMD calculation produce the right number for a 75-year-old with a $340,000 IRA? That's what WS4 answers. Automated output validation — not structure checks, actual numerical correctness.

Where it lives: e2e/intake/ suite (Vitest, post-deploy verification). Tests run after deploy-api on push to main. Test data uses e2e-test- prefix. Cleaned up automatically.

WS5 — Lions at Scale — Proactive Operations

This is the DEVOUR endgame. VOLTRON stops answering questions and starts delivering answers before they're asked. Four proactive operation patterns, all running on MDJ_SERVER.

Dependency: Proactive ops require accurate QUE wires (WS2) and full E2E test coverage (WS4). Lions cannot be proactive if they're producing wrong numbers. Accuracy before automation — always.

Pattern 1 — Review Prep: Before Meetings

Trigger: Google Calendar event with client name, 48 hours out, tagged as "Annual Review" or "Review" in title.

Action: VOLTRON fires WIRE_ANNUAL_REVIEW for the client — account summary, RMD status, Medicare plan comparison, 1035 opportunity check, outstanding case tasks.

Output: Review prep brief delivered to agent via VOLTRON chat panel + SMS summary. Ready before the agent opens the file.

  • Google Calendar MCP polling: mcp__google-calendar__list-events — scan next 72h, match client name
  • Cron: every 6 hours on MDJ_SERVER (mdj-agent/src/jobs/review-prep.ts)
  • Dedup: fire once per meeting (track fired meeting IDs in Firestore voltron_jobs collection)

Pattern 2 — RMD Alerts: Before Deadlines

Trigger: Nov 1 — 30 days before Dec 31 RMD deadline. Checks all clients with qualified accounts where RMD has not been distributed for the current year.

Action: Annuity Lion (or Investment Lion) runs CALC_RMD for each flagged client. Alert sent to assigned agent + client SMS (opt-in only).

Output: Slack DM to agent with client list + calculated amounts. SMS to client: "Your required distribution of $X.XX is due by Dec 31. Your advisor [Name] will be in touch."

  • Cron: Nov 1 annual job (mdj-agent/src/jobs/rmd-alert.ts)
  • SMS via mcp__rpi-comms__comms_send_sms
  • Slack via mcp__slack__slack_post_message to agent's DM
  • PHI rule: No account balance or SSN in Slack/SMS. Amount only.

Pattern 3 — AEP Triggers: Season Start

Trigger: Oct 1 at 08:00 CT. Fires for every client with Medicare coverage (any MAPD/Supplement) or T65 within the next 12 months.

Action: Medicare Lion runs WIRE_AEP_REVIEW for each eligible client — current plan analysis, available plan comparison (Humana + Aetna APIs), cost estimate for plan year.

Output: Review package queued in VOLTRON panel per agent. Agents start each AEP day with a pre-built review for every Medicare client on their book.

  • Cron: Oct 1 annual job, runs in batches of 20 clients (mdj-agent/src/jobs/aep-trigger.ts)
  • Rate-limit: Humana + Aetna API calls throttled per MCP guidelines
  • MUSASHI coordinates: campaign materials must be ready in advance (WS3)

Pattern 4 — 1035 Opportunities: Ongoing

Trigger: Monthly check. Annuity Lion scans all annuity accounts for clients exiting surrender charge period within 90 days.

Action: CALC_1035_OPPORTUNITY wire fires — checks current surrender schedule, remaining period, current product vs. available alternatives (better rate, better riders).

Output: Opportunity report delivered to agent: "Client [name] exits surrender period in 67 days. Current FIA: [Product]. Comparable: [Product B] offers X% cap vs Y%. Consider 1035."

  • Cron: 1st of each month (mdj-agent/src/jobs/opportunity-scan.ts)
  • Carrier surrender data must be current (WS2 dependency)
  • Output to VOLTRON panel + optional Slack DM to agent
