Why This Exists

RPI's booking pages at retireprotected.com/book/* embed a GAS web app via iframe. The toMachina booking API (services/api/src/routes/booking.ts) has been built for months but has zero frontend consumers. The GAS version hard-requires email — blocking COF clients (elderly, many without email) from booking.

CTO Decision: No GAS patches. Build the proper Next.js booking page, swap the WordPress embeds, kill the GAS deployment. When JDM rolls this out to the team, it works correctly from day one.

The Doctor Analogy applies here. The booking page is the first thing a COF client touches after getting a text. It behaves like a receptionist: "Let's get you on the calendar. Name and phone is all we need. If you've got email, great — if not, we'll call you." Professional. Zero friction. Respects the person in front of it.

Auth Carve-Out

ProDash auth is 100% client-side in apps/prodash/app/(portal)/layout.tsx. There is no middleware.ts. The (portal) route group convention means only routes nested inside it hit the auth gate.

app/book/ lives outside (portal)/, so it:

  • Inherits only app/layout.tsx (root layout — Providers + fonts)
  • Completely bypasses the auth gate
  • Has AuthProvider in context (from providers.tsx) but doesn't block on it
  • Is fully accessible to unauthenticated visitors (clients)

No changes to auth package, middleware, or next.config needed.

API Proxy Note

The booking page calls /api/booking/* which is proxied to Cloud Run. In dev, next.config.ts rewrites /api/* to localhost:8080. In prod, the app/api/ route handler forwards to Cloud Run.

Problem: The API auth middleware checks Firebase Auth. The booking page is public — no Firebase token. Booking API endpoints need to accept unauthenticated requests.

Solution: RONIN must read services/api/src/middleware/auth.ts and the route mount in src/index.ts to determine the correct auth bypass pattern. Options: mount booking before auth middleware, add an unless pattern, or create a /api/public/booking/* prefix. The search-clients endpoint MUST remain auth-gated (client PII).

Calendar Busy Sync — Cloud Function

What It Does

Reads Google Calendar events for all bookable agents. Writes busy periods to calendar_busy Firestore collection. Runs every 30 minutes via Cloud Scheduler.

Document Schema

// Collection: calendar_busy
// Document ID: "{email}_{year}_{month}"
// Example: "vince@retireprotected.com_2026_4"
{
  periods: [
    { start: "2026-04-08T14:00:00-05:00", end: "2026-04-08T15:00:00-05:00" },
    { start: "2026-04-08T16:30:00-05:00", end: "2026-04-08T17:00:00-05:00" }
  ],
  synced_at: "2026-04-08T12:00:00Z",
  agent_email: "vince@retireprotected.com",
  year: 2026,
  month: 4
}

Sync Logic

  1. Query Firestore users collection: status == 'active' AND has employee_profile.booking_slug
  2. For each agent email: call calendar.events.list with:
    • calendarId: 'primary'
    • timeMin: 1st of current month
    • timeMax: last day of current month + next month (2 months rolling)
    • singleEvents: true (expand recurring)
    • orderBy: 'startTime'
  3. Extract { start, end } from each event (use dateTime field, skip all-day events)
  4. Group by month, write to calendar_busy/{email}_{year}_{month}

Auth Pattern

Copy from services/api/src/lib/calendar-client.ts:

  • Local dev: GOOGLE_CALENDAR_CREDENTIALS env var (JSON service account key)
  • Cloud Run/Functions: keyless domain-wide delegation via IAM Credentials signJwt
  • Scope: https://www.googleapis.com/auth/calendar.readonly (read-only is sufficient)
  • Service account: claude-mcp-484718@appspot.gserviceaccount.com
