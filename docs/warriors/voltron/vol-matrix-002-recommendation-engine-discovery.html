Sprint Gate — What Makes This Worth Building

Tonight (2026-04-15), VOLTRON burned JDM's time on 4 product-doctrine corrections in one case (Westfall). EBR rider elected when it shouldn't be. MS DGI 260% recommended when 50/50 S&P/MARC is locked doctrine. MYGA language used when the category is banned. GLWB joint-life mechanics misapplied. Each correction consumed CEO bandwidth on details that should have been one tool call away. The Preferred Provider Matrix (VOL-MATRIX-001) captured the doctrine as a static HTML reference. This sprint makes it interactive inside VOLTRON — a recommendation engine the team talks to mid-case.

The Directive

JDM, 2026-04-15:

"So this becomes a recommendation engine that we probably could build in maybe even the MDJ engine? With how the wiring is right now in the MDJ engine, if somebody says, 'I've got a client who,' and then they are talking about their money, is that something that we currently do or could route to this decision tree? Like how would that work? Would it start to ask questions about the client to determine the right option? Or could it?"

Answer: Today MDJ routes these queries to the Annuity Lion by keyword — but the Lion answers from model knowledge, not from RPI doctrine. It doesn't consult the matrix. This sprint wires the matrix into the Lion as a structured tool, so every product recommendation is doctrine-locked by definition.

The Problem — One Sentence

VOLTRON's Annuity Lion knows annuity mechanics from training data, but doesn't know RPI's specific product doctrine — which products to recommend, which allocations to lock, which riders to elect, which carriers to prefer.

What We're Building

A structured recommendation tool inside the MDJ agent that:

  1. Detects "I've got a client who..." patterns in conversation
  2. Asks 3-5 clarifying questions about the client's money shape
  3. Walks the Decision Tree from the Preferred Provider Matrix
  4. Returns: lead product + alt product + default allocation + doctrinal flags + brochure links + illustration spec
  5. Every answer is backed by the matrix JSON — not model inference

Why Now — The Upstream Fix

Model-knowledge recommendations

VOLTRON recommends products from Claude's training data. Drift from RPI doctrine is invisible until JDM catches it. EBR gets elected. MS DGI gets allocated. MYGA language leaks through. Each correction costs 5-10 minutes of CEO time.

Three Components

Component 1

matrix.json — The Data

Machine-readable mirror of the Preferred Provider Matrix. Contains: carriers (tier + lineup), products (mechanics + rates + doctrine), decision tree (client_shape → product mapping), default allocations (locked per product), doctrine rules, and reference links. Single source of truth — HTML matrix renders FROM this JSON (or stays in sync via build step).

Component 2

recommend_product() — The Tool

MDJ agent tool that accepts structured intake (source_type, has_surrender, horizon, spouse_coverage, age_band) and walks the decision tree. Returns deterministic output. Can be called directly with all params (from the Annuity Lion after Q&A) OR called in state-machine mode where it asks questions one round at a time.

Component 3

Annuity Lion — Enhanced System Prompt

Annuity Lion's specialist prompt gets a new instruction block: "When a team member describes a client's money situation, call recommend_product() instead of answering from your own knowledge. Ask clarifying questions if the intake params aren't clear. Never recommend a product, allocation, or rider configuration without calling the tool first."

Architectural Constraint — Model Inference Locked Out

The Annuity Lion MUST call the tool for any product recommendation. It cannot self-answer from training data. This is the whole point: the Decision Tree is deterministic, the doctrine is locked, the output is auditable. If the Lion answers without calling the tool, it's a bug. The system prompt must be explicit about this — "You do not know RPI's product doctrine. The tool does. Call it."

The Intake State Machine — 5 Questions

When a team member says "I've got a client who..." and the Annuity Lion doesn't have enough information to call recommend_product() with all required params, it asks clarifying questions. Maximum 5 rounds — most cases resolve in 2-3.

RoundQuestionMaps ToOptions (not exhaustive)
1 "What's the money source?" source_type CD at maturity · Existing annuity (in surrender) · Existing annuity (past surrender) · IRA · Roth IRA · Cash / savings · Inheritance · 401k rollover
2 "Are there surrender charges on the incoming money?" has_surrender + surrender_pct No (liquid) · Yes, under 3% · Yes, 3-5% · Yes, over 5% · Not sure (flag for verification)
3 "What's the goal — grow it, generate income soon, or generate income later?" horizon Accumulation · Income soon (1-5 yr) · Income later (5-10+ yr) · Both (accumulation that converts to income)
4 "Does the client need spousal coverage — should income continue to the surviving spouse?" spouse_coverage Yes (joint life) · No (single life) · Not applicable (no spouse)
5 "How old is the client? (or age range)" age_band Under 60 · 60-70 · 71-80 · 81-85 · Over 85

Shortcut Detection

If the team member's initial message contains enough info to skip rounds, the Lion skips. Example: "Client is 67, CDs just matured, $400K, wants accumulation, married" → all 5 params are extractable → Lion calls the tool directly, no Q&A needed. The state machine is for partial-info cases.

Decision Tree Walk — Deterministic Output

Once all params are collected, recommend_product() walks the tree:

INPUT: { source_type, has_surrender, surrender_pct, horizon, spouse_coverage, age_band } STEP 1: Age gate → age_band == "81-85" → Ameritas GMIB / NAC GA 5/7 (narrow niche) → age_band == "over_85" → "Manual review — outside standard product menu" → else → continue STEP 2: Horizon filter → horizon == "income_later" → income products path → horizon == "income_soon" → income products path (shorter deferral) → horizon == "accumulation" → accumulation products path → horizon == "both" → accumulation w/ income rider option STEP 3: Bonus need (accumulation path only) → has_surrender == true AND surrender_pct > 5% → CP10 + EBR (narrow case) → has_surrender == true AND surrender_pct <= 5% → CP10 base (12% covers it) → has_surrender == false → CP10 base OR VC10 OR Control X (rate-driven pick) STEP 4: Income products path → spouse_coverage == true → Joint Life factor (Nassau IA or NAC BenSol) → spouse_coverage == false → Single Life factor → preferred carrier logic (Nassau IA for income flagship, BenSol for GMIB stack) STEP 5: Source-type modifiers → source_type == "roth" → Nassau Growth 10-yr (tax-free growth chassis) → source_type == "cd_matured" → emphasize "no bonus needed" message → source_type == "existing_annuity_in_surrender" → flag anniversary-before-1035 OUTPUT: { lead_product, alt_product, default_allocation, doctrine_flags: ["EBR_OFF", "NO_MYGA", etc.], brochure_urls, rate_spotlight_urls, illustration_spec: { carrier, product, premium, allocation_split, rider_elections }, caveats: ["Verify anniversary date before 1035", etc.] }

The learning loop is alive: tool reads, VP writes, doctrine stays current.

Together: The engine can't drift because it reads the matrix. The matrix can't go stale because the VP maintains it. The VP can't hallucinate recommendations because it doesn't make them — the tool does. Three layers, three distinct functions, no collision. Same pattern as MEGAZORD + ATLAS + Rangers.
