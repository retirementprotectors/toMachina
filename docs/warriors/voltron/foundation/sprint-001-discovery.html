What VOLTRON Becomes

VOLTRON v2 is JDM's replacement in RPI daily operations. Not in the sense of a chatbot that answers questions — but an agent that can execute multi-step work across the entire Platform the way JDM would direct a team member to do it.

Today: "What accounts does Jim Aswegan have?" → VOLTRON looks it up and tells you.
Tomorrow: "Prep Jim Aswegan for next Tuesday's review meeting" → VOLTRON reads his accounts, pulls his RMDs, checks the pipeline, drafts talking points, and returns a compiled brief.

The key insight: VOLTRON doesn't need a UI upgrade. The same ProDashX App that hosts the current chat widget gets a second mode — a task deployment mode — that wires to the Agent SDK backend. The user types a goal. VOLTRON works. The user gets results.

VOLTRON's Permanent Home: MDJ_SERVER is not a temporary dev host. All MCP servers run on MDJ_SERVER. Agent SDK sessions must be co-located with MCP processes to use them natively. Redundancy is GitHub (code) + Firestore/Drive (state). Cloud Run is for the toMachina API — not for VOLTRON.
Additive Approach — The Non-Negotiable
Sprint 001 is ADDITIVE. It adds Mode 1 (Agent SDK) alongside the existing Mode 2 (Raw SDK). It does NOT remove, rewrite, or break any current functionality. Mode 2 (the chat interface) stays exactly as-is. Mode 1 is a new endpoint, a new client wrapper, and a new ProDashX UI section layered on top.

Rationale: Mode 2 is live, in use by the team, and battle-tested. Mode 1 is new and unproven in production at this org. The safe path: ship Mode 1 in parallel, validate it, then decide in a future sprint if Mode 2 should be migrated or deprecated. Sprint 001 does not answer that question — it doesn't need to.

What Stays

Agent SDK handles the execution loop. VOLTRON's custom layer still owns the things that are RPI-specific and cannot be generic:

  • Specialist routing — keyword + Claude classification against Firestore configs. The SDK doesn't know our 6 specialists. We pre-route before calling query().
  • Tool filtering by user level — filterToolsForUser() enforces entitlement gates. The SDK executes whatever tools it sees, so we control what it sees.
  • TM API tools — the 57 toMachina API tools stay as custom definitions. The SDK auto-discovers MCP tools; TM API tools are REST and still need explicit definition or an MCP wrapper.
  • Auth middleware — Firebase auth token validation stays in Express. The SDK doesn't handle this.
  • Memory extraction + Firestore persistence — post-conversation insight storage stays custom.
  • Mode 2 entirely — the existing Raw SDK chat endpoint (/agent/chat) is not touched. It runs as-is.
  • FORGE runner — src/forge/ is completely untouched by Sprint 001. RONIN runs in the same process and uses the same server.

SSE Event Translation

The Agent SDK produces SDKEvent types. The ProDashX widget expects specific SSE event shapes. The voltron-sdk-client bridges them:

{
  // SDK: ContentBlockDelta (text) → SSE:
  { type: "text", text: "..." }

  // SDK: ToolUse → SSE:
  { type: "tool_call", tool_name: "...", tool_input: {...}, tool_use_id: "..." }

  // SDK: ToolResult → SSE:
  { type: "tool_result", tool_name: "...", result: {...} }

  // SDK: approval needed (canUseTool returns false) → SSE:
  { type: "approval_required", tool_name: "...", tool_input: {...}, call_id: "..." }

  // Session complete → SSE:
  "[DONE]"
}

canUseTool Callback (replaces ApprovalManager for Mode 1)

The Agent SDK provides a canUseTool callback that fires before each tool execution. Return true to allow, false to pause and emit an approval_required SSE event. The existing tool-filter.ts + toolRequiresApproval() logic feeds directly into this callback.

Key difference from Mode 2's ApprovalManager: In Mode 2, the approval manager uses a Promise that blocks the async loop. In Mode 1, canUseTool returning false tells the SDK to pause — the session is suspended until POST /api/voltron/approve/:sessionId resumes it. Cleaner, built-in.

interface IntentResult {
  mode: 'deploy' | 'chat'
  confidence: number        // 0.0 – 1.0
  reasoning: string         // For logging/debugging
  model_used: 'keyword' | 'haiku'  // Which path fired
}

Logic Flow

  1. Tokenize and lowercase the message
  2. Score against Mode 1 keyword list (deploy/build/create/run/prep/generate/set up/onboard/process/compile/send/schedule/automate/handle/take care of)
  3. Score against Mode 2 keyword list (what/who/when/where/how/tell me/show me/list/find/lookup/check/is there/do we have)
  4. If Mode 1 score >= 1 and > Mode 2 score → return mode: 'deploy', model_used: 'keyword'
  5. If Mode 2 score >= 1 and > Mode 1 score → return mode: 'chat', model_used: 'keyword'
  6. If tied or no signal → Claude Haiku classification call (max 128 tokens, cheap)
  7. Fallback default: mode: 'chat' (safe default)

Implementation: POST /webhook/deploy

~50 lines of Express. No external dependencies. Added to server.ts. No auth middleware — uses GitHub HMAC signature validation instead.

Signature Validation

// Validate GitHub webhook signature
import crypto from 'crypto'

function validateGitHubSignature(payload: string, signature: string, secret: string): boolean {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(payload, 'utf8')
    .digest('hex')
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expected)
  )
}

Deploy Sequence

// Webhook handler (conceptual — RONIN writes the real implementation)
app.post('/webhook/deploy', express.raw({ type: '*/*' }), async (req, res) => {
  const sig = req.headers['x-hub-signature-256'] as string
  const secret = process.env.GITHUB_WEBHOOK_SECRET

  if (!validateGitHubSignature(req.body.toString(), sig, secret)) {
    return res.status(401).json({ success: false, error: 'Invalid signature' })
  }

  const payload = JSON.parse(req.body.toString())

  // Only deploy on push to main
  if (payload.ref !== 'refs/heads/main') {
    return res.json({ success: true, data: { action: 'ignored', reason: 'not main branch' } })
  }

  // Respond immediately (GitHub expects 200 in < 10s)
  res.json({ success: true, data: { action: 'deploy_started' } })

  // Run deploy async
  runDeploy().catch(err => logger.error('[webhook] Deploy failed:', err))
})

runDeploy() Logic

async function runDeploy(): Promise<void> {
  const { execSync } = await import('child_process')
  const DEPLOY_DIR = '/home/jdm/mdj-agent'

  try {
    // Pull latest
    execSync('git pull origin main', { cwd: DEPLOY_DIR, stdio: 'inherit' })

    // Install dependencies
    execSync('npm ci --production=false', { cwd: DEPLOY_DIR, stdio: 'inherit' })

    // Build
    execSync('npx tsc', { cwd: DEPLOY_DIR, stdio: 'inherit' })

    // Restart service (keeps old process running until new one is ready)
    execSync('sudo systemctl restart mdj-agent.service', { stdio: 'inherit' })

    logger.info('[webhook] Deploy complete — VOLTRON restarted')

  } catch (err) {
    // Build failed — DO NOT restart. Old version keeps running.
    logger.error('[webhook] Deploy failed — keeping current version:', err)
    notifyError({
      title: 'VOLTRON Deploy Failed',
      message: err instanceof Error ? err.message : String(err),
      severity: 'error',
    })
  }
}
Rollback safety: If npm ci or tsc fails, systemctl restart is never called. The current running version continues serving requests. The failure is logged and a Slack notification fires.

OS Enforcement for RONIN
The hookify gap, why it matters, and how to compensate.
The Anti-Hallucination Golden Rule
"If you didn't READ IT, don't REPORT IT." — VOLTRON never fabricates data. Every fact in a response traces back to a tool call result.
The Hookify Gap

The RPI Operating System enforces 37 rules via the hookify plugin in Claude Code sessions. These rules fire as hooks on Write, Edit, and Bash tool calls — blocking violations before code is written.

RONIN's Agent SDK sessions bypass hookify entirely. When RONIN calls query(), the resulting agent runs in a fresh Claude context with no hookify plugin loaded. The 37 rules do not fire. This is not a bug — it is a structural gap that must be compensated with explicit instructions.

The solution: the 10 most critical rules are re-encoded as explicit instructions in RONIN's system prompt template (src/forge/templates.ts). These rules travel with every RONIN invocation regardless of execution context.
