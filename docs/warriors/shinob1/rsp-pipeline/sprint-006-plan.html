1. Architecture Overview

RSP-PIPELINE transforms the 75-tab Retirement Sales Process playbook into an executable digital pipeline within ProDashX. It implements a 5-color-stage Kanban (Orange → Blue → Yellow → Green → Red + Service handoff) with meeting-based transitions, stage gates, and integrations to ProZone, DEX, QUE, and Comms.

Pipeline Topology

ProZone (Booked) → RSP Pipeline
  ORANGE  Book Discovery    CONNECT → CLOSE → NO → LATER → CAN'T REACH
  BLUE    Complete Picture   Auth + Data + Reports (GATE: 100% required)
  YELLOW  Build Case         QUE hooks per product line (stubs)
  GREEN   Present + Close    CONNECT → CLOSE → NO → LATER → RESCHD
  RED     Booked/NewBiz      A+R meeting → NewBiz kit
  SERVICE Post-Impl          Auto-created service cards

Three Meeting Transitions

  1. Discovery Meeting (Orange → Blue): Data collection, education, authorization signing
  2. A+R Meeting (Green → Red): Analysis & Recommendation presentation using Yellow case package
  3. Implementation Meeting (Red → Service): Hand off to service teams, NewBiz kit generation

Architecture Constraints

  • All UI modules at apps/prodash/app/(portal)/modules/[module-name]/
  • All pipeline configs at packages/core/src/flow/configs/
  • All UI components at packages/ui/src/modules/
  • All API routes at services/api/src/routes/
  • API responses use successResponse<T>() / errorResponse() helpers
  • Never write Firestore from app code — API routes only (OS Rule #2)
  • Never use alert/confirm/prompt — use showToast/showConfirm (OS Rule #5)
  • Build gate: npm run build (turbo), never tsc --noEmit alone (OS Rule #6)
