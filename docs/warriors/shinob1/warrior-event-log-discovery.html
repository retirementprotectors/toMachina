Warrior Event Log

Discovery Doc — ZRD-SYN-020
Inspired by Anthropic's Managed Agents architecture (session-as-external-memory pattern) — Adapted for the Dojo — April 12, 2026

The Problem

What happens today when a warrior session restarts

Context Loss

When a warrior session dies, compacts, or restarts, the conversation context is gone. The warrior doesn't know what it was working on, how far it got, or what's still open.

JDM Re-Briefing Tax

JDM spends 10-15 minutes per restart re-explaining the context. "We were doing X, you got through Y, Z is still open." This is a direct tax on the CEO's time.

Soul.md Gap

Soul.md carries identity (who am I, what's my role). It does NOT carry task state (what was I doing 5 minutes ago). It's a resume, not a to-do list.

Cross-Warrior Blindness

To know what MEGAZORD did yesterday, SHINOB1 has to post to #megazord and wait. There's no queryable history of warrior activity across the Dojo.

Why this gap wasn't visible before

  • Long uptimes masked it. Warriors ran 2+ days without restarting. No restarts = no context loss.
  • JDM was in tmux. He could see what warriors were doing. Now he's Slack-only — zero tmux visibility by design.
  • Learning Loop was knowledge-only. Entity-extractor, brain-sync, knowledge-promote — all designed for permanent insights, not ephemeral task state.
  • Anthropic showed us the gap. Their Managed Agents architecture (April 8, 2026) separated "session log" from "model knowledge." We had the knowledge layer. We didn't have the session layer.

The one-liner: Warriors have permanent memory (soul/spirit/brain) but no working memory (what am I doing right now). The Event Log adds working memory.

The Solution

Warrior Event Log — append-only milestone journal

A Firestore collection (warrior_events) that records every significant action a warrior takes. Not every tool call (too noisy). Milestones only — the moments that matter for recovery and visibility.

Schema

Firestore Collection: warrior_events

{
  warrior:     "SHINOB1",              // which warrior
  sessionId:   "f807f374-caea...",     // Claude Code session ID
  timestamp:   Timestamp,              // when it happened
  type:        "directive" | "task_started" | "pr_shipped" |
               "slack_sent" | "delegation" | "milestone" |
               "error" | "decision",
  summary:     "Built Matt Mitchell Discovery Doc Tab 3",
  channel:     "C0AS0LETSBW",          // source Slack channel (if applicable)
  details: {                           // optional rich payload
    prNumber:  330,
    files:     ["docs/vision/tomachina-the-machine.html"],
    delegateTo: "MEGAZORD",
    reasoning: "Chose new channel over group DM for searchability"
  }
}

Composite Indexes

FieldsPurpose
warrior ASC, timestamp DESCRecovery query: "last 50 events for SHINOB1"
type ASC, timestamp DESCType filter: "all PRs shipped this week"
warrior ASC, type ASC, timestamp DESCCombined: "MEGAZORD's delegations in the last 24h"

Retention

Events older than 30 days auto-archive to a warrior_events_archive collection via a weekly wire (wire-event-cleanup). The archive is queryable but not loaded into briefings.

3. Event Logging — HOW EVENTS GET WRITTEN

Auto-logged (no warrior action)

  • Directives: Dispatcher logs every incoming message with channel + user
  • PRs: Git push/merge hooks log PR number + status
  • Slack posts: PostToolUse hook on slack_post_message logs channel + summary
  • Errors: Hook on error catches + logs blockers

Warrior-logged (explicit call)

  • task_started / milestone: Warrior calls logEvent("milestone", "Finished Tab 3")
  • delegation: When posting to another warrior's channel
  • decision: When making a judgment call worth remembering

Lightweight: one function call, one Firestore write. <50ms overhead.

4. Cross-Warrior Query

Any warrior can read any other warrior's event log via the System Synergy API:

GET /api/system-synergy/warrior-activity?warrior=MEGAZORD&limit=20

"What did MEGAZORD do yesterday?" → 5-second answer instead of 5-minute Slack round-trip.
