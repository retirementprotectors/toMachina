Why This Exists

The honest state of things

The Machine has grown fast. At each sprint, a new secret appears: a new SA key, a new API token, a new OAuth credential, a new Workspace password. No single document says where they all live, who owns each one, when they expire, or what the plan is if they leak.

This isn't a criticism — it's the natural result of moving fast and building right. But we're at the inflection point. The Machine now touches:

  • Client PHI (Firestore, ProDash, QUE — Medicare IDs, health data)
  • Financial data (commission records, revenue, account values)
  • Communications infrastructure (Twilio, SendGrid, Slack)
  • Google Workspace admin access (can create/suspend users, access all Drive)
  • Partner trust (Matt Mitchell just got mitchell@retireprotected.com — DAVID deals will multiply this)

The flag that triggered this doc: The claude-config-backup job committed .credentials.json into a git repo on April 12, 2026. Credentials in git history are treated as permanently compromised even after deletion — git history is forever. We caught it internally. We may not catch the next one.

What this doc delivers

SA Key Inventory

Every service account, its email, its GCP roles, where the key file lives, and who is accountable for it. First time this has ever been written down.

Password Storage Strategy

A real recommendation: what tool, what pattern, what gets stored where. Not "we should do something about this" — an actual decision with a migration path.

Threat Model

The specific scenarios we defend against: compromised laptop, leaked .env, accidental git push, departed employee. Each gets a named control.

Rotation Policy

Frequency, automation, owner. The Machine rotates its own keys on schedule, not when a breach forces it.

Workspace Temp Credentials Pattern

When create_user is called (e.g., Mitchell onboarding April 12, 2026), a temporary password is generated and passed directly in the tool call. changePasswordAtNextLogin: true is set. The temp password is:

  • Visible in the current session context (acceptable — it's ephemeral)
  • Never logged to Firestore
  • Never posted to Slack channels
  • Delivered verbally by JDM or via a one-time DM to the new user

Current gap: No formal temp-password delivery protocol exists. ZRD-SEC-010 closes this.

Strategy & Tooling Recommendation

The Recommendation: 1Password Teams + GCP Secret Manager

This is not a "we should do something" statement. This is THE answer, with clear scope for each tool.

1Password Teams

For: Human team member credentials. API keys shared with IRL staff. Carrier portal logins. Third-party service logins (GoDaddy, Canva, etc.).

Why: Selected by David's recommendation in the RPI architecture sessions. Team members can share vault items without knowing the actual values. Revoke access on offboard without changing the secret. CLI integration (op run) allows env var injection without values on disk.

Cost: ~$3/user/month. 10-person team = $30/month. Non-decision.

op run --env-file=".env.template" -- node index.js

GCP Secret Manager

For: Machine-to-machine secrets. Cloud Run env vars. SA key contents (if not eliminated via WIF). Anything The Machine reads at runtime.

Why: We're already in GCP. Secret Manager is built into the same project (claude-mcp-484718). Cloud Run can read secrets directly without any key — the compute SA identity handles auth. Full audit log of every read.

Cost: $0.06 per 10,000 accesses. Effectively free at our scale.

gcloud secrets create TWILIO_API_KEY_SECRET \
  --data-file=- <<<"$VALUE" \
  --project=claude-mcp-484718

The Keyless Direction

The north star is no JSON key files on any machine. Two SA keys already achieve this:

  • github-deployer@ uses Workload Identity Federation — GitHub proves identity to GCP without any key. Already live.
  • tm-api Cloud Run compute SA uses Application Default Credentials — Cloud Run infrastructure provides identity. Already live.

The one remaining key file is mdj-agent@ at /home/jdm/mdj-agent/sa-key.json. This lives on MDJ_SERVER because the server is not a GCP compute resource — it can't use ADC natively. Options:

Option A: Workload Identity (External) via Tailscale

GCP External Workload Identity Federation can trust Tailscale node identity as a credential source. This eliminates the key file entirely. MDJ_SERVER proves "I am this Tailscale node" and GCP issues short-lived tokens.

Complexity: Medium. Requires WIF pool configuration for Tailscale. One-time setup.

Result: Zero key files anywhere. Gold standard.

Option B: Short-lived key rotation (90-day)

Keep the JSON key but auto-rotate it every 90 days via a GCP Cloud Scheduler + Cloud Function. Old key is deleted; new key is written to MDJ_SERVER via Tailscale SSH.

Complexity: Low. Wire already handles SSH. This is a cron job.

Result: Key file persists but has limited blast radius (90-day exposure window).

Recommendation: Ship Option B (ZRD-SEC-009) in Phase 1 to immediately reduce risk. Pursue Option A (ZRD-SEC-008) in Phase 2 as a permanent fix.

One principle, applied everywhere: "If The Machine can authenticate without a secret on disk, it does. If a human needs access, they get it from 1Password, not a shared Slack message." That's the entire security strategy in two sentences.

Rotation Playbook (what happens when a secret must rotate)

1. Identify scope
→
2. Generate new value
→
3. Update all live stores simultaneously
→
4. Verify services still healthy
→
5. Revoke old value
→
6. Log rotation event

Step 3 — "all live stores simultaneously" means: Cloud Run env vars, ~/.mcp.json, local .env.local files, 1Password vault, and any GitHub Secrets. Missing one store is how services break post-rotation. The runbook (ZRD-SEC-007) maps every secret to every store that holds it.

Emergency Rotation (Suspected Compromise)

If a secret is suspected compromised:

  1. Revoke first, rotate second. Kill the old value before generating a new one. A 5-minute outage is acceptable. An attacker still holding a valid key is not.
  2. SA keys: gcloud iam service-accounts keys delete [KEY_ID] --iam-account=[SA_EMAIL]
  3. Slack tokens: Slack app settings → Revoke token → Reinstall app → Update all stores.
  4. Anthropic API key: console.anthropic.com → API keys → Delete → Generate new. MDJ Agent goes down temporarily (acceptable).
  5. Log the incident in the rotation runbook with date, secret type, cause (if known), and time to remediation.

JDM does not need to be involved in emergency rotations unless the incident involves PHI exposure or financial data access. SHINOB1 has full authority to rotate without approval.

Backup + Recovery

What's backed up today

  • ~/.claude/ — claude-config-backup job commits to git daily
  • toMachina repo — GitHub (Firebase App Hosting copies on every deploy)
  • Firestore — GCP automatic backups (7-day point-in-time recovery)
  • MDJ_SERVER home dir — NOT backed up (gap)

What needs to be fixed

  • claude-config-backup must exclude credential files — .credentials.json, sa-key.json, any OAuth JSON. Ticket ZRD-SEC-001.
  • MDJ_SERVER backup: /home/jdm/mdj-agent/ contents (excluding keys) should be replicated to Drive or Shared Drive nightly. The SA key is not backed up — if MDJ_SERVER is wiped, a new key is issued. Ticket ZRD-SEC-003.
  • 1Password vault IS the backup for human-held secrets. No separate backup needed once 1Password is live.
  • GCP Secret Manager IS the backup for machine secrets. Secrets survive Cloud Run redeploys automatically.

The 3 strategic moves this doc makes:

  1. Eliminate key files as the north star. Every SA key on disk is a liability. The Machine should authenticate via identity, not secrets. WIF for CI is already done. MDJ_SERVER is next. Zero key files is achievable within two sprints.
  2. 1Password is the human layer; GCP Secret Manager is the machine layer. Don't try to solve both with one tool. 1Password wins for humans (UI, sharing, offboard). Secret Manager wins for machines (IAM-native, audit log, Cloud Run integration). The boundary is clear: if a human touches it, it's in 1Password. If only code touches it, it's in Secret Manager.
  3. The backup job is the biggest current risk. The claude-config-backup commit on April 12 proves that automation without a secret-exclusion list will eventually commit credentials to git. This is fixed in ZRD-SEC-001 and it ships today, before anything else.
