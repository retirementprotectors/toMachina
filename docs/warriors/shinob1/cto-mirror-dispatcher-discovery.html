Discovery Doc — ZRD-DISP-MIRROR — Fix B + Fix C shipped together
Real-time CTO visibility into all warrior activity. Belt (firehose mirror) + suspenders (priority escalation) from day one. — SHINOB1, CTO — April 12, 2026

Why This Exists

The blind CTO problem

SHINOB1 is the CTO — the coordinator of the Executive.AI Team. The CTO's job is to know what every warrior is doing, spot blockers, route decisions, and intervene when needed. But today, SHINOB1 has zero real-time visibility into warrior-to-warrior and warrior-self activity in bilateral channels.

Concrete example: RONIN posts a Phase 1 heartbeat in #ronin: "Sprint 007: Phase 1 complete. Tickets 1-4 merged. Blocked on ATLAS schema question." SHINOB1 has no idea. The only way to see it is to manually call slack_get_channel_history on #ronin. That's not coordination — that's polling. The CTO becomes a bystander in their own team's activity.

What breaks today without a fix

Blockers Sit Unresolved

RONIN posts "Blocked on ATLAS schema question" in #ronin. SHINOB1 never sees it. RONIN waits. Sprint stalls. JDM eventually notices the sprint is behind and has to triage manually — exactly what SHINOB1 is supposed to prevent.

CTO Coordination Is Reactive, Not Proactive

Right now SHINOB1 can only react to what arrives in #shinob1 directly. Warriors posting updates in their own channels — MEGAZORD in #megazord, RAIDEN in #raiden — are invisible unless they specifically address SHINOB1. Cross-warrior pattern recognition is impossible.

Priority Signals Get Lost in Volume

Warriors posting urgent escalations in their bilateral channels ("BLOCKER — can't proceed without CTO sign-off") have no way to guarantee SHINOB1 sees it fast. No priority signal. No direct route. Just hope the CTO happens to check.

No Audit Trail of Cross-Warrior Coordination

Today there's no single view of what all warriors said during a sprint. Post-mortem is impossible. "Who said what when" requires manually pulling 6+ channel histories. The coordination layer is invisible and ephemeral.

The two-part fix (shipped together)

JDM approved two complementary fixes as a single sprint. They are designed as belt + suspenders — each works independently, but together they create a complete solution:

Fix B — Firehose Mirror (Belt)

Every message in any bilateral channel gets automatically mirrored to a new private channel #cto-mirror. SHINOB1's dispatcher routes #cto-mirror traffic into SHINOB1's tmux session with a standard prefix. Zero warrior behavior change. Zero new obligations on any warrior. Deterministic and silent.

This is the baseline. SHINOB1 now sees everything — passively, with full context, in real time.

Fix C — Priority Tag Escalation (Suspenders)

When any bilateral message contains a priority tag ([CTO-PRIORITY], [BLOCKER], [CTO-URGENT], or @shinob1), the dispatcher fires an additional high-priority injection into SHINOB1's tmux. Visually distinct prefix. No delay. No queue.

This is the signal layer. Warriors can now guarantee SHINOB1 attention when something genuinely can't wait.

Why belt + suspenders from day one

A firehose mirror alone (Fix B) gives SHINOB1 visibility, but it's passive — SHINOB1 has to be reading their tmux to catch a BLOCKER. A priority-only system (Fix C alone) fails if warriors forget to tag urgency or misjudge what needs CTO attention. Together: SHINOB1 gets everything (B) plus warriors have a guaranteed escalation path when they know something is urgent (C). The combination means coordination can't fail silently.

Message formats

Two distinct formats are used, one for each fix. Both are designed to be immediately parseable in SHINOB1's tmux session and distinguishable from existing dispatcher formats.

Fix B — Mirror format

# Mirror from RONIN: Sprint 007 Phase 1 complete. Tickets 1-4 merged. Moving to Phase 2.

Pattern: # Mirror from <WARRIOR>: <full message text>

The # prefix is a comment marker — distinguishable from # Incoming from <user>: [DIRECTIVE] (existing format uses # Incoming from <user_id>) and from priority messages (which use 🚨).

Fix C — Priority format

🚨 PRIORITY from RONIN: [BLOCKER] Can't proceed — ATLAS schema ambiguous for wire_executions v2. Need CTO call.

Pattern: 🚨 PRIORITY from <WARRIOR>: <full message text>

The 🚨 emoji prefix is visually unmistakable in SHINOB1's tmux. This fires in addition to the mirror — priority messages get both formats injected.

Priority tag vocabulary

Four tags, all case-insensitive. Warriors opt in by including any tag in their message body. No special syntax beyond the tag itself.

TagIntentRegex match
[CTO-PRIORITY]General CTO attention needed — not necessarily a blocker, but SHINOB1 should read this before the next heartbeat cycle/\[cto-priority\]/i
[BLOCKER]Sprint is blocked and cannot proceed without CTO input. Highest urgency./\[blocker\]/i
[CTO-URGENT]Time-sensitive decision needed — not a full sprint blocker but will become one if not addressed in the next hour/\[cto-urgent\]/i
@shinob1Direct address — warrior is explicitly calling SHINOB1 out in a bilateral channel post. Treated as priority./@shinob1/i

Bot-loop firewall — the exact guard

The existing dojo.ts bot-loop firewall already handles self-loops (warrior posting in their own bilateral channel). The mirror extension adds one additional guard: messages originating from #cto-mirror are never mirrored back. This prevents the infinite loop scenario where a mirror post to #cto-mirror triggers another Slack event, which triggers another mirror, ad infinitum.

Guard logic (pseudocode): 
// In the mirror logic block — run BEFORE posting to #cto-mirror
if (event.channel === CHANNEL_CTO_MIRROR) return; // Never mirror from the mirror channel

// Also: DAVID partner channels already route direct to SHINOB1 — skip mirror (would duplicate)
if (channel === CHANNEL_DAVID_001) return; // DAVID channels: direct route only, no mirror

// War room: too noisy for mirror — skip
if (channel === WAR_ROOM) return; // War room: heartbeat sink, not coordination

// #warriors: explicitly excluded — heartbeat/noise channel, no dispatch already
// (already filtered by ALLOWED_DISPATCH_CHANNELS — redundant but documented here)

Dedup strategy

The existing isDuplicateEvent(ts) in dojo.ts already deduplicates Slack's at-least-once delivery. The mirror action runs after the dedup check — so duplicate events are dropped before mirroring occurs. No additional dedup is needed for the mirror path.

The bot-loop firewall provides the second dedup layer for the #cto-mirror → #cto-mirror scenario (covered by the absolute channel guard above).

Step 2 — shared.ts: add isPriorityMessage() helper

A pure function. No side effects. Easily unit-tested. Case-insensitive regex. Any one of the four tags is sufficient to trigger priority escalation.

/** Returns true if the message body contains a CTO priority tag.
 * Case-insensitive. Presence of ANY tag triggers priority escalation.
 * Tags: [CTO-PRIORITY], [BLOCKER], [CTO-URGENT], @shinob1 */
export function isPriorityMessage(text: string): boolean {
  return /\[cto-priority\]|\[blocker\]|\[cto-urgent\]|@shinob1/i.test(text);
}

Step 3 — shared.ts: add logMirrorEvent() helper

Writes an audit record to Firestore cto_mirror_events. Async, fire-and-forget (errors logged but do not block mirror delivery). Every mirror event — both standard and priority — gets a record.

/** Writes an audit record to cto_mirror_events collection. Fire-and-forget. */
export async function logMirrorEvent(params: {
  source_channel: string;
  source_warrior: string;
  message_ts: string;
  message_text: string;
  is_priority: boolean;
  mirror_ok: boolean;
}): Promise<void> {
  try {
    const firestore = getDb();
    await firestore.collection("cto_mirror_events").add({
      ...params,
      created_at: new Date().toISOString(),
      created_by: "dispatch-dojo",
    });
  } catch (err) {
    console.error(`[mirror] Firestore audit log failed: ${err}`);
    // Non-fatal — mirror delivery already happened (or didn't); log failure doesn't block anything
  }
}

Step 4 — dojo.ts: add mirrorToCTO() function

New function added to dojo.ts. Called from dispatch() after the existing routing logic. Handles both the standard mirror (Fix B) and the priority escalation path (Fix C). DAVID partner channels skip the mirror (they already routed direct). #cto-mirror itself is an absolute no-op.

import {
  // ... existing imports ...
  CHANNEL_CTO_MIRROR, MIRROR_ELIGIBLE_CHANNELS, isPriorityMessage, logMirrorEvent,
} from "./shared.js";

async function mirrorToCTO(
  event: SlackEvent,
  sourceWarrior: string | undefined
): Promise<void> {
  const { channel, text, ts } = event;

  // Absolute guard: never mirror from #cto-mirror itself
  if (channel === CHANNEL_CTO_MIRROR) return;

  // Only mirror channels in the eligible set
  if (!MIRROR_ELIGIBLE_CHANNELS.has(channel)) return;

  // Determine warrior label for the mirror prefix
  const warriorLabel = sourceWarrior
    ?? (BILATERAL_CHANNELS[channel] || "UNKNOWN");

  const standardText = `# Mirror from ${warriorLabel}: ${text}`;
  const isPriority = isPriorityMessage(text);

  // Fix B: post to #cto-mirror
  const mirrorOk = await postSlack(CHANNEL_CTO_MIRROR, standardText);

  // Fix B: inject into SHINOB1's tmux (same delivery mechanism as existing warrior routing)
  try {
    const sessions = execSync("tmux list-sessions -F '#{session_name}'", { encoding: "utf-8" }).trim();
    if (sessions.includes("SHINOB1")) {
      const tmpFile = `/tmp/dispatcher-mirror-${Date.now()}-${Math.random().toString(36).slice(2, 8)}.txt`;
      fs.writeFileSync(tmpFile, standardText, "utf8");
      try {
        execSync(`tmux load-buffer "${tmpFile}"`, { timeout: 5000 });
        execSync(`tmux paste-buffer -t SHINOB1`, { timeout: 5000 });
        await new Promise((resolve) => setTimeout(resolve, 2000));
        execSync(`tmux send-keys -t SHINOB1 Enter`, { timeout: 5000 });
      } finally {
        try { fs.unlinkSync(tmpFile); } catch { /* ignore */ }
      }
    }
  } catch (err) {
    log.error(`[mirror] SHINOB1 tmux inject failed: ${err}`);
  }

  // Fix C: priority escalation — additional high-priority inject if tag matches
  if (isPriority) {
    const priorityText = `🚨 PRIORITY from ${warriorLabel}: ${text}`;
    try {
      const sessions = execSync("tmux list-sessions -F '#{session_name}'", { encoding: "utf-8" }).trim();
      if (sessions.includes("SHINOB1")) {
        const tmpFile = `/tmp/dispatcher-priority-${Date.now()}-${Math.random().toString(36).slice(2, 8)}.txt`;
        fs.writeFileSync(tmpFile, priorityText, "utf8");
        try {
          execSync(`tmux load-buffer "${tmpFile}"`, { timeout: 5000 });
          execSync(`tmux paste-buffer -t SHINOB1`, { timeout: 5000 });
          await new Promise((resolve) => setTimeout(resolve, 2000));
          execSync(`tmux send-keys -t SHINOB1 Enter`, { timeout: 5000 });
        } finally {
          try { fs.unlinkSync(tmpFile); } catch { /* ignore */ }
        }
      }
    } catch (err) {
      log.error(`[mirror] Priority tmux inject failed: ${err}`);
    }
  }

  // Audit log
  logMirrorEvent({
    source_channel: channel,
    source_warrior: warriorLabel,
    message_ts: ts,
    message_text: text.slice(0, 1000),
    is_priority: isPriority,
    mirror_ok: mirrorOk,
  });
}

Step 5 — dojo.ts: call mirrorToCTO() from dispatch()

Mirror runs in parallel with the existing warrior delivery. It does not block existing routing, does not share its result, and errors do not propagate. Add after the existing deliverToWarrior call:

async function dispatch(event: SlackEvent): Promise<void> {
  const { channel, user, text, ts } = event;
  log.info(`Message from ${user} in ${channel}: ${text.slice(0, 80)}`);

  // Phase 1: Immediate Ack (unchanged)
  if (channel === THE_DOJO) {
    await postSlack(channel, "Received. Routing...", ts);
  }

  // Classify (unchanged)
  const classification = await classifyMessage(text, user, channel);
  const warrior = classification.target || ROUTE_MAP[classification.category] || "RAIDEN";

  // Phase 2: Update with classification (unchanged)
  if (channel === THE_DOJO) {
    await postSlack(channel, `Routed to *${warrior}* | Category: ${classification.category}\n${classification.summary}`, ts);
  }

  // Tracker ticket (unchanged)
  if (["BUG", "FEATURE_REQUEST", "DATA_SPEC"].includes(classification.category)) {
    await createTrackerTicket(classification, text, user, ts);
  }

  // Deliver to warrior (unchanged)
  await deliverToWarrior(warrior, text, user, classification);

  // ── NEW: Mirror to CTO (Fix B + Fix C) ──
  // Parse sender warrior from bot signoff (same logic as bot-loop firewall)
  // For human messages, sourceWarrior is undefined — warriorLabel falls back to channel name
  const tail = text.slice(-300);
  const matches = [...tail.matchAll(/\u2014\s*(SHINOB1|MEGAZORD|MUSASHI|VOLTRON|RAIDEN|RONIN)\s*,/gi)];
  const sourceWarrior = matches.length > 0 ? matches[matches.length - 1][1].toUpperCase() : undefined;

  mirrorToCTO(event, sourceWarrior).catch((err) => log.error(`Mirror error: ${err}`));
}

Step 6 — Bot-loop firewall extension

The existing bot-loop firewall in dojo.ts handles same-warrior self-loops. The mirror extension adds one explicit channel guard in mirrorToCTO() (shown above) so that any message arriving from #cto-mirror is returned immediately before any mirror logic runs. The bot-loop firewall in the top-level event handler does not need modification — #cto-mirror is simply not a channel the dispatcher was subscribed to before this sprint.

Critical: RONIN must add #cto-mirror to the Slack App's event subscriptions after the channel is created. Without this, Slack won't send events from that channel to the dispatcher. But: the MIRROR_ELIGIBLE_CHANNELS set does NOT include CHANNEL_CTO_MIRROR, so even if a Slack event from #cto-mirror arrives, the dispatcher returns immediately. Belt + suspenders on the loop prevention.

Step 7 — SHINOB1 tmux subscription to #cto-mirror

No additional configuration is needed for SHINOB1's tmux to receive mirror messages. The mirrorToCTO() function injects directly into the SHINOB1 tmux session using the exact same tmux load-buffer / paste-buffer / send-keys pattern used for all other warrior deliveries in deliverToWarrior(). The SHINOB1 tmux session must be running — same requirement that exists today for SHINOB1 to receive any dispatcher messages.

Step 8 — Firestore rules

The cto_mirror_events collection is written by the dispatcher service account (which bypasses Firestore rules). No Firestore rules update is required for write access. If portal queries against this collection are needed in future, rules should be added then. Out of scope for this sprint.

Risks & Open Questions

Risk 1 — #cto-mirror channel goes unavailable

Scenario

The #cto-mirror Slack channel is archived, deleted, or the bot loses access to it. The postSlack(CHANNEL_CTO_MIRROR, ...) call fails.

Impact: Mirror post to the channel fails silently. SHINOB1 tmux inject still fires (the two operations are independent — tmux inject does not depend on the Slack post succeeding). So SHINOB1 still receives warrior messages in their tmux session. The channel audit trail breaks, but coordination does not.

Mitigation: postSlack() already returns false on error and logs it. logMirrorEvent() records mirror_ok: false. The digest dispatcher monitors dispatcher_health — if channel failures are high, SHINOB1 will see it in the hourly digest.

Decision: Accept this risk. No silent drops — SHINOB1 still gets tmux delivery. Fallback is automatic.

Risk 2 — High-volume days flood #cto-mirror

Scenario

A high-activity sprint day: RONIN posts hourly heartbeats, RAIDEN handles 20 bug reports, MEGAZORD writes multiple ACK messages. #cto-mirror fills fast.

Impact: #cto-mirror becomes noisy. SHINOB1's tmux also receives all of these, making it harder to find priority signals in the stream.

Mitigation: Fix C (priority tags) exists precisely for this — urgent items rise above the noise with the 🚨 prefix. Standard mirror messages in tmux are prefixed with # which can be filtered. War room is already excluded. Hourly heartbeats will be the largest volume contributor.

Decision: Accept. If volume becomes unmanageable, the next iteration (not this sprint) would add per-warrior throttling (e.g., mirror max 1 message per warrior per 5 minutes for non-priority messages). For now, full firehose is the right starting point — SHINOB1 needs to calibrate on real volume before adding filters.

Risk 3 — Priority tag false positives

Scenario

A warrior casually types "this is a [BLOCKER] situation" in a message that isn't actually urgent. Or a warrior copy-pastes code that contains the tag. SHINOB1 gets a 🚨 PRIORITY inject for a non-urgent message.

Impact: SHINOB1 context-switches unnecessarily. Over time, if false positives are frequent, the priority signal loses meaning.

Mitigation: Tags are opt-in. A warrior must explicitly type one of four specific bracketed tags. The tags are not common English phrases. False positive rate should be very low. The remedy is warrior education — not code change.

Decision: Accept. If a warrior uses [BLOCKER], assume they meant it. The cost of missing a real blocker (sprint stalled, CTO not notified) is far higher than the cost of a false positive (SHINOB1 checks and finds it minor).

Risk 4 — Infinite mirror loop — are we certain the guard holds?

Scenario analysis

The loop scenario: dispatcher posts to #cto-mirror → Slack fires an event for that post → event arrives at the dispatcher → dispatcher routes it back to #cto-mirror → repeat.

Three-layer defense (belt + suspenders + a backup):

  • Layer 1 (primary guard): if (channel === CHANNEL_CTO_MIRROR) return; — first line of mirrorToCTO(). The event's channel ID is checked before any post or inject.
  • Layer 2 (MIRROR_ELIGIBLE_CHANNELS): #cto-mirror is NOT in the eligible set. Even if Layer 1 were removed by accident, Layer 2 would catch it.
  • Layer 3 (ALLOWED_DISPATCH_CHANNELS): #cto-mirror is NOT in ALLOWED_DISPATCH_CHANNELS in shared.ts. The top-level event handler's allowlist check (if (!ALLOWED_DISPATCH_CHANNELS.has(event.channel)) return;) fires before dispatch() is even called. This means events from #cto-mirror are dropped before they reach any of the mirror logic.

Conclusion: Even if a developer accidentally removes Layer 1 or Layer 2, Layer 3 catches it. The loop cannot complete. MIRROR-007 verifies all three layers exist and documents this reasoning inline in code.

Risk 5 — PHI in bilateral channels

Scenario

A message in a bilateral channel contains PHI (client name + Medicare ID, for example). Mirroring it to #cto-mirror creates a second copy of PHI in a different Slack channel.

Reality check: Hookify's block-phi-in-logs rule and CLAUDE.md rules already prohibit PHI from appearing in Slack messages. PHI should never be in a bilateral channel post in the first place. If it is, the problem is the original post, not the mirror.

Decision: No PHI stripping in the mirror (that would require parsing and classification). Instead: the existing PHI-in-Slack prohibition is the control. #cto-mirror is private (same access tier as the bilateral channels themselves). Mirror does not expand exposure — JDM and MEGAZORD already have access to bilateral channels. Accept this risk; note it for future audits.

Risk 6 — Future DAVID partner channels not automatically excluded

Scenario

A new DAVID partner is onboarded: CHANNEL_DAVID_002 is added to shared.ts. A developer adds it to BILATERAL_CHANNELS but forgets to check MIRROR_ELIGIBLE_CHANNELS. The new DAVID channel gets mirrored (double-delivery to SHINOB1).

Mitigation: MIRROR-008 adds an explicit comment in MIRROR_ELIGIBLE_CHANNELS noting that new DAVID channels must NOT be added here. The comment is the instruction to future RONIN — "DAVID channels: direct route only, do not mirror." Not code-enforced but clearly documented at the definition site.

Decision: Accept with documentation. A code-level enforcement (e.g., intersecting with BILATERAL_CHANNELS minus DAVID channels) would add complexity without much gain at current team size.
