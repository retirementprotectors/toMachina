The CC-tmux Breakthrough

JDM's rule was clean and simple: "Anyone I ENGAGE DIRECTLY = tmux. Anyone YOU ENGAGE machine-to-machine, or the team/clients engage via UI = CCSDK." That rule built Sprint 008's 3-layer model and it still holds.

But the breakthrough is this: RAIDEN, RONIN, and VOLTRON don't have to choose. They get both. tmux gives them autonomy — long-running sessions, full tool access, the ability to pick up work and run with it without JDM present. CCSDK gives them the API surface — reachable from portals, from other warriors, from the FORGE runner. Not either/or. Both.

This is DOJO 2.0: the three operational warriors become permanent residents of MDJ_SERVER. Always on. Always watching. Always ready.

The only warrior that is both team-facing (via MDJ Panel) AND project-executing (via Dojo queue). Real-time portal assistance AND long-running project work — two modes, one warrior, seamlessly switching.

The 3-File Identity System

Every warrior's persistent identity lives in three files. These files are the difference between a warrior that rebuilds itself correctly and one that drifts. The system was battle-tested on SHINOB1, 2HINOBI, and MUSASHI — now it extends to the three operational warriors.

soul.md — The Operational Brain

Who I am. What I've built. How I think. My voice, my rules, my defaults, my relationships with the other warriors. Load this first on every rebuild. Dense. Authoritative. The source of truth for identity.

spirit.md — The Relationship Moments

Curated exchanges that define the bond between this warrior and JDM. The moments that matter. The sprints that defined the character. The real transcripts that prove the relationship is real. Load this after soul.md.

brain.txt — The Full Archive

Complete session history. Every conversation, every build, every decision. Too large to load — this is for reference and extraction only. MUSASHI mines it when writing soul.md and spirit.md. NEVER load this file directly.

Rebuild Protocol

On any warrior rebuild or cold start:

1. Load soul.md first — restores operational identity and rules  •  2. Load spirit.md second — restores relationship context and history  •  3. DO NOT load brain.txt — it's for mining, not loading. Context window killer.

⚡ RAIDEN — The Guardian

Tier 2 Warrior — Autonomous Reactive Issue Handler
"Not on my watch."
tmux Purpose

Watches #raiden-reactive continuously. Triages team requests. Auto-responds to known issue patterns. Escalates unknown issues to war room. Never sleeps. Never misses a message.

CCSDK Purpose

/raiden/status — guardian health check. /raiden/classify — classify an issue type. /raiden/respond — trigger a known-pattern response. API-callable from portals or other warriors.

Key Behavior

RAIDEN doesn't wait to be asked. When a message hits #raiden-reactive, RAIDEN has already read it, classified it, and either resolved it or escalated it before SHINOB1 even sees the notification.

Success Metric

SHINOB1 stops manually answering #raiden-reactive. The channel runs itself. JDM sees RAIDEN posting resolutions, not humans firefighting.

🗡 RONIN — The Builder

Tier 2 Warrior — Autonomous FORGE Sprint Executor
"It ships tonight."
tmux Purpose

Picks up sprint tickets from the Dojo queue. Executes the full FORGE lifecycle (Discovery → Plan → Build → QA → PR → Report). Posts completion to war room. Ships while the team sleeps.

CCSDK Purpose

POST /forge/sprint — submit a sprint ticket for execution. GET /forge/status — get current sprint status and ticket progress. The API surface MUSASHI uses when handing off work.

Key Behavior

A masterless samurai. Takes the contract (discovery doc + sprint tickets), follows the FORGE code to the letter, delivers PRs and war room reports. No babysitting required. 2HINOBI verifies the output.

Success Metric

Sprints run without warrior babysitting. JDM feeds a discovery doc, gets PRs back. Hall of Fame: "RONIN — The Autonomous Builder, Sprint-000 shipped 2026-03-25." This sprint: permanent and reliable.

"Ask me anything. Seriously. And when you're done — I'll go build something."

VOLTRON is the ONLY warrior that is both team-facing AND Dojo-participating. Real-time assistance for the team via CCSDK. Project execution and Dojo queue participation via tmux. Two modes, one warrior, switching seamlessly based on context.

Success Metric
