One Surgery, Three Outcomes

This sprint rewires existing infrastructure to deliver three things simultaneously:

  1. Sidebar reorganization — flat icon list becomes 4 collapsible tiers (Owner's Box, Executive Floor, Leader Level, Team Tray), gated by user level
  2. Permission enforcement — API routes actually respect what Admin panel sets (85 of 87 routes currently ignore permissions)
  3. Admin config update — App Config and Team Config show tray groupings, defaults auto-populate from level assignment
The principle: Everything listed below already exists in the codebase. The permission engine (900 lines), the RBAC middleware (with Firestore cache), the admin panel (1,864 lines), the entitlement evaluator, the module definitions (47 modules), the role templates (6 templates). We are connecting dots, not building new ones.

The Unbroken Chain

When JDM flips a toggle in Admin panel, this must happen end-to-end:

Admin Panel (Team Config) ↓ writes to Firestore users/{email}.module_permissions ↓ read by Auth Middleware (enriches req.user on every API call, 60s cache) ↓ checked by Route Guard (requireLevel + requireModuleAccess from rbac.ts) ↓ enforced at API Response (403 if unauthorized) ↓ reflected in Sidebar (canAccessModule hides tray / app icon) ↓ reflected in UI Component (V/E/A controls what buttons render)

Today the chain is broken at step 3 — 85 routes skip the guard. This sprint completes the circuit.

Firestore Rules

Comprehensive rule set with isRPIUser(), isOwner(), isExecutiveOrAbove(), isLeaderOrAbove() helpers. Covers all collections. This is the real enforcement layer for client-side reads/writes. The API uses Admin SDK which bypasses these — route guards are the only protection for API calls.

What's Broken — The Chain Gaps

Critical: VOLTRON Entitlement Bypass

req.user.role is NEVER populated for Firebase-authenticated users. No setCustomUserClaims() exists anywhere. VOLTRON routes default unknown roles to ADMIN (rank 5, highest). Every authenticated user executes wires at full admin entitlement.

FileLineWhat Happens
voltron-wire.ts172req.user?.role || 'ADMIN' — always ADMIN
voltron-registry.ts116req.user?.role — undefined → defaults ADMIN

High: 85 of 87 Routes Skip RBAC

rbac.ts exists with proper requireLevel() + Firestore lookup + cache. Only 2 routes use it:

  • PATCH /api/users/:email — requireLevel('EXECUTIVE')
  • PUT /api/unit-defaults/:unitKey — requireLevel('OWNER')

Everything else is either unprotected or uses ad-hoc inline guards that duplicate the logic without the cache.

High: system-synergy.ts Guard is a Stub

Named requireExecutive but only checks @retireprotected.com domain. All 10 platform observability endpoints accessible to any authenticated user.

High: leadership.ts Has Zero Protection

Meetings CRUD, roadmaps, actions, dashboard — all open to any @retireprotected.com user.

Medium: Admin Pages Have No Server-Side Guard

/admin, /admin/guardian, /admin/intake-queue render for anyone who navigates directly. Sidebar hides the links but the pages load.

Medium: GET /api/users Exposes Full Roster

Any authenticated user can list all team members including level, role, module_permissions. Should require LEADER+.

Sidebar: Flat Icon List

All apps shown in one flat section. No tray grouping. No collapse/expand. Level gating works (via canAccessModule) but the visual organization doesn't communicate the hierarchy.

The Rewire — 6 Steps

No new features. Connect what exists.

Step 1: Enrich Auth Middleware

File: services/api/src/middleware/auth.ts

After Firebase token verification, call the same Firestore lookup that rbac.ts already does. Attach level, role_template, module_permissions to req.user. Use the existing 60-second cache from rbac.ts.

Change: ~15 lines. Import getUserProfile from rbac, call after token verify, spread onto req.user.

Result: Every route handler now has req.user.level, req.user.module_permissions available without any route-level code.

Replace all ad-hoc inline guards with requireLevel() from rbac.ts. Each route group gets a single guard at mount or router level.

Files: voltron-wire.ts, voltron-registry.ts, voltron-deploy.ts

// Map numeric level to VOLTRON_ROLE_RANK const LEVEL_TO_VOLTRON: Record<number, string> = { 0: 'ADMIN', // OWNER 1: 'VP', // EXECUTIVE 2: 'DIRECTOR', // LEADER 3: 'COORDINATOR', // USER } const voltronRole = LEVEL_TO_VOLTRON[req.user.level] || 'COORDINATOR'

Admin UI — Two Views, One Truth

Both views read/write the same Firestore data: users/{email}.module_permissions + users/{email}.level. They are two lenses into one truth.

App/Module Config View

"What does this app look like?"

  • Pick an app (e.g., ProZone)
  • See which tray it lives in
  • See all users with their V/E/A access for this app
  • Toggle V/E/A per user
  • Changes immediately reflect in Team Config view

Team Config View

"What does this person see?"

  • Pick a user (e.g., Nikki)
  • See her level → which trays she sees
  • See all modules grouped by tray with V/E/A status
  • Assign level → defaults auto-populate
  • Override individual modules after defaults
  • Changes immediately reflect in App Config view
