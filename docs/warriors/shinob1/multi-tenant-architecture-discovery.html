Multi-Tenant Firestore Architecture

Discovery Doc — ZRD-PLAT-MT — Platform Infrastructure
JDM + SHINOB1 — April 12, 2026 — "Your data is yours. Day one. Full stop."

Why This Exists

The business case in one sentence

Matt Mitchell just became DAVID Partner #1. There will be a Partner #2 and #3. Every one of them will ask a version of the same question before they sign anything: "What happens to my 6,000 clients' data if this partnership ends?" Right now, the answer is uncomfortable. This doc is how we fix that.

"Your data is yours." That's the promise JDM makes to every DAVID partner. Today, The Machine cannot technically keep that promise — all partner data lives in the same (default) Firestore database as RPI's own data, with no physical boundary between them. Multi-tenant architecture is what makes the promise true.

The four forces driving this

Exit Clause Portability

Matt's term sheet will include an exit clause. That clause means: if the partnership dissolves, Matt gets his clients' data back — cleanly, completely, immediately. With a single shared DB, "cleanly" is a manual export process that takes days and risks leaving data behind. With per-partner isolation, it's one command: export the partner-midwest-medigap database. Done in minutes.

HIPAA Comfort

We don't need to claim separate databases are legally required for HIPAA — they aren't. But every sophisticated partner and their counsel will ask about data isolation. "Your clients' Medicare data lives in the same database as another partner's clients" is a friction point in every enterprise deal. Physical isolation eliminates the question entirely. The audit story is clean.

Cross-Tenant Leak Prevention

Today's Firestore rules are written for a single-tenant world: isRPIUser() grants access across all 29,000+ documents. As partner users are onboarded, that function becomes a landmine. A misconfigured rule or a query that forgets a filter could surface Matt's client records to RPI staff or vice versa. Physical separation at the DB level means a bug in a query can never reach another tenant's data.

Per-Partner Analytics

Matt will want a dashboard showing his book's performance — his clients, his revenue, his AEP pipeline. RPI will want to see cross-portfolio analytics across all partners. Per-partner databases feed into per-partner BigQuery datasets. Cross-partner rollup happens in BigQuery, where there's no risk of live cross-tenant data access. Clean analytics from day one.

The current state: what's wired in today

The platform has 556 direct getFirestore() calls across 87 API route files and cron jobs — every single one assumes a single Firestore database (the GCP default). The @tomachina/db package exports a getDb() function (used in frontend) that initializes against the same implicit default. The Firestore rules file has one set of rules that applies to that one database.

None of this is wrong for where The Machine was six months ago. It's a real refactor for where it needs to go. The question is how we sequence it.

DAVID's scaling trajectory

Matt is Partner #1. The DAVID channel strategy calls for 3–5 partners in 2026, 10–15 by 2027. Each partner brings a book of business measured in hundreds to thousands of clients. At 5 partners, a single-tenant architecture becomes untenable — and retrofitting it after 5 firms are already loaded is a much harder job than wiring it right for Partner #1.

The window is now. Matt's book hasn't landed in Firestore yet. MEGAZORD built the intake field mapper (PR #345). That wire has nowhere to write yet. This is the greenfield moment — we design the destination before we move the data.

What We're Building

The three options — and why Option B wins

Option A: Shared DB with partner_id field

Add a partner_id field to every client/account document. All data stays in the (default) database. Queries add .where('partner_id', '==', partnerId). Firestore rules check the field.

Why rejected:

  • One bug (a query without a partner_id filter) surfaces another tenant's data. No physical safety net.
  • Exit clause portability is still a manual export problem — you cannot export one tenant's data cleanly without scanning the entire collection.
  • Firestore rules that check document fields for tenant scope are notoriously fragile and cannot be validated with the same confidence as DB-level isolation.
  • The easiest path today becomes permanent technical debt. "We'll refactor later" is how we end up explaining a cross-tenant leak to Matt's attorney.

REJECTED

Option B: Per-Partner Named Firestore Database — RECOMMENDED

GCP Firestore supports multiple named databases within a single GCP project (GA since 2023). Each partner gets a named database: partner-midwest-medigap, partner-xyz, etc. RPI stays in (default). Shared catalogs (carriers, products, campaigns) live in (default) and are read-only accessed via a shared-catalog DB reference.

Why this wins:

  • Physical isolation: a query in the wrong tenant context gets a permissions error, not a data leak.
  • Exit portability: gcloud firestore export --database=partner-midwest-medigap. One command. Clean.
  • Firestore rules per database: Matt's DB has rules that only recognize Matt's users. RPI users can have a super-admin claim that crosses DBs for support purposes, but it's explicit and audited.
  • BigQuery streaming: each DB has its own export sink. Per-partner analytics and cross-partner rollup are architecturally clean.
  • Cost: no per-DB minimum charge. Named databases use the same free tier quota as the default. At RPI's current write volume, this is $0 additional cost.
  • Same GCP project, same billing account, same IAM, same observability stack. We're not managing multiple cloud accounts.

RECOMMENDED

Option C: Per-Partner GCP Project

Each partner gets their own GCP project with their own Firestore, their own Firebase project, their own billing. Maximum isolation — each partner's data is literally in a different cloud account.

Why deferred: The overhead is significant. Separate GCP projects means separate CI/CD targets, separate IAM management, separate billing, and a dramatically more complex deploy pipeline. The isolation benefit over Option B is marginal for HIPAA purposes. Reserve Option C for partners with explicit regulatory mandates that require separate billing or separate cloud accounts (e.g., certain government contractors, HIPAA-covered entities that require contractually separate infrastructure). Matt doesn't need this.

DEFERRED — regulatory exceptions only

Firebase Auth custom claims design

Auth stays in a single Firebase Auth pool. Custom claims on the ID token carry tenant context. The API extracts claims from the verified token — no round-trip to Firestore on every request.

// Custom claim shapes
// RPI user (full platform access)
{ role: 'rpi', email_verified: true }

// Partner admin (manages their tenant)
{ role: 'partner_admin', partner_id: 'midwest-medigap' }

// Partner agent (accesses their tenant's data)
{ role: 'partner_agent', partner_id: 'midwest-medigap' }

// Super admin (JDM + John) — crosses all tenant DBs for support
{ role: 'superadmin', partner_ids: ['*'] }

The getDb(partnerId) helper contract

The single most important code change in this entire doc. One function replaces the raw getFirestore() call that appears 556 times.

// packages/db/src/firestore-admin.ts (server-side, new file)
// Uses firebase-admin (not firebase client SDK)

const DB_CACHE = new Map<string, Firestore>()

export function getDb(partnerId?: string): Firestore {
  // No partnerId → default database (RPI data + shared catalogs)
  const dbId = partnerId ? `partner-${partnerId}` : '(default)'

  if (DB_CACHE.has(dbId)) return DB_CACHE.get(dbId)!

  const app = getAdminApp() // singleton admin app
  const db = getFirestore(app, dbId)
  DB_CACHE.set(dbId, db)
  return db
}

BigQuery per-partner sink

Each partner DB streams to its own BigQuery dataset: partner_midwest_medigap. Cross-partner rollup happens in BigQuery views, not in Firestore. JDM's executive analytics query the rollup view. Matt's analytics query his dataset only. No live cross-tenant access at any layer.

Technical Implementation — How It Gets Built

The migration truth: this is an API-level refactor, not a rewrite

The good news: the @tomachina/db frontend package (packages/db/src/firestore.ts) uses the Firebase client SDK, which is auth-scoped by Firebase rules. The multi-tenant rules change applies here. But the heavy lifting is in services/api/ — 87 route files call getFirestore() directly via firebase-admin.

The approach:

  1. Create a tenant-aware getDb(partnerId) helper in a new services/api/src/lib/db.ts module that wraps firebase-admin/firestore's named DB support.
  2. Add API middleware that extracts partner_id from the verified Firebase Auth token and attaches it to req.partnerId.
  3. Refactor route handlers to replace getFirestore() with getDb(req.partnerId). Since req.partnerId is null for RPI users, they continue hitting the (default) DB transparently.
  4. Create partner-specific Firestore rules per named DB. These rules are simpler than the default rules — they only need to validate that the authenticated user belongs to that partner.
  5. Build the provisioning script that creates a new partner DB, seeds it with required structure, sets rules, and registers the partner in partner_registry.
  6. Wire MEGAZORD's intake to point at partner-midwest-medigap when writing Matt's imported data.

What stays the same for RPI users

Every RPI user (Vince, Nikki, Aprille, etc.) continues to work exactly as today. Their auth token has no partner_id claim, so getDb(undefined) returns the (default) database. From their perspective nothing changed. The refactor is transparent to them.

What changes for partner users

Partner users authenticate with their own @midwestmedigap.com email via Google Workspace SSO (or email/password if they don't use Google Workspace). On first sign-in, a Cloud Function sets their custom claim: { role: 'partner_agent', partner_id: 'midwest-medigap' }. Every subsequent API call automatically routes to their tenant DB. The portal UI looks the same — same modules, same navigation — but data is scoped to their book.

Super-admin cross-tenant access

JDM and John Behn have the superadmin claim. A super-admin support view in RIIMO can query any partner DB by passing an explicit partner_id query param. The API checks for superadmin claim before allowing cross-tenant access. Every cross-tenant read is logged to guardian_audits in the (default) DB.

The collectionGroup complication

The accounts route currently uses db.collectionGroup('accounts') to query across all clients. With named databases, a collection group query is scoped to that database — it cannot span databases. This is actually a feature, not a problem: partner users can only collectionGroup-query their own DB. RPI users collection-group-query the default DB. Cross-partner rollups happen in BigQuery. No code change needed beyond the getDb() swap.

VOLTRON and the MDJ Agent

VOLTRON currently uses the mdj-agent SA key to query Firestore. That SA key needs access to both the (default) DB and any partner DB it serves. The SA already has project-level Firestore access — named databases are included automatically. The MDJ Agent code needs to be updated to call getFirestore(app, dbName) when a partner context is passed via the conversation (from the auth token in the portal widget request). One file change in ~/mdj-agent/.

Timeline & Sequencing — Three Paths

All estimates assume RONIN execution speed on this codebase. The route-swap work (MT-003–006) is mechanical — it's the kind of task where RONIN's speed exceeds a human dev by 4–5x. The architecture tickets (MT-001, MT-002, MT-008, MT-009) require careful thought and SHINOB1 review cycles that cannot be compressed.

Parallel track capacity: This sprint can run 3–4 parallel RONIN tracks efficiently. Track A (route refactor) is 4 tickets that can subdivide further. Track B (auth/provisioning) and Track C (intake/analytics) are independent of each other after MT-001 + MT-002 land. Wall-clock time is significantly lower than sum-of-hours for that reason.

Path 1 — Quick & Dirty: Ship Matt in default with partner_id field

Tickets: Only MT-010 (modified to write a partner_id field on each doc). Skip everything else.

Honest hour estimate: 2–3 hours of RONIN time. Matt's data is in Firestore today.

When Matt can see a demo: Tomorrow.

The real cost:

  • Physical isolation is not achieved. A Firestore rules bug or a query without a filter = cross-tenant data exposure. You're betting on code quality, not architecture.
  • Exit portability is still a manual export problem. When Matt's attorney asks for a data export on exit, you're running a collection query and hoping nothing is missed.
  • Every partner that joins after Matt lands in the same shared DB. At Partner #3–4, you WILL do this refactor under pressure, while partner data is live, with less runway.

Risks & Open Questions

Critical risks

Firestore rules mistake → HIPAA incident

Scenario: A bug in the partner Firestore rules template allows an RPI user to read Matt's clients' collection, or allows Matt's staff to read another partner's data. With 6,000+ Medicare clients' PHI in the partner DB, this is a reportable HIPAA breach — even if the read was accidental and limited.

Mitigation: MT-009 (rules template) requires SHINOB1 review and security rules simulator validation before it deploys to any real partner DB. MT-012 (isolation test suite) must pass on the live API before data moves. The rules template is simple by design — the fewer lines in a security rules file, the fewer ways it can be wrong.

Severity: CRITICAL

Shared users with cross-tenant read needs

Scenario: JDM or John Behn needs to see Matt's data for support or QA purposes. The super-admin claim handles this — but if the claim logic is wrong, they either can't read it (support is blocked) or can read everything without logging (audit gap).

Mitigation: Super-admin access is implemented in the API layer, not the Firestore rules. Every cross-tenant read via the API explicitly writes to guardian_audits. The claim check is binary: role === 'superadmin'. MT-013 (the RIIMO support view) makes this access intentional and audited, not accidental.

Severity: MEDIUM

VOLTRON serves RPI data to partner users before MT-014 ships

Scenario: If Path 3 (Hybrid) is chosen and MT-014 is deferred, a partner user who opens the MDJ Panel and asks about "my clients" could receive results from the default DB (RPI's clients). This is a cross-tenant PHI exposure via the AI layer.

Mitigation: In Path 3, disable the MDJ Panel for partner users until MT-014 ships. The portal widget can check for the partner_id claim and hide the panel if VOLTRON isn't partner-aware yet. One-line conditional in the portal sidebar component.

Severity: MEDIUM — manageable with a portal feature flag

Firestore named database creation API permissions

Scenario: The provisioning script (MT-008) calls the GCP Firestore Admin API to create a new named database. The service account used for Cloud Run (tm-api) may not have the datastore.databases.create IAM permission — this permission was historically restricted. The script may fail on the database creation step even though it has full Firestore data access.

Mitigation: Run the provisioning script manually the first time from JDM's gcloud CLI (which has owner-level IAM). Once confirmed, grant the necessary IAM role to the SA for subsequent automated provisioning. If the IAM issue surfaces: one console click by JDM, TCO, done. Alternatively, pre-create databases manually via GCP console and have the script skip creation if the DB already exists.

Severity: LOW — operational, not architectural

ATLAS tooling tenant scope

Scenario: ATLAS tools in the source_registry and tool_registry are in the (default) DB and are global. If a partner ever needs tenant-specific ATLAS configurations (e.g., partner-specific import sources), the current architecture has no mechanism for that.

Mitigation: ATLAS stays global (Option B, as designed). Per-partner intake configuration lives in the partner's DB as a separate intake_config collection (already in the per-partner collection list). ATLAS source and tool registries describe what's possible; per-partner intake config describes what's activated. This boundary is clean and doesn't need architecture change.

Severity: LOW — solved by design

Open questions

QuestionAnswer
Any per-DB minimum charge in Firestore? No. Named databases in Firestore use the same pricing model as the default database. No per-DB minimum. At our write volume (<1K writes/day per partner), the marginal cost per partner DB is $0. Pricing changes only at >50K writes/day per DB or >1GB storage per DB.
Firestore concurrency limits — does named DB model affect throughput? Each named database has its own 5,000 writes/second limit. This is an improvement over a shared DB. Matt's 6,000 clients at bulk import pace might hit 200–300 writes/second during intake, well under 5K. The named DB model actually gives us more headroom, not less.
What about org_structure — shared or per-partner? The org collection in (default) is RPI's internal org structure (divisions, teams). Partners have their own org structure (staff, agents). Partners get their own org collection in their DB. The two are not shared. Matt's team roster has nothing to do with RPI's org structure.
MCPs and CCSDK — does VOLTRON need per-partner SA credentials? No. The mdj-agent SA key already has project-level Firestore access, which includes all named databases in the same project. VOLTRON just needs to know which database to query (determined by the user's partner_id claim in the portal widget auth token). No additional credentials needed.
What if a partner staff member leaves? Offboarding? Firebase Auth offboarding disables the user account, which revokes the custom claim. No per-DB-level user management needed. Standard Google Workspace offboard process applies. The custom claim provisioning Cloud Function (MT-007) is the only new step: it fires on sign-in, not on offboard, so no offboard trigger is needed for the claims themselves.
Can two partners' staff see each other's data if they share a domain? The custom claim is tied to partner_id from the partner_registry domain lookup, not just the email domain. If two different agencies happen to use the same email provider (e.g., both use Gmail), the lookup fails to find a match and defaults to no partner claim (access blocked). Partners with shared email domains need a different claim-setting mechanism — handled case-by-case via manual claim assignment. Covered in the provisioning script notes.
What about the BigQuery streaming Cloud Function that was disabled in Sprint 5? The BigQuery streaming is handled by a Cloud Function in services/bigquery-stream/. MT-011 extends its config to include named databases. The disabled service (Sprint 5) was replaced by a direct streaming mechanism — check current state before MT-011 executes. If the streaming is already live via another mechanism, MT-011 scope narrows to adding the new DB to whatever config that uses.

The single most important risk in this document

The VOLTRON PHI cross-contamination risk (before MT-014). Every other risk in this doc has a clean technical mitigation. The VOLTRON risk is the one where the failure mode is silent: partner user opens MDJ Panel, asks a question, gets an answer based on another tenant's PHI, doesn't notice, and the breach goes undetected. The mitigation is a feature flag on the portal sidebar. But it must be in place before any partner user is given access to the portal. This is the non-negotiable item that JDM needs to be aware of when sequencing Path 3.
