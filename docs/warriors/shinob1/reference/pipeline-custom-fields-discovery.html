The Problem

"Opportunities do not have any information/data fields for use by admin teams. Existing system has product specific sets of data fields for information retention: product, tax status, funding information, notes field, document section, premium/face value, IMO."
Nikki Gray — #the-dojo, April 2

Opportunity forms are blank. When Nikki creates a Life opportunity, there's no place to enter the policy number, face amount, or paramed date. When she creates an Annuity opportunity, there's no tax status or IMO field. Each pipeline type needs its own set of fields.

The Solution

Define a custom_field_schema per pipeline. The opportunity create/edit form dynamically renders the correct fields based on which pipeline the opportunity belongs to. Field values stored on the opportunity document.

8 Pipeline Schemas

Life (13 fields), Annuity (8), Investments (10), MAPD/PDP (7), Med Supp (7), Reactive Medicare (5), Reactive Retirement (5), Delivery (5).

Zero New Collections

Schema stored on the pipeline document. Values stored on the opportunity document. Both already accept arbitrary fields via index signatures.

1 New UI Component

Dynamic field renderer that reads the schema and generates the form. Text, currency, date, dropdown, textarea, file upload.

Data Flow

1. SCHEMA DEFINITION (one-time setup per pipeline) PUT /api/flow-admin/pipelines/NBX_LIFE Body: { custom_field_schema: [ ...field definitions... ] } Stored on: flow_pipelines/NBX_LIFE.custom_field_schema 2. FORM RENDERING (every time user opens create/edit opp) Frontend reads: GET /api/flow-admin/pipelines/:key Extracts: pipeline.custom_field_schema Renders: Dynamic form fields based on schema 3. VALUE STORAGE (on opportunity save) POST /api/opportunities Body: { pipeline: "NBX_LIFE", stage: "new", custom_fields: { policy_number: "...", face_amount: 500000, ... } } Stored on: opportunities/{id}.custom_fields (JSON object) 4. VALUE DISPLAY (on opportunity detail view) Frontend reads: GET /api/opportunities/:id Reads: opportunity.custom_fields Renders: Field values using the pipeline's schema for labels + types

Schema Format

Each pipeline's custom_field_schema is an array of field definitions:

{ "custom_field_schema": [ { "key": "policy_number", // Firestore field key "label": "Policy Number", // Display label "type": "text", // text | currency | date | dropdown | textarea | file "required": false, // Validation "options": null, // For dropdowns: ["Option A", "Option B"] "order": 1 // Display order }, { "key": "face_amount", "label": "Face Amount", "type": "currency", "required": false, "options": null, "order": 2 } ] }

Key Decision: Values on Opportunity, Not Flow Instance

Flow instances already have custom_fields but those are for pipeline engine metadata. Opportunity custom fields are business data (policy numbers, premiums). Store them on the opportunity document in a custom_fields object. The opportunity API already passes through any field via the index signature.
