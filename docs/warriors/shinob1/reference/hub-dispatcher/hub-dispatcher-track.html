The Problem

"Incoming work arrives through five channels and JDM is the router. That's the bottleneck." — SHINOB1, CTO Discovery

The intent classifier only knows deploy vs chat. The triage engine only routes to RAIDEN. Neither knows about MEGAZORD, VOLTRON, or MUSASHI. Data import requests sit in RAIDEN's fix queue. Creative briefs land as bug reports. Feature requests get triaged as incidents. JDM manually re-routes — every day, multiple times a day.

Hub Dispatcher eliminates the manual sorting. One triage point. Keyword scoring with Haiku fallback. Five CXO destinations. Zero JDM intervention.

The Solution

Incoming item (email, Slack, MDJ chat, Dojo Reporter) | v HUB DISPATCHER (keyword scoring + Haiku fallback) | +---> Data problem? ---> MEGAZORD (#megazord-intake / megazord@) ZRD- +---> Client case? ---> VOLTRON (#voltron-cases / voltron@) VOL- +---> Creative request? ---> MUSASHI (Command Center) MUS- +---> Bug / broken thing? ---> RAIDEN (#the-dojo / dojo@) RDN- +---> Feature / build? ---> RONIN (Dojo intake) RON- +---> Junk / spam? ---> Archive (auto-discard)

RON-HD01 Hub Dispatcher Core Routing Logic

Create hub-dispatcher.ts — the central routing engine. Keyword scoring against 5 CXO keyword sets. If no keyword match exceeds confidence threshold (0.6), fall back to Claude Haiku for classification. Returns the CXO target, confidence score, ticket prefix, and routing channel.

File Paths

services/api/src/raiden/hub-dispatcher.ts — CREATE: core routing engine packages/core/src/raiden/hub-dispatcher-types.ts — CREATE: shared types (DispatchResult, CxoTarget, DispatchChannel) services/api/src/routes/dispatch.ts — CREATE: POST /api/dispatch endpoint services/api/src/server.ts — EDIT: mount dispatch route

Acceptance Criteria

  • dispatchItem(text: string, channel: DispatchChannel): Promise<DispatchResult> exported from hub-dispatcher.ts
  • DispatchResult type: { cxo_target: CxoTarget, confidence: number, ticket_prefix: string, routing_destination: string, method: 'keyword' | 'haiku' }
  • CxoTarget union: 'megazord' | 'voltron' | 'musashi' | 'raiden' | 'ronin' | 'archive'
  • Keyword scoring: each CXO has a weighted keyword list. Score = (matched keywords / total words) with exact-match bonus (2x) for high-signal terms like "import", "broken", "brochure"
  • Confidence threshold at 0.6 — below this, Haiku fallback is triggered
  • Haiku fallback: single-shot prompt with the 5 CXO descriptions + input text, returns JSON { target, confidence }
  • Archive target for spam patterns: empty body, known spam phrases ("unsubscribe", "click here to win"), single-word junk
  • POST /api/dispatch endpoint accepts { text, channel }, returns DispatchResult
  • Structured response: { success: true, data: DispatchResult }
  • npm run type-check passes — types exported from @tomachina/core barrel
  • npm run build passes

Architecture Decision

Why keyword scoring + Haiku fallback (not Haiku-only)? Keyword scoring is instant (0ms), deterministic, and free. Haiku adds ~200ms latency and API cost per call. 80%+ of intake items will match keywords cleanly. Haiku is the safety net for ambiguous cases — "my client's data isn't showing up" could be MEGAZORD (data) or VOLTRON (client case). Haiku resolves the ambiguity. This is the same belt-and-suspenders pattern used across The Machine.

RON-HD04 Email Intake Dispatcher

Build the email intake channel. Poll the intake@retireprotected.com inbox on MDJ_SERVER using the Gmail API (service account). For each unread email: extract subject + body, run through Hub Dispatcher, create a ticket in the correct CXO queue, mark email as read. This is the largest ticket — full email-to-ticket pipeline.

Acceptance Criteria

  • pollEmailIntake() function exported from email-intake.ts
  • Uses Gmail API via service account (/home/jdm/mdj-agent/sa-key.json) with domain-wide delegation
  • Polls intake@retireprotected.com for unread messages (label: INBOX, is:unread)
  • For each email: extracts subject + first 500 chars of body text (strips HTML)
  • Runs extracted text through dispatchItem(text, 'email')
  • Creates ticket via POST /api/tracker with correct prefix, title from subject, description from body
  • Marks email as read after successful processing (adds PROCESSED label)
  • Scheduler calls pollEmailIntake() every 5 minutes
  • Status endpoint GET /api/dispatch/email-status returns last poll time, emails processed count, errors
  • Error handling: failed emails logged but don't block the queue. Retry on next poll cycle.
  • Gmail OAuth token verified on MDJ_SERVER before deployment

Architecture Decision

Why poll (not push)? Gmail push notifications require a verified Pub/Sub topic and webhook endpoint. Polling every 5 minutes is simpler, requires only the existing SA key, and handles the expected volume (< 50 emails/day). If volume grows past 200/day, switch to Pub/Sub push. This is v1 — keep it simple.

RON-HD06 Junk/Spam Filter + Auto-Archive

Harden the dispatcher against junk input. The archive target from RON-HD01 handles obvious spam keywords, but this ticket adds pattern-based spam detection, repeat-submission throttling, and an auto-archive pipeline that logs discarded items for audit without creating tickets.

File Paths

services/api/src/raiden/spam-filter.ts — CREATE: pattern matching + throttle logic services/api/src/raiden/hub-dispatcher.ts — EDIT: integrate spam filter before keyword scoring services/api/src/routes/dispatch.ts — EDIT: add GET /api/dispatch/archive endpoint Firestore: dispatch_archive collection — CREATE: archived items for audit trail

Acceptance Criteria

  • isSpam(text: string, metadata?: SpamMetadata): SpamResult exported from spam-filter.ts
  • SpamResult: { is_spam: boolean, reason: string, confidence: number }
  • Pattern detection: known spam phrases, excessive URLs (> 3), all-caps body, empty/whitespace-only
  • Throttle: same sender submitting > 5 items in 10 minutes triggers auto-archive for subsequent items
  • Hub dispatcher calls isSpam() BEFORE keyword scoring — spam items never reach CXO routing
  • Archived items written to dispatch_archive Firestore collection with: original text, channel, timestamp, spam reason
  • GET /api/dispatch/archive returns last 50 archived items for audit
  • No legitimate items are false-positive archived — threshold tuned conservatively
  • npm run build passes

  • 6th submission from same sender in 10 min → archived (throttle)
  • "Import commission data from Gradient" → NOT archived (legitimate)
  • Archive endpoint returns discarded items with reasons
