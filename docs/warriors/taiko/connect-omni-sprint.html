The Unlock

What SHINOB1 greenlit: TAIKO completed its Discovery Doc. The warrior owns the pipes. This sprint builds the channels those pipes feed. Two parallel tracks: Track A fixes the comms plumbing (A2P unblock, Twilio refactor, SendGrid audit) and Track B connects the internal layer (Google Chat bridge, Meet transcripts, RPI Connect rebuild). Both tracks run simultaneously. RONIN picks them up and ships.

What was shipped in the Scope Preview

The CONNECT + OMNI Scope Preview established the 14-ticket architecture, assigned tracks, and confirmed that two parallel build tracks are safe to run independently. That was the plan. This doc is the contract RONIN builds from.

Why two tracks, not one

Track A (TKO-COMMS-*) is entirely in services/api/src/routes/comms.ts, webhooks.ts, and the ProDash opt-in page. It touches client-facing channels. Track B (TKO-CONN-*) is entirely in connect.ts, the Google Workspace credential layer, and the ConnectPanel UI. The two tracks share no files. They can be built by independent RONIN sessions in parallel and merged separately.

Track A — TKO-COMMS-001 through 006

The fix that unlocks everything SMS. Twilio A2P campaign is blocked. No outbound texts go to clients. Campaign SMS, appointment reminders, VOLTRON follow-ups — all of it is sitting behind one carrier compliance gate. Six tickets unblock it and harden the surrounding plumbing.

Track B — TKO-CONN-001 through 008

The upgrade that makes Google Chat + RPI Connect the team's home. The platform can talk to clients (Twilio). It can't yet talk to itself (Google Chat) or to its own team (RPI Connect real-time messaging). Eight tickets close that gap and build the foundation for the Slack migration.

What this sprint does NOT include

  • Alerts infrastructure — tracked separately. See Out of Scope below. Was originally COMMS-009 in the discovery doc. Moved to its own sprint because it depends on both tracks being stable and introduces routing logic that deserves dedicated attention.
  • Slack migration execution — the runbook is built in TKO-CONN-007. The actual migration is a JDM approval gate and team event, not a RONIN deliverable.
  • Campaign engine (C3) — lives in campaign-send.ts, not touched here.
  • TCPA compliance collection — the tcpa_consents schema is specified in TKO-COMMS-001 (consent write on form submit), but the full compliance sprint (recording retention, email archive audit) is a subsequent sprint.

Out of Scope — Future Sprint: Alerts Infrastructure. The central alert dispatcher (POST /api/alerts/dispatch, routing rules, channel selection, wire-failure hooks) is a fully independent system that deserves its own sprint. It is NOT in TKO-COMMS or TKO-CONN. When Track A and Track B are stable and delivering, TAIKO files that sprint. RONIN does not build it here.

Execution Approach

Hard constraint throughout this sprint: Inbound client calls to +18886208587 must never go dark. Every voice route change is staged behind a feature flag or deployed incrementally. The 888 route is touch-last, test-first.

MUSASHI

PR #371 must land before TKO-COMMS-001 starts. VOLTRON

VOLTRON calls comms_send_sms, comms_initiate_call, comms_send_email via the rpi-comms MCP. The MCP wraps these API routes. When TAIKO refactors the underlying routes, the MCP surface stays identical — VOLTRON sees no change. If the A2P gate blocks a send, VOLTRON gets a structured error with { success: false, error: 'A2P campaign not yet approved', code: 'A2P_BLOCKED' } and can relay that to the rep. VOLTRON decides when to call the channel; TAIKO decides whether the channel can accept the call.

RAIDEN

RAIDEN watches #raiden-reactive for comms failures. When the team reports texts not sending, RAIDEN triages and routes to TAIKO's sprint queue. After TKO-CONN-002 ships, RAIDEN also gains the ability to post structured alerts to Google Chat spaces — the same POST /api/connect/chat/post endpoint TAIKO builds. RAIDEN benefits from this sprint without touching any of its files.

SHINOB1

Phase 2 — Twilio Core Refactor (Build while A2P is in Twilio's 48–72 hr queue)
TKO-COMMS-003 Inbound SMS webhook — client matching, structured logging, agent notification
Phase 2 2–3 hrs Track A

Owner: RONIN  •  Gate: Unit test — mock Twilio POST, assert Firestore write shape

Dependency: None. Can start Day 1 in parallel with COMMS-001.

What: Refactor the Twilio SMS webhook handler at POST /api/webhooks/twilio/sms in webhooks.ts. Current handler receives the SMS payload but does nothing structured with it.

Implementation:

  • Match req.body.From (E.164 phone number) against Firestore clients collection querying phone_primary and phone_secondary. If no match, log as client_id: 'unknown' — do not drop the message.
  • Write to communications collection: { type: 'sms', direction: 'inbound', channel: 'sms', client_id, phone_from, phone_to, body, twilio_sid: req.body.MessageSid, timestamp: serverTimestamp(), status: 'received' }.
  • If client was matched, call createNotification() for the assigned agent: { type: 'inbound_sms', client_id, message: 'New text from [client name]', target_user: assigned_agent_uid }.
  • Return valid TwiML response: <Response/> (empty, no auto-reply).

Files: services/api/src/routes/webhooks.ts (Twilio SMS section), packages/core/src/types/comms.ts (add CommunicationLog type if not present)

Acceptance criteria: (1) Inbound SMS from a known client phone creates a communications doc with correct shape. (2) Inbound SMS from unknown number creates a doc with client_id: 'unknown'. (3) Twilio receives a valid 200 TwiML response. (4) Assigned agent receives a notification. Unit test covers all three paths.

TKO-COMMS-004 Outbound SMS — A2P gate, rate limiting, template variables, SendGrid Inbound Parse webhook URL

What: Refactor POST /api/comms/sms in comms.ts.

Template variables: Accept body string with {{first_name}}, {{agent_name}}, {{appointment_date}} placeholders. Resolve against client record and authenticated user before sending.

Log on send: Write to communications with { direction: 'outbound', twilio_sid: result.sid, status: 'sent', ... }.

Risks & Open Questions

[HIGH] Twilio A2P approval timing is out of TAIKO's control.

Twilio typically takes 48–72 hours to review a CTA Verification submission. But "typically" is not a guarantee. If the campaign is rejected again (because the description is still wrong, or because there's a carrier-level brand trust issue), the remediation path is longer. This is the single highest-stakes unknown in Track A. Mitigation: submit with the exact description specified in TKO-COMMS-002, not a variation. If rejected, Twilio will give a reason code. TAIKO escalates to JDM same day. Do not let a Twilio rejection sit in a queue.

[HIGH] OAuth re-consent for team members.

Adding Chat scopes to domain-wide delegation (DWD) does not force individual re-auth. However, if any team member has granted consent to the toMachina OAuth app outside the DWD path (e.g., through a personal sign-in that authorized the app separately), they will see a re-consent screen. SHINOB1 must confirm the DWD is the only auth path in use before CONN-001 executes. If personal OAuth grants exist, the team will see unexpected re-consent prompts. TAIKO communicates proactively: "You may be asked to re-authorize the portal. This is expected and safe. Click Allow." Done before the deploy, not after.

[MEDIUM] Slack Connect deprecation timing.

Slack's policy around Connect has changed for free and lower-tier plans. If Slack deprecates features that RPI is currently using (particularly Connect channels with external partners), the migration timeline accelerates against our will. TAIKO monitors Slack's release notes monthly. If a deprecation notice arrives, TAIKO files a P1 sprint to move the affected channels before the deadline. The runbook (CONN-007) should explicitly cover which channels are Connect-dependent.

[MEDIUM] TCPA tail risk on historical outbound SMS.

Before the A2P campaign was blocked, some outbound SMS may have been sent to recipients who did not go through the compliant opt-in flow. If any of those recipients filed a TCPA complaint, that is a historical liability that exists regardless of what this sprint builds. TKO-COMMS-001 and the tcpa_consents collection establish compliant consent going forward. It does not retroactively document past sends. JDM and the legal/compliance layer need to assess whether there is historical exposure. This is a business decision, not a RONIN decision. TAIKO flags it, does not resolve it.

[MEDIUM] RPI Connect SSE connection stability under load.

Server-Sent Events over Cloud Run have a known concern: Cloud Run has a request timeout (default 60 minutes, configurable). An SSE connection that is open longer than the timeout will be dropped. The ConnectPanel must handle reconnection gracefully: when the stream drops, silently attempt to reconnect with exponential backoff, refetch the last 20 messages to fill any gap, then resume the stream. RONIN implements this reconnection pattern in TKO-CONN-004. If not handled, users will see the message feed go stale after an hour with no visible error.

[LOW] Google Meet recording API availability.

The Google Meet REST API for recording retrieval requires the meeting to be created through the Workspace Calendar API with recording enabled. Ad-hoc Meet sessions created directly from the Google Meet UI may not be accessible via the API. TKO-CONN-003 scopes to meetings created through the platform (POST /api/connect/meet). Meetings started directly in Google Meet are out of scope for this sprint. TAIKO notes this as a future sprint item.

[LOW] ConnectPanel.tsx size.

ConnectPanel is already 824 lines. Adding real-time messaging, DMs, Meet launch, and a Spaces section will push it past 1,200 lines. RONIN should extract the message feed into a separate component (ConnectMessageFeed.tsx) and the DM thread into ConnectDMThread.tsx to keep the panel file manageable. This is a code quality call RONIN makes during TKO-CONN-004 — do not wait for a follow-up ticket.
