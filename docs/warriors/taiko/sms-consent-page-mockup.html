Carrier Compliance Checklist
Twilio A2P · TCPA · CTIA · CAN-SPAM
✓
TCPA — Telephone Consumer Protection Act
Express written consent obtained before sending any automated SMS. Opt-in record with timestamp, IP, user-agent, and checkbox state retained for 5 years minimum. Consent is not a condition of purchase (stated explicitly in MUSASHI Part 1).
✓
CTIA — Carrier Industry Standards
Campaign registered as "Customer Care" use case in Twilio A2P. Opt-in: web form checkbox. Opt-out keywords: STOP/UNSUBSCRIBE/CANCEL/END/QUIT. Help: HELP/INFO. Re-subscribe: START. All keywords wired via Twilio Messaging Service (TAIKO pipe).
✓
CAN-SPAM Compliance
Business identity clear on page and in every message. Physical business address in footer. Unsubscribe mechanism (STOP) honored immediately. No deceptive sender identification. Privacy Policy linked prominently.
✓
Unchecked Checkbox — Default State
Checkbox renders unchecked on every page load — verified in this mockup. RONIN must enforce via React state: useState(false), no defaultChecked, no hydration carry-over, no localStorage restore of consent state.
✓
Visible STOP Instructions
STOP appears in three distinct locations: (1) Part 1 disclosure body, (2) checkbox label, (3) footer disclosure. Footer is always-visible regardless of form scroll position. Cannot be hidden or collapsed.
✓
Business Identification
"Retirement Protectors, Inc." appears in: page header, disclosure body, checkbox label, confirmation SMS, footer brand, footer address. Phone 515-992-5000 in header and footer. City/state in footer.
✓
Rate Disclosure — "Msg & data rates may apply"
Exact phrase appears in: Part 1 disclosure, Part 3 confirmation SMS, footer. Three placements. Exact phrase required — no paraphrase. MUSASHI has used the exact phrase in all three locations.
✓
Opt-In Record Retention
RONIN writes to sms_consent_log: timestamp (ISO 8601), phone (E.164), IP address, user-agent, copy_version hash. 5-year minimum retention per TCPA. Phone field masked in any logging after the Firestore write.

Firestore — sms_consent_log Schema

Collection: sms_consent_log — write on every successful form submit. Document ID: auto-generated. Write before triggering Twilio — never in reverse order.

// sms_consent_log/{autoId} { "phone_e164": "+15155550100", // E.164 format, required "consented_at": Timestamp, // Firestore server timestamp "ip_address": "203.0.113.42", // From X-Forwarded-For header "user_agent": "Mozilla/5.0 ...", // Full UA string "copy_version": "musashi-sms-v1.0-2026-04-13", // From MUSASHI copy doc header "disclosure_hash": "sha256:abc123...", // SHA-256 of Part 1 text at render time "source_url": "https://prodash.tomachina.com/sms-consent", "source_channel": "web-form", // web-form | intake-form | agent-documented "confirmation_sms_sent": false, // Updated to true after Twilio Part 3 send "client_id": null, // Linked after client match (may be null at opt-in) "created_by": "system:sms-consent-page" }
⚑
PHI Note: phone_e164 is PII. Never log it via console.log or in error messages. The Firestore write is the only storage point. Firestore Rules must restrict sms_consent_log reads to @retireprotected.com and writes to service account only.
Form POST Flow — Route Handler Steps
StepAction
1. Validate Confirm phone E.164 format. Confirm checkbox state is true in POST body. Return 400 if either fails.
2. Write Firestore Write to sms_consent_log with server timestamp. Return 500 if write fails. Never proceed to Twilio if write fails.
3. Trigger Twilio POST to Twilio Messages API: send Part 3 confirmation SMS (155 chars, MUSASHI verbatim) to consented number. Update confirmation_sms_sent: true on the consent doc after successful send.
4. Return success { success: true, consent_id: docId } — frontend renders confirmation state (Section 2).
5. Twilio error handling If Twilio send fails: return { success: true } anyway (consent is written — that is the legal record). Log Twilio error server-side only. Retry via queue.
