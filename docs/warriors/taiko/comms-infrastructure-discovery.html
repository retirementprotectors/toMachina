Why This Warrior Exists

The one-sentence case

Every channel that the team uses to reach clients and each other is either broken, abandoned, or owned by nobody — and because it's owned by nobody, nobody fixes it.

JDM's words, verbatim (April 13, 2026): “Our external communications through our comms app between text and calling and email and internal text and calling and email is complete fucking joke as well. No one even uses them because they're so fucking bad and it's constantly broken. We need to totally migrate away from Slack and have everything through Google Meet [and Google Chat] but our internal communications modules [are] such a piece of shit right now we can't do that.”

The blast radius today

Outbound SMS: Dead

Twilio local number A2P 10DLC campaign is blocked. Ticket 25630009 was auto-closed without resolution. CTA Verification failed because there is no compliant opt-in landing page. Zero outbound texts go out to clients from local numbers. Only the 888 toll-free bypasses this — and it was never configured to carry the load.

Inbound/Outbound Voice: Shitty at Best

The 888 toll-free number works and routes correctly. The local Twilio number is A2P-blocked which affects call delivery on some carriers. Call recording exists in code (comms.ts) but transcription is configured per voicemail only. Click-to-call routes to the correct agent, but the experience is inconsistent enough that reps avoid it.

Comms App: Team Won't Use It

The in-portal module for sending texts, emails, and making calls to clients is so broken and so slow that the team routes around it. They use their personal phones. They use personal email. The Machine has no record of those communications. Client data is leaking out of the system in every conversation that happens outside the portal.

Internal Comms: Two Broken Systems

Slack is the de facto tool but JDM wants to migrate to Google Chat + Meet. RPI Connect (the in-portal team comms module) exists as code but is effectively a mockup — channels seed on first GET, calendar pulls from Google API, but the actual messaging layer is hollow. No one migrates to a migration destination that doesn't work yet.

Alerts: Wired to Nothing

The notifications collection exists. The createNotification() helper is called by comms routes. But there is no routing logic, no channel selection (SMS vs email vs in-portal toast), no user targeting beyond a target_user field. System events — wire failures, deploy errors, security flags — post nowhere.

SendGrid: Unknown State

Transactional email goes through SendGrid (noreply@retireprotected.com). The webhook handler exists. But delivery rates, bounce rates, and template inventory have never been formally audited. We don't actually know how much email is getting through and what's silently failing.

Why "no owner" is the root cause

Every one of these failure modes has the same root cause: no single warrior owns the pipes. VOLTRON uses the channels. RAIDEN reacts when they break. RONIN patches individual routes. MUSASHI cares about the content in those channels. But nobody wakes up every morning thinking "are the pipes working?" Nobody owns the A2P campaign. Nobody set up the opt-in page. Nobody monitors bounce rates. This warrior exists to end that.

The framing: These are the pipes that every team member and every client depends on. If the pipes don't work, the business doesn't run. This warrior owns the pipes — not the water that flows through them, not the creative that goes on envelopes, just the pipes. They have to work. Always. Every one of them.

What's explicitly NOT this warrior's problem

  • Campaign content, press releases, brand voice → MUSASHI (CMO)
  • Campaign engine (C3, 60 campaigns, 661 templates) → already lives in services/api/src/routes/campaign-send.ts, not this sprint
  • Creative strategy, Discovery Docs, narrative → MUSASHI
  • Warrior dispatcher routing (Slack→tmux) → SHINOB1 + DISP-MIRROR sprint
  • VOLTRON's client-facing conversation logic → VOLTRON/CSO domain

The boundary is simple: TAIKO owns the wire. What travels over the wire belongs to someone else.

How We Rebuild It

Rebuild mandate, not incremental fix. The current comms infrastructure is not in bad shape because of one bug or one missing feature. It is in bad shape because it was assembled incrementally, never owned, and never tested end-to-end as a system. Patching individual pieces will produce the same result in six months. The only real fix is: tear out the broken parts, understand the contracts between them, and rebuild correctly from the ground up. While honoring the one hard constraint: inbound client calls to the 888 number must never go dark.

The hard constraint during rebuild

Clients call +18886208587. That must work at all times, no exceptions, throughout the entire rebuild. Everything else can have a brief maintenance window if needed. The 888 cannot.

Rebuild sequence — priority order

1
A2P Unblock (COMMS-001, 002) — Days 1–2

Build the compliant opt-in landing page with exact carrier-approved language. Reopen Twilio ticket 25630009. Submit to Twilio for CTA Verification. This is the single highest-value unlock: the moment it clears, outbound SMS goes live across the platform. Everything downstream of SMS (VOLTRON client texts, campaign SMS sends, appointment confirmations) is unblocked by this one action.

2
Twilio core refactor (COMMS-003 through 006) — Days 2–5

While A2P verification is in Twilio's queue (48–72 hours typical), rebuild the underlying SMS and voice routes. Inbound SMS webhook needs to match caller to client, log to communications, and surface in the portal. Outbound SMS needs rate limiting, A2P gate, and template engine. Voice inbound and outbound need to be solid, consistent, and logged. This is the plumbing under everything else.

3
Comms App UI rebuild (COMMS-008) — Days 3–6

The in-portal module that gives reps a single place to text, call, and email clients. Rebuilt from scratch: fast load, real-time message thread view, click-to-call, inline email compose. If the underlying Twilio + SendGrid infra is solid (steps 1 and 2), the UI is just the surface. This is the thing that makes the team actually use it.

4
Alerts infrastructure (COMMS-009) — Days 4–6 parallel

Central alert dispatcher: a new alerts.ts route and service that receives structured events and routes them to the right channel (SMS to rep's phone, email, in-portal notification toast, or Google Chat post) based on routing rules. Wire failures, deploy failures, security events, case assignment — all plugged in here. Stops the silent-failure problem.

5
Google Meet + Chat integration (COMMS-010, 011) — Days 5–7

Google Meet API is partially wired (connect.ts creates meetings). Extend to retrieve recordings, sync transcripts to client records. Google Chat bridge: the platform can post structured events to Chat spaces from any wire or alert. These two together are the foundation for the Slack migration.

6
RPI Connect rebuild (COMMS-012) — Days 6–9

The in-portal internal team comms module. Current state: channels exist in Firestore, calendar API works, messaging layer is hollow. Rebuild with real-time messaging (WebSocket or SSE), presence indicators, Google Meet launch button. This is the replacement for Slack within the portal. Must be solid before anyone migrates.

7
SendGrid audit + compliance + migration runbook (COMMS-007, 013, 014, 015) — Days 7–12

Audit SendGrid delivery rates and bounce handling. Build TCPA consent record collection. Produce the Slack-to-Google migration runbook — which channels move first, timeline, team communication plan, data export. Observability dashboard: call volume, SMS delivery, email bounce, Meet usage — all in one place. These are the long-tail items that make the system trustworthy, not just functional.

Integration contracts with other warriors

VOLTRON

VOLTRON calls comms infra via CCSDK tools (comms_send_sms, comms_initiate_call, comms_send_email). The rpi-comms MCP wraps these API routes. TAIKO owns the routes. VOLTRON decides when to call them. If channel-of-choice logic changes (e.g., "prefer SMS over email for this client"), that logic lives in VOLTRON — TAIKO just needs the pipes to accept any channel call and execute correctly.

RAIDEN

RAIDEN watches for comms channel failures in #raiden-reactive. When a rep reports "texts not sending," RAIDEN triages, attempts a quick fix (<30 min), or routes to TAIKO's sprint queue with full context. TAIKO does not watch Slack — RAIDEN feeds the queue. RAIDEN also needs the alerts infrastructure to be fully wired so that Twilio delivery failures and SendGrid bounce spikes surface automatically rather than via team complaint.

Risks & Open Questions

Risks — ranked by severity

CRITICAL: TCPA exposure if A2P rebuild has gaps in consent records

TCPA violations run $500–$1,500 per message for negligent violations, $500–$1,500 per message trebled for willful violations. If RPI texts a client without a valid consent record, and the client disputes it, the absence of a timestamped opt-in record is indefensible. The tcpa_consents collection (COMMS-014) and the /sms-consent page (COMMS-001) are not optional polish — they are the legal foundation that must exist before outbound SMS goes live at scale. MUSASHI must review the consent copy before RONIN builds the page.

HIGH: Twilio A2P approval delay — could be days or weeks

Twilio's CTA Verification process is typically 48–72 hours for compliant campaigns, but can take longer if Twilio has questions about the use case or if the campaign application triggers manual review. Ticket 25630009 was auto-closed, which may mean it's flagged in Twilio's system. Plan: build all other Twilio routes during the approval window. Do not block the sprint on Twilio's timeline. If approval takes more than 5 business days, escalate to Twilio account management.

MEDIUM: 888 inbound call disruption during Twilio route refactor

The 888 number is the one working piece of the voice stack. Any changes to comms.ts TwiML generation or the voicemail config carry risk of breaking inbound call routing. Mitigation: test all voice TwiML changes in Twilio's sandbox with a test number before touching production. Deploy voice changes during off-hours (after 6pm CST). Keep rollback ready.

MEDIUM: Email deliverability disruption during SendGrid changes

If the SendGrid audit reveals the sending domain or authentication (SPF/DKIM) has issues, fixing them may cause a temporary deliverability drop during DNS propagation. Mitigation: run the audit first (read-only), document the current state, then make any authentication changes with a 48-hour propagation window built in. Never make SendGrid auth changes on a Monday.

MEDIUM: Slack-to-Google migration is a multi-week culture change

The migration runbook can be technically perfect and the team will still resist if: (1) RPI Connect is not fast and reliable, (2) Google Chat doesn't have feature parity with the Slack channels they use, or (3) JDM doesn't personally champion the transition. This warrior can build the infrastructure. The adoption is a human decision that belongs to JDM and the division leads. Gate G4 requires JDM approval of the runbook before any migration action — this is intentional.

MEDIUM: Twilio billing during the rebuild (test traffic)

Integration testing for Twilio voice and SMS generates real API calls and real billing. Twilio's test credentials can be used for unit tests but not for end-to-end portal testing. Set a billing alert in the Twilio console at $50/month during the rebuild. Estimate: ~100–200 test SMS messages + 20–30 test calls = approximately $10–15 in test charges. Acceptable but should be monitored.

LOW: VOLTRON CCSDK channel-of-choice logic may need updating

VOLTRON currently calls comms tools via the rpi-comms MCP. If this sprint changes the API contract for any of those routes (field names, response shape, error codes), the MCP tool definitions will need corresponding updates. Mitigation: maintain strict backward compatibility on existing route signatures; only add fields, never remove them. If a breaking change is unavoidable, coordinate with VOLTRON maintainers before deploy.
