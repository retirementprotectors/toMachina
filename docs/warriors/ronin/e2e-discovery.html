Manual Verification Burden

470 audited items currently require manual CEO visual confirmation ("Confirm Walkthrough") after each deploy. This process is:

  • Time-intensive (requires navigating each module, verifying renders)
  • Error-prone (human eyes miss subtle regressions)
  • Blocking (CEO time is the scarcest resource)
  • Unscalable (grows linearly with shipped features)

Gaps Identified

# Gap Impact Risk Level
G1 No regression detection Breaking changes only caught by type-check and build. Runtime behavior regressions are invisible to CI. CRITICAL
G2 No pipeline verification Intake wires (8-stage document processing) are completely untested. A broken wire means client documents silently fail to process. CRITICAL
G3 No UI verification 470+ items need manual visual check on every release. CEO time spent on verification instead of strategy. HIGH
G4 No CI test gate Deploys go to production with only type-check + build. Functionally broken code can ship if it compiles. CRITICAL
G5 No test data isolation No emulator, no test fixtures, no cleanup strategy. Cannot run tests without polluting production data. HIGH
G6 No visual regression UI changes are invisible to CI. Broken layouts, missing components, and styling regressions ship undetected. HIGH

Layer 1: Backend Pipeline Tests (Vitest)

Purpose

Verify the 4 intake pipelines (TRK-497–500) actually process documents end-to-end through the full 8-stage wire.

The 8-Stage Wire

PREPARE → CLASSIFY → EXTRACT → VALIDATE → NORMALIZE → MATCH → WRITE → ACF_FINALIZE

4 Pipeline Tests

Test Plan Ticket Code Ticket Description
ACF_UPLOAD TRK-497 TRK-539 Upload PDF to ACF folder, verify wire produces classified document with extracted data
MAIL TRK-498 TRK-540 Simulate inbound mail document, verify wire classifies and routes to correct client
SPC_INTAKE TRK-499 TRK-541 Simulate specialist intake submission, verify wire processes and writes to Firestore
ACF_SCAN TRK-500 TRK-542 Simulate batch scan trigger on ACF folder, verify all documents process through wire

Execution Model

  • Real Firestore + Real Drive — no emulator. Drive API cannot be emulated; these are TRUE end-to-end tests.
  • Firebase Admin custom token auth — test user gets a custom token via Admin SDK, exchanged for ID token.
  • Direct wire execution — tests call POST /api/intake/execute-wire directly, bypassing the Cloud Function trigger (trigger is trivial plumbing).
  • Sequential execution — tests share real Firestore/Drive state and must not race.
  • Cleanup after each test — batch delete test data from Firestore + Drive to leave no trace.
  • CI timing — runs AFTER deploy-api on push to main (tests the live production API).

Layer 2: UI Visual Verification (Playwright)

Purpose

Navigate to each major ProDash module, verify key elements render correctly, take screenshots. Replaces manual Confirm Walkthrough for the majority of shipped items.

Execution Model

  • @playwright/test framework — same tool we use via MCP, but as a proper test suite with assertions and reporting.
  • Runs against localhost dev server — CI starts npm run dev, waits for ready, then runs Playwright.
  • HTML test report with screenshot gallery — every test captures screenshots, failures include visual diff.
  • CI timing — runs BEFORE deploy (catches UI regressions before they reach production).
  • Manual walkthrough reduction — 470 manual items reduced to ~50 that require human judgment (subjective UX, new features not yet covered).

Key Design Decisions

# Decision Rationale Alternative Rejected
D1 Vitest over Jest TypeScript-native with zero config. Fast execution via esbuild. First-class monorepo support with workspace configs. No ts-jest transform layer needed. Jest + ts-jest
D2 Real Firestore, not emulator Drive API cannot be emulated. The purpose is TRUE end-to-end verification. Emulator would give false confidence since production integration is what actually breaks. Firebase Emulator Suite
D3 Direct wire execution Tests call POST /api/intake/execute-wire directly, bypassing the Cloud Function trigger. The trigger is trivial plumbing (reads event, calls API). The wire is where complexity and breakage live. Trigger-based (upload and wait)
D4 Sequential execution Tests share real Firestore/Drive state. Parallel execution would cause race conditions on shared resources. Pipeline tests are inherently stateful and sequential. Parallel test execution
D5 @playwright/test for UI Same tool already used via MCP plugin, but wrapped in a proper test framework with assertions, reporters, and CI-friendly output. Team already has muscle memory with Playwright. Cypress, Puppeteer
D6 Two separate CI jobs e2e-intake runs AFTER deploy (needs live API). e2e-ui runs BEFORE deploy (needs only localhost). Different timing, different dependencies, different failure modes. Single combined job
