Strategic Context — The Trinity

The Trinity Data Method™ is RPI's proprietary data reconciliation framework. Three data sources, triangulated to achieve 100% Data Confidence:

SourceRoleWhere It Lives
Carrier Data Source of Truth — direct from insurance company portals WIRE_DATA_IMPORT, WIRE_COMMISSION_SYNC
Digital Files Supporting Documentation — PDFs, statements, scans, correspondence ACF_CLEANUP_WIRE (this document)
CRM Data Actionable Intelligence — ProDash + partner agency CRM exports WIRE_DATA_IMPORT (client enrichment path)
Why this wire matters: Carrier data tells you what exists. CRM data tells you who it belongs to. But the documents — the actual statements, applications, tax forms, illustrations — are the evidence. Without organized digital files, the other two sources are claims without proof. The ACF Wire turns a chaotic Google Drive into a structured, queryable, self-indexing document system.

This discovery doc covers the Digital Files pillar: how every document that enters RPI's ecosystem gets organized, named, deduplicated, and linked to its client. The other two Trinity discovery docs cover Carrier Data (WIRE_DATA_IMPORT) and CRM Data (WIRE_CLIENT_ENRICHMENT).

Before the ACF Wire, client documents lived in a flat folder structure with inconsistent naming, duplicate files, orphaned folders, and no connection to Firestore client records. The March 21 incident — where 1,060 ACF folders were accidentally batch-deleted during a Drive reorg — exposed the fragility. All folders were restored, but the incident proved that operating without guardrails on the Digital Files pillar was a ticking bomb.

The ACF 2.0 Sprint (23 tickets, shipped and audited) cleaned up the existing mess. This wire definition ensures it stays clean — every new document that enters the system runs through the same pipeline automatically.

ACF Folder Architecture

Key insight: The 5 subfolders are a lifecycle, not categories. A client relationship flows through these stages. Documents land where they belong in the client's journey.

Folder Naming Convention

ACF - Last, First
Example: ACF - Millang, Josh
Example: ACF - Rosentreter, Gary

Rules: Title case. No ALL CAPS. No trailing commas. No ACF- (missing space). No double spaces. The ACF_RENAME atomic tool enforces all of these.

Document Naming Convention

The naming convention IS the index. No metadata database needed. No search tokens. The file name tells you exactly what it is, when it's from, who issued it, and which policy it belongs to. If you can read a filename, you can find any document.
{YYYY-MM}_{DocType}_{Carrier}_{PolicyNum}.{ext}
SegmentFormatSourceExample
YYYY-MM 4-digit year, 2-digit month Document date (statement period, application date, etc.) 2026-03
DocType PascalCase from taxonomy document_taxonomy Firestore collection Statement, Application, Illustration
Carrier Normalized carrier short name carriers Firestore collection Athene, NorthAmerican, Nationwide
PolicyNum Raw policy/contract number Extracted from document or account record GA12345678, N1234567
ext Original file extension Preserved from source file .pdf, .png, .jpg

When a segment is unknown (no carrier detected, no policy number extracted), use UNKNOWN as placeholder. The ACF_RENAME_FILES tool flags these as exceptions for the ACF_AUDIT tool to surface for human review.

The Wire: ACF_CLEANUP_WIRE

Three super tools, executed in sequence, with ACF_SNAPSHOT guardrails bracketing each one. This is the ATLAS-registered wire that turns a messy Drive folder into a clean, linked, deduplicated, properly-named Active Client File.

ACF_SNAPSHOT (before) | SUPER TOOL 1: FOLDER_CLEANUP | ACF_RENAME — fix parent folder naming | ACF_MERGE — merge duplicate folders | ACF_SUBFOLDER — create/verify 5 lifecycle subfolders | ACF_ROUTE_FILES — move loose root files into correct subfolder | ACF_LINK — link Drive folder to Firestore client doc | ACF_SNAPSHOT (after) | SUPER TOOL 2: DOCUMENT_CLEANUP | ACF_FLATTEN — shred nested sub-sub-folders | ACF_DEDUPE_FILES — same doc in multiple places, keep one | ACF_RENAME_FILES — apply {YYYY-MM}_{DocType}_{Carrier}_{PolicyNum}.ext | ACF_RECLASSIFY — file in wrong subfolder, move to correct one | ACF_SNAPSHOT (after) | SUPER TOOL 3: AUDIT_REVIEW | ACF_AUDIT — generate exceptions report | → Routes to human review UI (dedup page) | ACF_SNAPSHOT (final)

Super Tool 1: FOLDER_CLEANUP

FOLDER_CLEANUP SUPER_FOLDER_CLEANUP

Structural integrity of the ACF folder tree. Ensures every client has one properly-named folder with 5 lifecycle subfolders, linked to their Firestore record.

  • ACF_SNAPSHOT Capture folder inventory to Firestore acf_snapshots — before/after guardrail Source: acf-guardrails.ts → snapshot
  • ACF_RENAME Fix parent folder naming convention (Title Case, spacing, ACF prefix) ACF- → ACF - | ALL CAPS → Title Case | trailing comma → removed | double spaces → single Source: acf-cleanup.ts (rename phase) + acf-fix-stragglers.ts
  • ACF_MERGE Detect & merge duplicate folders for same client (keep richer, move files from smaller) Source: acf-cleanup.ts (merge phase) + acf-fix-stragglers.ts (specific merges)
  • ACF_SUBFOLDER Create/verify 5 lifecycle subfolders: Client, NewBiz, Cases, Account, Reactive Source: acf-subfolder-audit.ts
  • ACF_ROUTE_FILES Move loose files from folder root into correct lifecycle subfolder Source: acf-cleanup.ts (file routing phase)
  • ACF_LINK Write acf_folder_id + acf_folder_url to Firestore client doc. Bidirectional link. Source: acf-cleanup.ts (link phase)

Super Tool 2: DOCUMENT_CLEANUP

DOCUMENT_CLEANUP SUPER_DOCUMENT_CLEANUP

File-level hygiene within each ACF. Flatten nesting, eliminate duplicates, apply the naming convention that makes every filename a queryable index entry, and ensure files are in the right lifecycle subfolder.

  • ACF_FLATTEN Shred nested sub-sub-folders — bring all files up to one of the 5 root subfolders Drive nesting beyond 1 level is chaos. Max depth = ACF / Subfolder / files. Nothing deeper. Source: new (extract from acf-cleanup.ts recursive listing)
  • ACF_DEDUPE_FILES Same document in multiple subfolders — keep one copy in the correct location Match by content hash (MD5 from Drive API) + filename similarity. Keep the copy in the correct subfolder. Delete the duplicate. Source: acf-account-dedup.ts (account-level dedup pattern) + acf-client-dedup.ts (merge pattern)
  • ACF_RENAME_FILES Apply {YYYY-MM}_{DocType}_{Carrier}_{PolicyNum}.ext naming convention Uses SUPER_EXTRACT (Vision mode) output + document_taxonomy + carrier normalization. Files with UNKNOWN segments flagged for ACF_AUDIT. Source: acf-extract-batch.ts (extraction) + new (rename logic)
  • ACF_RECLASSIFY File in wrong subfolder — move to correct lifecycle stage based on document type Uses ACF_FINALIZE's DOCUMENT_TYPE_TO_SUBFOLDER mapping. Statement in Client/ → moves to Account/. Application in Reactive/ → moves to NewBiz/. Source: acf-finalize.ts (routing map) + new (batch move logic)

Super Tool 3: AUDIT_REVIEW

AUDIT_REVIEW SUPER_AUDIT_REVIEW

Everything auto-resolution couldn't handle gets surfaced for human review. This is the last mile — the exceptions that need a human eye.

  • ACF_AUDIT Generate exceptions report: unresolvable duplicates, UNKNOWN naming segments, orphan folders, broken links, missing subfolders Writes to acf_audit_exceptions collection. Each exception has type, severity, client_id, suggested_action. Source: acf-verify-final.ts + acf-orphan-resolve.ts (orphan detection)
  • HUMAN_REVIEW Exceptions route to the ACF audit page in ProDashX. COR/AST resolves. SPC verifies. The dedup review UI already exists. Extend it to handle all ACF exception types.

The ACF_SNAPSHOT Guardrail

Origin story: March 21, 2026. 1,060 ACF folders batch-deleted from Shared Drive during a reorg. No snapshot existed. No guardrail. Folders sat in Trash for 10 days. All restored ($0 cost, 926 linked to Firestore clients), but the incident proved that any batch operation on the Digital Files pillar must be bracketed by snapshots.
OperationWhat It CapturesStorage
snapshot Complete folder inventory: ID, name, created/modified timestamps. Written to Firestore acf_snapshots collection. Firestore (permanent audit trail)
trash-audit Scan ACF parent folder for recently trashed items. Fires Slack notification if count exceeds threshold (default: 5). Slack notification + Firestore log
verify Compare current Drive state against last snapshot. Report drift: new folders, missing folders, renamed folders. Console output + optional Firestore

Every super tool in the wire runs ACF_SNAPSHOT before and after execution. If the after-snapshot shows unexpected deletions (folder count drops by more than the trash-audit threshold), the wire halts and notifies.

Intake Wire Alignment

One naming convention. One routing table. Every channel. Whether a document arrives via scanned mail, email attachment, direct upload, or SPC intake folder — it goes through the same naming and routing logic.

The existing WIRE_INCOMING_CORRESPONDENCE already terminates with ACF_FINALIZE, which routes files to the correct ACF subfolder. The ACF Wire adds the naming convention and dedup layers that ACF_FINALIZE currently doesn't handle.

The Shared Routing Map

ACF_FINALIZE already defines the DOCUMENT_TYPE_TO_SUBFOLDER map (see packages/core/src/atlas/super-tools/acf-finalize.ts). This same map is used by ACF_RECLASSIFY and ACF_ROUTE_FILES. One map, three tools, zero drift.

DOCUMENT_TYPE_TO_SUBFOLDER = {
  // Client subfolder
  id_document     → Client
  voided_check    → Client
  tax_document    → Client
  trust_document  → Client
  poa_hipaa       → Client
  fact_finder     → Client

  // NewBiz subfolder
  application_form  → NewBiz
  transfer_form     → NewBiz
  delivery_receipt  → NewBiz
  replacement_form  → NewBiz
  suitability       → NewBiz

  // Cases subfolder
  illustration  → Cases
  comparison    → Cases
  proposal      → Cases
  analysis      → Cases

  // Account subfolder
  statement      → Account
  confirmation   → Account
  annual_review  → Account
  distribution   → Account

  // Fallback
  (anything else) → Reactive
}

The document_taxonomy Firestore collection is the authoritative source for document types. When a new document type is added to the taxonomy, it gets a pipeline field that maps it to a subfolder. The routing map above is the compiled output.

Risk & Guardrails

RiskMitigation
Batch file deletion (March 21 repeat) ACF_SNAPSHOT before/after every super tool. Halt + Slack notification on unexpected count drop.
Wrong file moved to wrong subfolder ACF_RECLASSIFY uses the same DOCUMENT_TYPE_TO_SUBFOLDER map as ACF_FINALIZE. One source of truth.
Naming convention loses information Original filename preserved in document_index Firestore collection. Rename is non-destructive — metadata survives even if the name is wrong.
Dedup deletes the wrong copy ACF_DEDUPE_FILES uses MD5 hash matching (zero false positives). Keeps the copy in the correct subfolder per routing map. Moves loser to Trash (30-day recovery).
Vision extraction misclassifies Unconfident classifications (< 70%) route to Reactive subfolder + get flagged by ACF_AUDIT for human review.
Drive API rate limits on batch ops All scripts use chunked processing (10-50 ops per batch) with exponential backoff. Proven at 277-folder scale.
