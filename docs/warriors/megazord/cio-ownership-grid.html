Data Migration Protocol

7-Stage Pipeline — No Exceptions

Every field rename, delete, or structural change in Firestore goes through all 7 stages. Two JDM approval gates. Hook-enforced (gate-data-migration).

Stage 1 — INVENTORY

CheckHow
Code referencesgrep -rn "field_name" apps/ packages/ services/ — count every file
TypeScript typesCheck packages/core/src/types/index.ts + atlas types
Firestore indexesCheck Firebase Console → Firestore → Indexes for the field
CollectionGroup queriesgrep for orderBy('field') and where('field' on collectionGroup
API route queriesgrep services/api/src/ for the field in Firestore reads
NormalizersCheck packages/core/src/normalizers/ for field references
Seed scriptsCheck services/api/src/scripts/ for field references

Output: File list + reference count + index list

Stage 2 — IMPACT REPORT  JDM GATE

DeliverableFormat
Blast radius tableEvery file that breaks if the field disappears
Query impactEvery orderBy / where that needs updating
Index impactEvery Firestore index that needs recreation
UI impactEvery page/component that renders this field

JDM reviews and approves before proceeding

Stage 3 — CODE FIRST

ActionDetail
Update all referencesSupport BOTH old and new field names: data.new_field || data.old_field
Update TypeScript typesAdd new field to interface (keep old as optional)
Create new Firestore indexesAny orderBy / where on the new field name needs an index BEFORE migration
DeployPush code, wait for deploy, verify live
VerifyPages load, queries work, no errors in console

Code is live and working with BOTH field names before touching data

Stage 4 — DRY RUN  JDM GATE

ActionDetail
Read-only migrationRun migration logic without writing. Log what WOULD change.
Sample outputShow 10-15 before/after examples
CountsTotal documents affected, fields written, fields deleted

JDM reviews sample + counts and approves before execution

Stage 5 — EXECUTE

ActionDetail
Atomic writesSET new field + DELETE old field in SAME Firestore update (not separate ops)
Batch size450 per batch (Firestore limit is 500)
Progress loggingReport every batch: updated count, errors
Error handlingStop on first error, report which document failed

Stage 6 — VERIFY

CheckHow
Every affected pageOpen each page that renders the field — does it show data?
Firestore queriesRun the orderBy / where queries — do they return results?
Record countsCompare total records before and after — same number?
Data spot checkPull 5 random records, verify field values are correct

Nothing moves to cleanup until production is confirmed working

Stage 7 — CLEANUP

ActionDetail
Remove old field fallbacksChange data.new || data.old to just data.new
Remove old field from typesDelete the optional old field from TypeScript interface
Delete old Firestore indexesRemove indexes on the old field name
Deploy final cleanupPush, verify, done

Incident Record — 2026-04-08

What happened: MEGAZORD migrated carrier_name → carrier in Firestore (11,645 accounts) BEFORE updating 249 code references across 52 files. Skipped stages 1-3.

Impact: Accounts page showed 35 records instead of 12,000+. Firestore index for carrier didn't exist. Three separate production issues from one migration.

Root cause: Data first, code second. Should have been code first, data second.

Gates added: Hookify rule gate-data-migration blocks FieldValue.delete(). This protocol tab. Firestore index check added to Stage 1.
