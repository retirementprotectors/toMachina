Why CONSUME Is Track 1

"You can't register tools into a registry that doesn't exist."

MEGAZORD had ATLAS. VOLTRON had QUE. Both registries existed before their CXO Roadmaps. MUSASHI has nothing. packages/core/src/cmo/ does not exist. The tools are scattered across 4 MCP servers, 1 plugin, and 1 standalone service. The discovery docs are scattered across Shared Drive. The FORGE tickets are seeded but unowned.

CONSUME is the heaviest track because it's building the foundation itself — not maturing an existing one. Every other track depends on what we build here.

MUS-C01 CMO Registry Scaffold + Types

Create the packages/core/src/cmo/ directory structure and define all CMO types following the QUE registry pattern.

File Paths

packages/core/src/cmo/types.ts — NEW packages/core/src/cmo/tools/index.ts — NEW (empty barrel, populated by C03-C07) packages/core/src/cmo/wires/index.ts — NEW (empty barrel, populated by C08)

Acceptance Criteria

  • CmoRegistryEntry interface matches QUE pattern: id, type ('TOOL' | 'SUPER_TOOL' | 'WIRE'), domain: 'cmo', name, description, composedOf?
  • CmoChannel type: 'print' | 'digital' | 'web' | 'social' | 'video'
  • CmoToolDomain type: 'canva' | 'wordpress' | 'veo' | 'c3' | 'pdf' | 'drive' | 'frontend-design'
  • CmoWireStep interface: stepId, toolId, description, input schema ref, output schema ref
  • CmoWireDefinition interface: wireId, name, channel, steps: CmoWireStep[], description
  • All types exported from types.ts
  • Directory structure matches target architecture diagram
  • Empty barrels build-safe: tools/index.ts exports an empty typed array (export const CMO_TOOLS: CmoRegistryEntry[] = []) and wires/index.ts exports an empty typed array (export const CMO_WIRES: CmoWireDefinition[] = []). Both satisfy the type system with zero entries. npm run type-check AND npm run build must pass with empty barrels present — these files are the foundation that every subsequent ticket depends on.

Reference Pattern

Follow packages/core/src/que/registry.ts lines 9–17 for QueRegistryEntry structure. Add channel?: CmoChannel and toolDomain?: CmoToolDomain fields unique to CMO.

Test Plan

  • TypeScript compilation: npm run type-check passes with new types (13/13)
  • Full build: npm run build passes with empty barrels (11/11) — catches webpack/bundler barrel issues that type-check misses
  • Import test: types importable from @tomachina/core barrel

MUS-C02 Registry Class Implementation

Create the CMORegistry class that provides lookup, filter, and composition capabilities over the registered tools.

File Paths

packages/core/src/cmo/registry.ts — NEW

Acceptance Criteria

  • CMO_REGISTRY: CmoRegistryEntry[] exported as the master array (populated by tool registration tickets)
  • getCmoTool(id: string) — returns single entry or undefined
  • getCmoToolsByDomain(domain: CmoToolDomain) — returns filtered array
  • getCmoToolsByChannel(channel: CmoChannel) — returns tools relevant to channel
  • getCmoToolsByType(type: 'TOOL' | 'SUPER_TOOL' | 'WIRE') — filters by registry entry type
  • getCmoWire(wireId: string) — returns wire definition with composed steps
  • All functions are pure — no side effects, no API calls
  • Export both the array and the helper functions

Reference Pattern

Follow QUE registry pattern at packages/core/src/que/registry.ts. QUE uses a flat array with filter helpers. Same pattern here. Do NOT over-engineer — no classes with state, no singletons. Flat array + pure functions.

Test Plan

  • getCmoTool('canva-generate-design') returns correct entry
  • getCmoToolsByDomain('canva') returns exactly 21 entries
  • getCmoToolsByChannel('print') returns Canva + PDF_SERVICE tools
  • getCmoToolsByType('WIRE') returns exactly 3 wire entries
  • getCmoWire('WIRE_BROCHURE') returns wire definition with steps
  • All helpers return empty arrays (not undefined) for no-match queries
