Field Alias Audit

CRITICAL: Before adding ANY new fields to the UI, these alias conflicts MUST be resolved. Adding UI for both face_amount and face_value when they represent the same data creates confusion for the team and double-maintenance forever.

The field inventory identified 19 alias groups where multiple Firestore fields may represent the same data. For each pair, the table shows population counts, which systems use them, and the recommended resolution.

Field Parity Enforcement

The alias problem happened because data arrived from multiple sources (GHL, carrier imports, SPARK, manual entry) with no enforcement that field names match the canonical schema. This section proposes a three-layer defense.

Layer 1: Guardian Schema Enforcement (Existing System)

The Guardian data protection engine already defines CollectionSchema. Extend it:

// packages/core/src/guardian/schemas.ts
export const CLIENT_SCHEMA: CollectionSchema = {
  collection: 'clients',
  required_fields: ['client_id', 'first_name', 'last_name', 'status'],
  canonical_fields: {
    // Every allowed field name → its type
    'dob': 'string',
    'phone': 'string',
    'state': 'string',
    'county': 'string',
    // ... all 92 fields
  },
  banned_aliases: {
    // Maps known bad names to their canonical version
    'date_of_birth': 'dob',
    'address_street': 'address',
    'address_city': 'city',
    'address_state': 'state',
    'address_zip': 'zip',
    'address_county': 'county',
    'client_status': 'status',
    'face_value': 'face_amount',
    'underwriting_charter': 'carrier_charter',
    'fund_value': 'account_value',
    'annuity_balance': 'account_value',
  }
};

Guardian audit phase checks for banned aliases in new writes and flags them as findings.

Layer 2: Import Normalizer (Intake Gateway)

The FIELD_NORMALIZERS in packages/core/src/normalizers/field-normalizers.ts already handles 90+ fields. Add an alias resolution step that runs before normalization:

// packages/core/src/normalizers/alias-resolver.ts
const ALIAS_MAP: Record<string, string> = {
  'date_of_birth': 'dob',
  'address_street': 'address',
  'address_city': 'city',
  'address_state': 'state',
  'address_zip': 'zip',
  'address_county': 'county',
  'face_value': 'face_amount',
  'fund_value': 'account_value',
  'annuity_balance': 'account_value',
  'underwriting_charter': 'carrier_charter',
  'client_status': 'status',
  'net_surrender_value': 'surrender_value',
  'base_cash_value': 'cash_value',
  'annualized_premium': 'annual_premium',
  'insured': 'insured_name',
  'account_owner': 'owner_name', // on context
};

export function resolveAliases(
  record: Record<string, unknown>
): Record<string, unknown> {
  const resolved: Record<string, unknown> = {};
  for (const [key, value] of Object.entries(record)) {
    const canonical = ALIAS_MAP[key] ?? key;
    // Only overwrite if canonical field is empty
    if (!resolved[canonical] || resolved[canonical] === '') {
      resolved[canonical] = value;
    }
  }
  return resolved;
}

This runs in the intake pipeline (services/intake/) so every new record arrives with canonical field names. No aliases ever reach Firestore again.
