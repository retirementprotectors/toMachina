ZRD-O08Ranger Orchestration API

REST API control plane for the Ranger mesh. This is a Cloud Run Express route module — NOT a CCSDK agent, NOT a standalone service. It follows the same pattern as wire.ts and atlas.ts: Express router mounted at /api/rangers/* inside the existing tm-api Cloud Run service.

Architecture Decision

What the "dispatcher" is: A synchronous Express route handler that (1) looks up the Ranger class from registry.ts by rangerId, (2) instantiates it with the provided params, (3) kicks off ranger.execute() as a background task, (4) writes a ranger_runs/{runId} Firestore doc with status running, and (5) returns { success: true, data: { runId } } immediately. The Ranger itself updates its Firestore run doc as it progresses through steps. No queue, no pub/sub, no CCSDK — direct in-process execution on Cloud Run.

Why not CCSDK: Rangers are deterministic multi-step pipelines (read file → normalize → validate → write). They don't need LLM reasoning. CCSDK agents are for conversational AI (VOLTRON, Specialists). Different tool, different job.

Why not a separate service: Rangers share the same Firestore client, Firebase Auth middleware, and ATLAS registry access as the existing API. A separate service would duplicate all of that for no benefit. If load requires it later, Rangers can be split to their own Cloud Run service — but that's a scaling decision, not a v1 decision.

File Paths

services/api/src/routes/rangers.ts — NEW: Express router (dispatch, status, history, cancel, registry endpoints) services/api/src/rangers/registry.ts — NEW: Map<rangerId, RangerClass> + lookup + validation services/api/src/rangers/types.ts — REFERENCE (from ZRD-O01): RangerRun, RangerStep, RangerStatus types

Acceptance Criteria

  • POST /api/rangers/dispatch — validates rangerId against registry, creates ranger_runs/{runId} Firestore doc, starts execution, returns { success: true, data: { runId } }
  • GET /api/rangers/:runId/status — reads ranger_runs/{runId}, returns step-level progress with timestamps
  • GET /api/rangers/history — paginated query on ranger_runs collection, filterable by rangerId + status + date range
  • POST /api/rangers/:runId/cancel — sets status: 'cancelled' on Firestore doc; Ranger checks this flag between steps and halts
  • GET /api/rangers/registry — returns all registered Rangers with wire bindings, step definitions, and current idle/running status
  • registry.ts exports getRanger(rangerId): RangerClass | null and listRangers(): RangerMeta[]
  • Firebase Auth required (EXECUTIVE+ entitlement)
  • Response format: { success, data?, error? }
  • Firestore collection: ranger_runs — one doc per execution with fields: runId, rangerId, status, steps[], startedAt, completedAt, triggeredBy, input, output, error

Reference Pattern

services/api/src/routes/wire.ts (230 lines) for the Express router + background execution pattern. services/api/src/routes/atlas.ts for the registry-read pattern.

Failure Modes

Silent step failure: Each step MUST check previous step status before proceeding. Wire executor already has this — verify Rangers don't bypass it.

Orphaned runs: Ranger crashes, status stays 'running' forever. Timeout: >15 min with no step progress = auto-mark 'failed'.

Intake misroute: SUPER_INTROSPECT misidentifies format. Confidence <80% = queue for manual review.
