Phase 1 — Cleanup

Cleanup Remove noise so the real data stands out. Trash template shells, fix errors, relocate orphans.

TRK-445 Trash verified template-only shell folders

List each folder's contents. If ALL contents match the template baseline (subfolders: *Accounts, _Accounts, *Opportunities, _Opportunities, CLIENT FACING; files: Q325 Ai3 TEMPLATE, RPI 2025 Discovery Meeting Kit, Implemented Medicare + Retirement Strategies) → trash via deleteItem. Any content beyond baseline → skip.

Tool: gdrive MCP via sub-agents (parallel, 50+ folders per agent) Recovery: 30-day Google Drive Trash window Status: IN PROGRESS (~316 trashed)
TRK-446 Clean up undefined + error folders

Trash 5 "ACF - undefined" folders, 1 "ACF - cli test", 1 "ACF - Messy Test". Rename 3 "AFC -" typo folders to "ACF -" (Susan Rogness, Steven Schau, Jerry & Sondra Bottenfield).

Tool: gdrive MCP — deleteItem for junk, renameItem for typos
TRK-447 Relocate 16 loose root-level files

For each loose file at root of Active Client Files: parse client name / policy number from filename, search Firestore for match, move to client's ACF folder. No match → move to "_Unsorted" holding folder. 4 template ZIPs → trash. 3 client PDFs (Callahan, Soss, Rempe) → match to ACF.

Tool: gdrive MCP + Firestore lookup

Phase 2 — Audit Real Data

Data Catalog every populated folder. Know exactly what exists before restructuring anything.

TRK-448 Catalog ~120 populated ACF- folders

For each populated ACF- folder: list all files + subfolders, record name/mimeType/size, classify document type by filename pattern matching against the 59 types in _DOCUMENT_TAXONOMY, record which of the proposed 5 subfolders each doc maps to.

Tool: gdrive MCP via parallel sub-agents Output: .claude/acf-2-0-document-intelligence/CATALOG_ACF.json
TRK-449 Catalog ~98 old-format folders

Same method as TRK-448 but for xACF-, zACF-, ACFx-, AFC- folders. Additionally flag folders with unique data not duplicated in any standard ACF- folder.

Tool: gdrive MCP via parallel sub-agents Output: .claude/acf-2-0-document-intelligence/CATALOG_OLD_FORMAT.json
TRK-450 Validate proposed subfolder structure

Load both catalogs. For every document found, attempt to classify into NewBiz / Reactive / Cases / Account / Client. Count documents per category, list any that don't fit cleanly. Present distribution + gaps to JDM for validation.

Tool: Analysis of TRK-448 + TRK-449 catalogs Output: .claude/acf-2-0-document-intelligence/STRUCTURE_VALIDATION.md
Gate: JDM approval required on subfolder structure validation before Phase 3 proceeds.

Phase 3 — Restructure

Data Apply the new 5-folder structure. Update config, re-map taxonomy, create subfolders, move documents, standardize names.

TRK-451 Update ACF Config with new 5-folder structure

Update Firestore acf_config/default document: set subfolders to ['NewBiz', 'Reactive', 'Cases', 'Account', 'Client'].

Tool: Firestore Admin SDK Script: services/api/src/scripts/update-acf-config.ts
TRK-452 Re-map _DOCUMENT_TAXONOMY acf_subfolder column

For each of 59 rows: pipeline=NewBiz → NewBiz. pipeline=REACTIVE + owner_role=AST → Reactive. pipeline=REACTIVE + owner_role=COR → Cases or Account. pipeline=PRO → Account. Personal docs (DL, Medicare card) → Client.

Tool: Google Sheets API via updateGoogleSheet Gate: Mapping must be reviewed by JDM before write
TRK-453 Create new subfolders in existing ACFs

For each remaining ACF folder: list existing subfolders, create any of the 5 new subfolders that don't exist, log folder ID + created subfolders. Batch 50 folders at a time.

Tool: GAS script with UrlFetchApp.fetchAll (batch parallel)
TRK-454 Categorize + move existing documents

Using catalog from Phase 2: for each document, look up classified subfolder, move file from current location to target subfolder. Documents that can't be classified → leave in place, log for manual review.

Tool: GAS script with UrlFetchApp.fetchAll
TRK-455 Rename folders to standard format

Rename non-standard prefixes: xACF- → ACF -, zACF- → ACF -, ACFx- → ACF -, AFC - → ACF -. Fix double spaces, case inconsistencies. Folder IDs stay the same on rename — no Firestore update needed.

Tool: gdrive MCP renameItem

Phase 4 — De-dupe + Consolidate

Cleanup Merge duplicates, flag internal dupes, fix known misfiles.

TRK-456 Merge 33+ duplicate folder pairs

For each duplicate pair: list both folders' contents, pick winner (more files + more recent modifications), move unique files from loser to winner, trash empty loser, update Firestore client doc to point to winner's folder ID.

Tool: gdrive MCP + Firestore Admin SDK
TRK-457 De-dupe documents within folders

For each ACF folder: list all files, compare by exact name match, name similarity >90%, identical file size. Flag potential dupes in report — DO NOT auto-delete.

Tool: gdrive MCP + analysis script Output: .claude/acf-2-0-document-intelligence/DUPE_REPORT.md
TRK-458 Fix Dickson misfiles

Search all ACF folders for files containing "Dickson" in name. For each hit NOT in the Dickson family ACF → delete. Verify Dickson family ACF has their correct Ai3.

Tool: gdrive MCP

Phase 5 — ProDash UI Document Linking

App Wire documents directly into ProDash. Config-driven, product-aware, real-time.

Can start parallel with Phase 4. UI component work is completely independent of Drive cleanup.
TRK-459 Build document linking config

Firestore collection document_link_config. Each doc defines: document_type, display_name, target_ui (client_detail | account_detail | case_detail), acf_subfolder, product_types, file_patterns, priority, icon.

Location: Firestore document_link_config collection
TRK-460 Proactive pull (scheduled scan)

Cron job hits API endpoint. For each client with acf_folder_id: list files in each subfolder, match files to document_link_config patterns, upsert to Firestore document_index collection.

Location: services/api/src/routes/document-index.ts + Cloud Scheduler Schedule: Daily at 2am CT
TRK-461 Reactive pull (on document arrival)

After Watcher routes doc to ACF subfolder: match file against document_link_config, if match → upsert document_index entry immediately. Firestore onSnapshot triggers UI refresh.

Location: Watcher post-processing hook
TRK-462 Client detail page document widgets

React component reads document_index for client_id, filters by target_ui = 'client_detail'. Renders: DL thumbnail, Medicare card, voided check, latest Ai3. Click → opens Drive file in new tab.

Location: packages/ui/src/modules/ClientDocuments.tsx
TRK-463 Account detail page document widgets

Same pattern as TRK-462 but filtered by target_ui = 'account_detail' and product_type match. Shows: latest statement, policy doc, product-specific docs.

Location: packages/ui/src/modules/AccountDocuments.tsx

Phase 6 — Routing Rules

Infra Teach the Watcher the new structure. Seed config from taxonomy, enforce naming conventions.

Can start parallel with Phase 5. Routing rules config is independent of UI wiring.
TRK-464 Seed routing_rules from _DOCUMENT_TAXONOMY

Read all 59 rows from Sheets API, transform to routing_rules format, write to acf_config/default in Firestore.

Tool: Script reading Sheets API → writing Firestore
TRK-465 Update Watcher subfolder targets

The Watcher already reads acf_subfolder from _DOCUMENT_TAXONOMY. After TRK-452 re-maps the values, the Watcher automatically uses the new targets. Verify with test documents.

Location: ~/Projects/RAPID_TOOLS/MCP-Hub/document-processor/src/watcher.js Method: Verification only — no code change needed
TRK-466 File naming enforcement

After document classification: check if filename matches PolicyNum- LASTNAME Carrier DocType YYYY-MM-DD. If not → rename using resolved file label from taxonomy. Log original filename in extraction metadata.

Location: Watcher post-processing
