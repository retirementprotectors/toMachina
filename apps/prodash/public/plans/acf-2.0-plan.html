TRK-445 — Template Shell Trash

Trash ~1,085 empty template-only folders gdrive MCP

Approach: Spawn sub-agents (already in progress). Each agent scans a batch of folders.

Template baseline signature (all 6 must match):
  • *Accounts
  • *Opportunities
  • Ai3 TEMPLATE (or Ai3 Template variant)
  • Discovery Kit
  • CLIENT FACING
  • Implemented Medicare
Execution steps:
  1. Use mcp__gdrive__listFolder on ACF root to get all folder IDs
  2. For each folder, listFolder to get contents
  3. If folder contains ONLY template baseline items (no real files) → mark for trash
  4. Verify each folder before trashing — never trash a folder with non-template content
  5. Use mcp__gdrive__deleteItem (moves to Drive Trash, 30-day recovery window)
  6. Parallelize across 5+ sub-agents, each handling ~200 folders
Safety: 30-day recovery from Drive Trash. Log every trashed folder ID + name to a Drive text file for audit trail. If any folder has even 1 file beyond the template baseline, SKIP it.

TRK-448 — Scan Populated ACF Folders

Catalog ~120 real client folders gdrive MCP sub-agents

For each of ~120 populated ACF - folders:
  1. listFolder recursively (folder + all subfolders)
  2. For each file, record: file_name, file_type (extension), parent_subfolder, size
  3. Categorize by document type using filename pattern matching:
    • Statements → contains "statement" or carrier + date pattern
    • DL / ID → contains "DL", "license", "ID"

    • Ai3 → contains "Ai3", "ai3"
    • Medicare Card → contains "medicare", "MBI"
    • Unknown → everything else
  4. Output: JSON catalog per folder, aggregated into master catalog
Parallelism: Spawn 6 sub-agents, each handling ~20 folders. Each agent writes its catalog to a shared Drive file or returns structured JSON. GA aggregates.

TRK-456 — Merge Duplicate Folder Pairs

Consolidate 33+ duplicate pairs gdrive MCP Firestore

For each of 33+ duplicate pairs:
  1. listFolder on both folders — get full file inventory of each
  2. Pick winner: folder with more real files (not template items)
  3. Move unique files from loser folder into winner's matching subfolders
  4. If both folders have the same file (same name + similar size) → keep the newer one
  5. Trash the loser folder (30-day recovery)
  6. Update Firestore: If the loser folder's ID was the acf_folder_id on the client doc, update it to the winner's ID
  7. Log: winner_id, loser_id, files_moved, files_skipped (dupes), firestore_updated
Duplicate detection types:
  • Exact name match: Two ACF - John Smith folders
  • Cross-format: ACF - John Smith + xACF- John Smith
  • Misspelling: ACF - Jon Smith + ACF - John Smith (requires Firestore client lookup to confirm same person)
TRK-457 — Intra-Folder File De-dupe

Flag duplicate files within folders gdrive MCP

Approach: Report only — NO auto-delete
  1. For each ACF folder, list all files across all subfolders
  2. Compare files by:
    • Name similarity: Levenshtein distance < 3, or identical base name with different date suffix
    • Size match: Same file size (within 1KB) + similar name → likely dupe
    • Exact match: Same name + same size = definite dupe
  3. Generate a de-dupe report (Google Sheet or Drive text file):
    • Columns: folder_name, file_1_name, file_1_id, file_2_name, file_2_id, match_type, confidence
  4. Present report to JDM for review before any action
Safety: This ticket produces a REPORT only. No files are deleted. JDM reviews and approves deletions from the report.

TRK-460 — Document Indexer (Scheduled)

Cloud Run cron job — scan ACF, index to Firestore Cloud Run Drive API Firestore

New file: services/api/src/jobs/document-indexer.ts
Trigger: Cloud Scheduler → hits POST /api/internal/index-documents (IAM-protected)
Schedule: Every 6 hours (adjustable)
  1. Query all clients with acf_folder_id set in Firestore
  2. For each client's ACF folder: list files in all 5 subfolders via Drive API (drive.files.list with parent filter)
  3. For each file, upsert to Firestore document_index collection:
    { client_id, file_id, file_name, document_type, subfolder, drive_url, mime_type, size, modified_at, indexed_at }
  4. Document type detection: match filename against _DOCUMENT_TAXONOMY patterns

Performance: ~218 folders × 5 subfolders = ~1,090 Drive API listFiles calls per full scan. At 10 QPS sustained, ~2 minutes. Incremental scans will be much faster.

TRK-466 — File Naming Enforcement

Auto-rename incoming files to standard pattern services/api

Standard pattern: PolicyNum- LASTNAME Carrier DocType YYYY-MM-DD
Example: 12345- SMITH Nationwide Statement 2026-03-15
  1. Add naming validation to watcher post-processing (after document is routed to subfolder)
  2. Parse incoming filename against the expected pattern:
    /^(\S+)-\s+([A-Z]+)\s+(.+)\s+(\d{4}-\d{2}-\d{2})$/
  3. If filename doesn't match:
    • Attempt auto-rename using metadata from the watcher context (policy number, client last name, carrier, doc type, date)
    • Use drive.files.update to rename the file in Drive
    • Log: { original_filename, new_filename, rename_reason, file_id }
  4. If insufficient metadata for auto-rename: keep original filename, flag in document_index as naming_compliant: false
  5. Future: ProDash Admin UI can show non-compliant files for manual rename

Key Decisions

Approval Gates

  • TRK-450 → P3: JDM must approve the document-to-subfolder mapping before restructuring begins
  • TRK-457: De-dupe report requires JDM review before any file deletions

Rollback Strategy

  • Drive Trash: All deleted/trashed folders have 30-day recovery
  • Firestore: legacy_subfolders field preserves old config for rollback
  • Taxonomy: Sheets version history preserves pre-migration state
  • Move logs: Every file move is logged with original + destination paths

Performance Constraints

  • Drive API rate limit: 12,000 queries/min per user, 1B/day. Sub-agent parallelism stays well within limits.
  • Firestore batch writes: Max 500 operations per batch. Document indexer batches accordingly.
  • Cloud Scheduler: 6-hour interval balances freshness vs. API cost. Adjustable post-launch.
