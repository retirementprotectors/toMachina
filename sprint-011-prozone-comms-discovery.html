The Problem with REST API Calls

CallPanel currently uses POST /api/comms/send-voice (REST) to initiate calls. REST calls fire-and-forget — there's no call event feedback. The component may never receive the signal to transition from pre_call to calling. Without entering the calling state, the disposition step is unreachable. This is likely why Vince says "there's no way to disposition." The feature exists — the state transition is broken.

Pipeline integration: On disposition, CallPanel auto-advances pipeline stages: booked advances stage, not_interested closes the opportunity. This logic is sound — it just needs to be reachable.

TwilioDeviceProvider.tsx — The Bridge We're Not Using

The browser-based Twilio Voice SDK is already wired into the platform. It exposes makeCall, hangup, answerCall, and full call event callbacks (onRinging, onConnect, onDisconnect).

What's There

Full Twilio browser SDK: makeCall, hangup, answerCall, call event callbacks. ProZone doesn't use it — it uses REST instead. Switching CallPanel to TwilioDeviceProvider gives us real-time call state (ringing, connected, disconnected) which drives the calling → disposition transition.

Feature 1 — Auto-Dial Power Dialer

Current state: Single prospect click-to-call via green phone button in TargetTab. CallPanel slides in, handles one prospect, closes.

Needed: Batch/sequential auto-dial queue. Select multiple prospects, start a session, the system auto-loads each one in sequence after disposition is logged.

How It Works

Step 1 — Select

Checkbox column added to TargetTab. Sales rep selects 5, 10, 20 prospects. "Start Call Session" button activates at the bottom of the grid.

Step 2 — Session Start

ProZoneApp enters power dialer mode. CallSession.tsx wraps CallPanel with queue context. First prospect auto-loads — rep makes the call.

Step 3 — Disposition & Advance

Rep logs disposition. useCallSession advances queue to next prospect. CallPanel auto-loads next name and number. Repeat until queue is empty.

Step 4 — Session End

Queue empties or rep ends manually. Session summary shown: total calls, booked, no-answer, callbacks, duration. Firestore batch log written.

Session Controls

ControlBehavior
Pause SessionStops auto-advance after current call completes. Rep can take a break, review notes.
Resume SessionRe-activates auto-advance. Loads next prospect in queue.
End SessionExits power dialer mode at any point. Shows partial session summary. Logs all completed calls.
Skip ProspectRemoves current prospect from queue without logging. Moves to next.

Technical Approach

Browser SDK over REST: Power dialer mode requires real call event feedback to drive auto-advance. TwilioDeviceProvider.makeCall fires events on connect/disconnect — these are what trigger the calling → disposition → next prospect chain. REST has no equivalent.

Feature 2 — Call Disposition Fix

Vince says "there's no way to disposition the call based on how it ended." The feature is built. Something is preventing him from reaching it.

What's Already Built in CallPanel.tsx

4 disposition outcomes: booked, callback, no_answer, not_interested. Pipeline auto-advance on disposition (booked = advance stage, not_interested = close opportunity). The disposition step is the 3rd state in a 3-state flow.

The Suspected Root Cause

CallPanel uses POST /api/comms/send-voice (REST) to initiate calls. A REST call fires and returns a 200. It does not emit call lifecycle events. Without a onConnect event, the component has no signal to transition from pre_call to calling. Without entering calling, the "Hang Up" button never appears. Without hanging up, disposition is never reached. Vince is stuck at pre_call — looking at a panel that says it called but never advanced.

The Fix Plan

1. Switch to Browser SDK

Replace POST /api/comms/send-voice in CallPanel with TwilioDeviceProvider.makeCall(). The browser SDK fires real events: onRinging, onConnect, onDisconnect. These map directly to pre_call → calling → disposition state transitions.

// CallPanel.tsx — after fix const { makeCall, hangup, on } = useTwilioDevice(); const handleCall = () => { makeCall(prospect.phone); }; on('connect', () => setCallState('calling')); // enters calling state on('disconnect', () => setCallState('disposition')); // enters disposition

2. Make Disposition Mandatory

The panel must not close without a disposition logged. No X button in disposition state. Rep must pick an outcome before the panel closes. This ensures every call gets logged — no orphaned communications records.

3. Expand Outcome List

Feature 3 — MMS Outbound

Vince wants to send JPEG and PDF files as attachments in texts. Inbound MMS is already captured. Outbound is the missing piece.

What Already Works (Inbound)

The POST /api/comms/webhooks/sms-incoming webhook already captures media_urls[] from inbound MMS messages and stores them in Firestore. CommsTimeline already knows how to display received media attachments. The infrastructure understands MMS — we just don't send it yet.

Twilio MMS Constraints

ConstraintValue
GeographyUS and Canada only (Twilio MMS limitation)
Max file size5MB per Twilio MMS spec
Supported typesJPEG, PNG, GIF (animated), PDF, MP3, MP4 (varies by carrier)
Cost~$0.01 per message segment + media overage
Supported file types for this sprintJPEG, PNG, PDF (most common sales use case)

Implementation Spec

Frontend — SendSmsDialog.tsx

Add a file attachment input below the message body. Accept JPEG, PNG, PDF. Show a thumbnail preview for images, filename for PDFs. Enforce 5MB client-side before upload attempt.

// SendSmsDialog.tsx additions <input type="file" accept="image/jpeg,image/png,application/pdf" onChange={handleFileSelect} /> // File preview — image thumbnail or PDF filename badge {selectedFile && <AttachmentPreview file={selectedFile} onRemove={clearFile} />} // Size guard if (file.size > 5 * 1024 * 1024) { showToast('File must be under 5MB for MMS', 'error'); return; }

Upload — Cloud Storage

Files upload to Cloud Storage bucket (temp path: mms-uploads/{userId}/{timestamp}/{filename}) before sending. The API route receives a storage URL, not the raw file. Storage rules restrict access to authenticated RPI users only.

API Route — comms.ts

Extend POST /api/comms/send-sms to accept an optional mediaUrl field. Pass it as MediaUrl parameter to Twilio. Twilio fetches the file from Cloud Storage and attaches it to the MMS.

// comms.ts send-sms route const { to, body, mediaUrl } = req.body; const messageParams: Twilio.MessageCreateOptions = { to, from: TWILIO_PHONE_NUMBER, body, ...(mediaUrl && { mediaUrl: [mediaUrl] }), // MMS only if file attached }; const message = await twilioClient.messages.create(messageParams);

Display — CommsTimeline.tsx

Sent MMS attachments render in the timeline alongside the message bubble. Images show as thumbnails (click to expand). PDFs show as a labeled attachment with download link. Matches the existing inbound MMS display pattern.

What Vince Sees

A paperclip icon in the SMS compose panel. He picks a JPEG or PDF from his machine. It previews before sending. He hits Send. The attachment appears in the conversation timeline alongside the message.

What the Client Sees

A standard MMS from RPI's number. The image or PDF renders inline on most modern smartphones. No special app required. PDF may render as a download link on some carriers.
