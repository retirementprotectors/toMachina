The Problem

JDM is manually firing FORGE prompts at every gate. He is the QA agent, the Jr QA, the conductor, and the relay between builders — simultaneously. At 181 agencies and 90,000 clients, the platform can't scale if the CEO is the one pushing every sprint through its gates.

Every phase transition requires JDM to copy a prompt, wait, review, copy another prompt, wait again. The FORGE lifecycle is solid — but it runs at human speed because a human is running it.

The Solution

FORGE Runner — an autonomous orchestrator running on MDJ1 that receives a Discovery Document and drives the entire 11-step FORGE lifecycle without JDM in the loop. It creates sprints, fires audits, distributes work to builder subagents, verifies results, and deploys code. JDM approves at key gates — everything else is automated.

Creates the Sprint

POSTs to /api/sprints, generates Discovery HTML, creates Tracker items for all tickets. No manual setup.

Drives the Lifecycle

Advances through all 11 FORGE phases automatically. Audits pass → next phase. Audits fail → auto-fix and re-audit.

Routes to Builders

Spawns parallel Sonnet subagents per ticket. Each builder follows the BUILDER_PROMPT format. Results flow back to the runner.

Deploys Code

Runs the full #SendIt sequence. If CI is red, diagnoses and fixes. If CI is green, auto-merges. Reports #LandedIt.

Survives Restarts

All state stored in Firestore. If MDJ1 reboots mid-sprint, the runner resumes from exactly where it left off.

Pauses for JDM

Three human gates with ProDashX notifications. JDM clicks approve in RIIMO or via API. One tap to resume.

The Outcome

JDM gives a vision — a sentence, a doc, a Slack message. The Machine builds it, tests it, audits it, deploys it. JDM reviews at three key gates. Development scales without JDM being the bottleneck.

Where It Lives

MDJ1 Server

~/mdj-agent/src/forge/ — FORGE Runner runs as a persistent Express service on the MDJ1 PowerEdge server. Always on, survives reboots.

Firestore State

mdj_forge_runs collection stores all run state. If MDJ1 reboots mid-sprint, the runner reads Firestore and resumes the correct phase.

toMachina API

Runner calls /api/sprints and /api/tracker to create and manage FORGE artifacts. Same routes the RIIMO portal uses.

Agent SDK

@anthropic-ai/claude-agent-sdk — every Claude invocation goes through the SDK wrapper, with per-phase model routing and budget caps.

Parallel Builder Pattern

Step 8 (Build) is the only phase that fans out. The runner spawns one Sonnet subagent per ticket. All tickets in a phase run in parallel. Results are merged and fed into the Audit Build phase.

Runner (Step 8 — Build) | |-- subagent: FORGE-001 (sonnet) ─┐ |-- subagent: FORGE-002 (sonnet) | |-- subagent: FORGE-003 (sonnet) ├─→ results merged → Step 9 Audit |-- subagent: FORGE-004 (sonnet) | |-- subagent: FORGE-005 (sonnet) ─┘ ...all run simultaneously

@anthropic-ai/claude-agent-sdk (TypeScript)

The FORGE Runner uses the official Claude Agent SDK for all Claude invocations. Every phase is a query() call with phase-specific options. No raw API calls — the SDK handles streaming, tool use, session continuity, and subagent spawning.

Core: query()

Async iterator. Streams output tokens as they arrive. Used for every phase from discovery through deployment.

import { query } from '@anthropic-ai/claude-agent-sdk';

// Run a phase with full options
const result = await collectOutput(query({
  prompt: planPrompt,
  options: {
    model: 'claude-opus-4-5',           // Opus for planning
    allowedTools: ['Read', 'Glob', 'Grep'], // Read-only in planning
    permissionMode: 'acceptEdits',           // Auto-approve file changes
    maxBudgetUsd: 5.00,                    // Cap spend this phase
  }
}));

Model Routing via options.model

Each phase specifies its own model. The runner never defaults — every invocation is explicit.

// Planning — Opus
options: { model: 'claude-opus-4-5', maxBudgetUsd: 5.00 }

// Building — Sonnet
options: { model: 'claude-sonnet-4-5', maxBudgetUsd: 3.00 }

// Audits — Haiku
options: { model: 'claude-haiku-4-5', maxBudgetUsd: 1.00 }

Parallel Builders via options.agents

The build phase spawns one subagent per ticket. All run concurrently. Results are collected and merged.

const builderResults = await Promise.all(
  tickets.map(ticket =>
    collectOutput(query({
      prompt: buildPrompt(ticket),
      options: {
        model: 'claude-sonnet-4-5',
        allowedTools: ['Read', 'Write', 'Edit', 'Bash'],
        permissionMode: 'acceptEdits',
        maxBudgetUsd: 3.00,
      }
    }))
  )
);

File Change Hooks via options.hooks

Every file write during the build phase triggers a hook that logs the change to the ForgeRun audit trail in Firestore.

options: {
  hooks: {
    onFileWrite: async (filePath, content) => {
      await logAuditEvent(runId, {
        type: 'file_write',
        path: filePath,
        timestamp: new Date().toISOString(),
      });
    }
  }
}

MCP Injection via createSdkMcpServer()

The FORGE Runner injects custom MCP tools so the Claude session can query sprint state, look up tracker items, and check gate status without leaving the SDK context.

import { createSdkMcpServer } from '@anthropic-ai/claude-agent-sdk';

const forgeMcp = createSdkMcpServer({
  tools: {
    get_sprint: async ({ sprintId }) => forgeApi.getSprint(sprintId),
    get_tracker_items: async ({ sprintId }) => forgeApi.getItems(sprintId),
    get_run_state: async ({ runId }) => getForgeRunState(runId),
  }
});

options: { mcpServers: [forgeMcp] }

Session Continuity via options.resume

If MDJ1 reboots mid-phase, the runner reads the last session ID from Firestore and resumes without losing context.

const state = await getForgeRunState(runId);

const result = await collectOutput(query({
  prompt: 'Resume and continue from where you left off.',
  options: {
    resume: state.lastSessionId,  // Full context restored
    model: state.currentPhaseModel,
  }
}));

Session Management

The runner tracks every session ID per phase, enabling full audit trails and resumption.

ForgeRunState Interface

Stored in Firestore collection mdj_forge_runs. Every field is updated atomically on state transitions.

interface ForgeRunState {
  // Identity
  id: string;                         // UUID, also the Firestore doc ID
  sprintId: string;                   // /api/sprints record ID
  sprintName: string;
  createdAt: string;                  // ISO 8601
  createdBy: string;                  // uid of initiator (JDM)

  // Phase tracking
  phase: ForgePhase;                  // Current state (enum)
  phaseStartedAt: string;
  phaseAttempts: number;              // Retry counter for audits
  phaseLog: ForgePhaseLogEntry[];     // Full history of all phases

  // Session continuity
  lastSessionId: string | null;      // For SDK resume
  sessionsByPhase: Record<string, string>; // phase → sessionId

  // Artifacts
  discoveryHtml: string | null;
  planMarkdown: string | null;
  tickets: ForgeTicket[];
  buildResults: ForgeBuildResult[];

  // Gate state
  gatesPending: ForgeGate[];
  gatesApproved: ForgeGate[];
  gateNotificationSentAt: string | null;

  // CI loop
  ciAttempts: number;
  ciStatus: 'pending' | 'green' | 'red' | null;
  prUrl: string | null;
  mergedAt: string | null;

  // Cost tracking
  totalCostUsd: number;
  costByPhase: Record<string, number>;

  // Terminal state
  completedAt: string | null;
  failedAt: string | null;
  failureReason: string | null;
}

type ForgePhase =

  | 'creating_sprint'

  | 'building'
  | 'auditing_build'
  | 'awaiting_build_approval'
  | 'deploying'

Three Human Gates

FORGE Runner is autonomous except at three gates where JDM's judgment is required. The runner pauses, notifies, and waits. JDM taps Approve — the runner resumes from exactly where it stopped.

Gate 1 — After Plan Audit

"Is this the right plan?" — Triggered after Opus generates the plan and Haiku audits it for gaps. JDM reviews the plan document and approves before any code is written. This is the last point to catch direction errors before builder spend starts.

Gate 2 — After Build Audit

"Does this look right?" — Triggered after all builders complete and Sonnet audits the code against the plan. JDM reviews the diff summary and audit report before any deployment starts.

Gate 3 — CI Failure Escalation

"I can't fix this" — Triggered if CI fails 3 consecutive times and the runner cannot self-diagnose. Runner pauses, sends a full diagnostic report to JDM, and waits for instruction. This is the safety valve for genuinely broken environments.

Gate Mechanism

1

Runner pauses + saves state

Phase transitions to awaiting_plan_approval or awaiting_build_approval. All state flushed to Firestore. Runner enters a polling loop — checks every 30 seconds for approval.

2

ProDashX notification sent

Runner posts a notification to the in-app notification system. JDM sees it in ProDashX, RIIMO, and optionally Slack. Notification links directly to the FORGE run status page with the relevant doc attached.

3

JDM approves

JDM clicks Approve in RIIMO admin, or hits POST /forge/approve/:id directly. Can also reject with a note, which triggers a revision cycle.

4

Runner resumes

Firestore gate status flips to approved. Runner's polling loop detects the change and advances to the next phase. No restart required.

Key Module Responsibilities

runner.ts

The orchestrator. Reads Firestore state, calls the correct phase executor, writes state back. Never contains Claude logic directly — delegates to phases.ts.

phases.ts

One function per phase. Each function calls claude-invoke.ts, handles PASS/FAIL branching, and returns results to runner.ts. Self-contained, independently testable.

claude-invoke.ts

The only file that imports from the Agent SDK. Wraps query() with model routing, budget caps, session tracking, and cost extraction. Everything else is SDK-agnostic.

templates.ts

All prompts live here. Changing a prompt never requires touching runner or phase logic. Templates are functions: (state: ForgeRunState) => string.

forge-api.ts

All HTTP calls to /api/sprints and /api/tracker. Handles auth headers, retries, and error mapping. Never makes direct Firestore calls.

types.ts

Single source of truth for all FORGE types. No duplication across files. Imported everywhere. ForgeRunState is defined here and nowhere else.

FORGE-006

runner.ts — Core state machine

The orchestrator. Reads ForgeRunState from Firestore, determines which phase to execute, delegates to phases.ts, writes updated state back. Handles restart recovery (if phase is mid-execution, re-run from current phase start). Manages the gate polling loop.

FORGE-007

phases.ts — Phase executors (all 11 steps)

One exported async function per phase. Each calls invokePhase(), processes the result, updates the run state, and returns control to runner.ts. Build phase fans out to parallel subagents. Audit phases evaluate PASS/FAIL and set retry counter.

FORGE-008

routes.ts — Express endpoints

Mount FORGE routes on the MDJ1 Express server. POST /forge/sprint kicks off a new run. GET /forge/status/:id returns current state. POST /forge/approve/:id flips gate status and runner detects it in the polling loop. GET /forge/runs returns list with summary stats.

FORGE-009

Human gate mechanism — Firestore + ProDashX notifications

When runner reaches a gate state, it writes a notification to Firestore notifications collection (same collection the portal reads). The ProDashX notification includes the gate name, a link to the run status page, and the artifact to review (plan doc or build diff). Gate polling loop in runner.ts checks gateStatus every 30s.
