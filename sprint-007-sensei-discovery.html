The Problem

JDM answers the same questions over and over. "How do I use Households?" "Where's the pipeline?" "What happened to that feature?" Every question burns CEO time. RAIDEN has a TRAIN triage outcome built in — but no training content to point to. VOLTRON can walk someone through anything — but has no knowledge base to draw from. The team gets trained once, forgets, and asks again.

Meanwhile, CI already screenshots every single UI state on every deploy via Playwright. The raw material for always-current training exists. It's just not organized, templated, or accessible.

The Solution: SENSEI

Every question answered once. Every answer available everywhere. Zero repeat training.

1. Content Pipeline

CI runs Playwright → screenshots captured → uploaded to Firebase Storage organized by module → training generator builds role-based HTML docs. Ticket type = content type: workflow → step-by-step, ui → visual, feat → feature overview. Deploys replace. Always current. Zero maintenance.

2. SENSEI Mode (The Toggle)

Top-level toggle in the portal header. OFF by default. When ON: hover over anything → training popup for that exact element. Context-aware: Workspace → Tab → Module → App. Every layer teaches itself. One click, everything has a guide.

3. Knowledge Wiring

RAIDEN: Ticket about something that exists → "We already have that. Here's the training." → ticket closed. No human in loop.

VOLTRON: "How do I use Households?" → pulls SENSEI content → walks them through conversationally. Same content, two delivery mechanisms.

The Insight

E2E tests already photograph the entire UI. Ticket types already categorize the content. Permissions already know who sees what. ATLAS already knows the terminology. The training system isn't being built from scratch — it's a pipeline that converts what already exists into training docs. The screenshots are the raw material. The work is: organize → template → publish → wire.

The Gold Standard

The DEX Training Guide we built for Nikki and Angie is what every auto-generated training looks like:

The Screenshot Pipeline

CI already has the raw material. The ticket type IS the content type. When it deploys, it replaces. Always current. Zero maintenance.

Workflow Tickets

CI captures every step of the workflow. Output: step-by-step training with numbered instructions + screenshots at each stage. Like the NBX/DEX doc for Nikki and Angie.

UI Tickets

CI captures the visual state of every screen. Output: visual training — annotated screenshots showing what each element does, where things are, how to navigate.

Feature Tickets

CI captures the feature in context. Output: feature overview — what it does, where to find it, what changed from the previous version.

Training Generator

The generator reads organized screenshots + pulls live stats from Firestore + applies the DEX Training Doc HTML template. Output is a self-contained HTML per module.

UI Specification

The Toggle

Portal header, right side near user avatar. Icon: graduation cap or lightbulb. Label: "SENSEI". Amber glow when active. Persists per user in Firestore user preferences. OFF by default.

Overlay Indicators

When SENSEI Mode ON: subtle pulsing dots on interactive elements. Non-intrusive. Amber color to match SENSEI brand. Click/hover reveals training popup.

Training Popup

Floating card near element. Screenshot + short description + "View Full Training" link to the full HTML doc. Dismissible. Doesn't block the UI.

Here's what you see when you open it: [screenshot] Step 1: Select the client... Step 2: Review their accounts... Step 3: Calculate the distribution..."

The Deduplication Loop

Every time RAIDEN returns a TRAIN response, it logs the query. Over time, this creates a heat map of what people ask about most. SENSEI can prioritize training depth for high-traffic modules. The system literally teaches itself what to teach better.
