What VOLTRON Becomes

The Aswegan Proof. "Full review on the Aswegans" → 8 autonomous tool calls → complete $1M+ portfolio review in seconds. That's not a chatbot. That's a capability executor. Sprint 002 formalizes it.

Before Sprint 002

82 hand-coded tools. No composition. No workflow. Each request hits VOLTRON one tool at a time. The Aswegan review works — but only because VOLTRON stumbles through it via model reasoning.

After Sprint 002

Structured Super Tools + Wires. VOLTRON receives "build casework for the Minearts" → executes BUILD_CASEWORK wire → pulls data, runs QUE, generates HTML, saves to ACF, Slacks Archer. Done. Archer closed $100k.

User request→ get_client→ run_que_quote→ generate_html→ save_to_acf→ notify_agent (Slack)→ Closed deal

This is the full client-to-close pipeline in a single VOLTRON command. Sprint 002 makes every wire like this — deterministic, auditable, repeatable.

ATLAS Pattern — Applied to Capabilities

ATLAS manages data movement. The Action Engine manages capability execution. Same architecture, different domain. New home: packages/core/src/voltron/

Concept ATLAS (Data In) Action Engine (Capabilities Out)
Atomic Unit Atomic Tool — pure data transform function Atomic Tool — pure capability function (get_client, send_sms, etc.)
Result Type AtomicToolResult<T> { success, data?, error? } VoltronToolResult<T> { success, data?, error?, metadata? }
Orchestrator Super Tool — stage composition (SUPER_PREPARE, SUPER_CLASSIFY…) Super Tool — task composition (REVIEW_PREP, BUILD_CASEWORK…)
Orchestrator Result SuperToolResult<T> { success, data?, tool_results?, stats? } VoltronSuperResult<T> { success, data?, tool_results?, duration_ms? }
End-to-End Flow Wire — source → transform → Firestore Wire — request → orchestrate → output + artifacts
Wire Engine atlas/wire-executor.ts — sequential super tool chain voltron/wire-executor.ts — adapted, adds approval gates + streaming
Context Object SuperToolContext — carries file IDs, mappings, client_id VoltronContext — carries client_id, user_role, wire_id, entitlement
Registry ATLAS Firestore collection (source registry) voltron_registry Firestore collection (auto-generated)
Source of truth Hand-maintained + carrier introspection Auto-generated from codebase at deploy time
Approval gates approval_required flag on WireContext Per-wire gate config (e.g. confirm before send_sms)

Auto-Registry — Tools That Don't Go Stale

Problem: 82 hand-coded tool definitions in the MDJ agent. Every API route change, new Firestore field, or MCP update requires a manual edit to the tool list. They fall out of sync within days of a deploy.

Solution: Generate at Deploy Time

API routes (services/api/src/routes/*.ts)→ Registry Generator
Firestore schema (packages/db/src/types/*.ts)→ Registry Generator
MCP server manifests→ Registry Generator
Voltron tool declarations (voltron/tools/*.ts)→ Registry Generator
Merged voltron_registry (Firestore)

Generator Pipeline

SourceParse MethodOutput
API routesAST parse route files — extract method, path, params, return typeendpoint tool definitions
Firestore schemasRead TypeScript interface declarations from packages/dbfield-aware read/write tools
MCP serversRead MCP manifest JSON / tool list endpointMCP-backed tool wrappers
Voltron tool filesExport-scan voltron/tools/*.ts for tool metadata decoratorsfirst-class voltron tools

Super Tools — Task Orchestrators

Wires — End-to-End Workflows

Wires compose Super Tools into complete business workflows. The Wire Executor (adapted from atlas/wire-executor.ts) runs them sequentially, chains output N→N+1, enforces approval gates, and writes an audit trail to Firestore.

Wire Executor Adaptations

ATLAS Wire ExecutorVoltron Wire Executor (new)
WireInput has file_id / csv / commission modesVoltronWireInput has client_id, wire_id, params, entitlement
Approval gates for bulk data importsApproval gates for comms (send_sms, send_email) and irreversible actions
Writes created_records to FirestoreWrites wire_executions audit log to Firestore
Dry run mode for testingSimulation mode — shows what would happen without executing
No streamingServer-Sent Events for real-time progress in Mode 1 UI

Wire Execution Audit Trail

Each wire execution writes to wire_executions/{execution_id}: wire_id, user_email, client_id, started_at, completed_at, stage_results[], status, artifacts[]. Same pattern as ATLAS wire runs.

Entitlement — Tool Filtering at Registry Level

Entitlement is enforced at GET /api/voltron/registry. Each request is authenticated via Firebase Auth. The user's role determines which tools are returned. VOLTRON never sees tools the user can't run.

Role Definitions

Coordinator1–3 (Atomic only)COR — support, service intake

Director1–5 (+ Wires basic)SPC — team lead, senior agent

Implementation: entitlement_min field on each registry entry maps to a role rank (Coordinator=1 … Admin=5). GET /api/voltron/registry reads caller's role from Firestore users collection, filters registry to entries where entitlement_rank <= caller_rank. VOLTRON's tool list is role-scoped before the model ever sees it.
